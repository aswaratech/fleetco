# Glossary

This is the project's semantic memory: the place where words have agreed meanings. Add new terms here on first use. The glossary is alphabetical within sections. Entries are not silently removed; if a term changes meaning over time, the change is documented in the entry rather than its history erased.

## Fleet and operations vocabulary

**Active trip** has two recorded usages, deliberately distinct so they cannot drift silently into each other. On the **Home dashboard**, "Active trips" means trips with status `IN_PROGRESS` only — "trips in progress now" (`lib/dashboard.ts`, the card's original definition). On the **Live map's active-trips layer** (ADR-0048), "active" means the wider dispatched, non-terminal set — `OFFERED`, `ACCEPTED`, or `IN_PROGRESS` — because a dispatched-but-unmoved truck is exactly what the fleet glance should surface; `PLANNED` (not dispatched) and the terminal `COMPLETED`/`CANCELLED` are never "active" in either usage. When a vehicle carries more than one active trip (no DB uniqueness forbids it — see `docs/tech-debt.md`), map attribution is deterministic: `IN_PROGRESS` > `ACCEPTED` > `OFFERED`, then recency. See the **Trip** / **trip.status** / **Dispatch** entries and ADR-0048.

**Bishesh Anumati (विशेष अनुमति)** is the Nepali term for a special permit, often required to operate certain heavy vehicles or to operate on certain routes. It is one of the compliance documents tracked by the system, and its expiry triggers a reminder.

**Bluebook (नीलो किताब)** is the vehicle registration certificate issued by Nepal's Department of Transport Management. It is called the Bluebook because of the color of its cover. Every vehicle in FleetCo has a Bluebook number and metadata, and Bluebook expiry triggers compliance reminders. As of Phase 1 iter 14, persisted on Vehicle as `bluebookNumber` (string, nullable) and `bluebookExpiresAt` (DateTime, nullable).

**consignee** is the party who *receives* a haulage load at its drop-off site — the person a tipper's driver asks for on arrival ("deliver to the Pokhara site, ask for Ram (98…)"). Introduced with **ADR-0047** (trip dispatch) as two nullable **Trip** columns, `consigneeName` and `consigneePhone`, captured per dispatch alongside the drop-off **Site**. The consignee is deliberately **distinct from the billing Customer** (ADR-0047 c6): the receiver standing at a site is generally *not* the party FleetCo invoices, so the consignee is not shoehorned into the **Customer** aggregate (nor the site-level `contactName`/`contactPhone` on the **Site** — a different person again). Consignee identifiers are **Tier-2 PII** (ADR-0013, amended by ADR-0047 c6, the Customer-slice precedent): `consigneePhone` inherits the pino `*.phone` redaction and a new `*.consigneeName` redact path is added, so the consignee never rides in a log line, a span, or a URL query string; the driver app renders `consigneePhone` as a **tap-to-call** `tel:` affordance, never a printed or logged number. See the **Dispatch** / **Site** / **Trip** entries and ADR-0047 c6.

**Customer** is a party who hires FleetCo's vehicles for jobs. A customer is distinct from a lease-taker. A customer engages FleetCo to do work; a lease-taker rents vehicles to do their own work. As of Phase 1 iter 15, Customer is persisted as its own aggregate with `name` (Tier 3 business identifier), `contactPerson` (Tier 2 PII, nullable), `phone` (Tier 2 PII), `email` (Tier 2 PII, nullable), `panNumber` (Permanent Account Number; Tier 3 tax-identifier, unique-when-present), `address` (Tier 3, nullable), and `status` (`customer.status` enum), all under `model Customer` in `apps/api/prisma/schema.prisma`. Iter 15 shipped the read path (schema + migration, `GET /api/v1/customers`, `GET /api/v1/customers/:id`, list and detail UI, home-page nav link). Iter 16 shipped the write path (`POST` / `PATCH` / `DELETE /api/v1/customers/:id`, plus create / edit / delete UI under `apps/web/src/app/(app)/customers/`), with PAN normalization (trim + uppercase) at the service layer, the PAN-uniqueness `P2002` → 409 mapping mirroring the iter-7 license-number rule on Drivers (response body carries `field: "panNumber"` so the web form surfaces the conflict inline on the PAN input), and forward-compatible `P2003` → 409 ("Cannot delete customer: it is referenced by other records.") on `DELETE` ahead of the first inbound FK. Customer has no inbound FKs from other aggregates yet — the iter-17 Jobs slice will be the first cross-slice consumer and the first to exercise the delete-conflict branch.

**customer.status** is the enum tracking whether a customer is current business or dormant business. Values are `ACTIVE` (default; current paying business) and `INACTIVE` (a customer the operator has stopped working with, kept on file for historical / accounting reference). The Customer status set is deliberately smaller than the Driver / Vehicle status sets: a customer relationship does not have intermediate "on leave" / "in maintenance" semantics — either they are a customer right now or they are a former customer. Phase 1 iter 15 shipped the read filter; iter 16 shipped the write path which preserves the same two-value set (the create form defaults `status` to `ACTIVE`; the edit form lets the operator toggle between `ACTIVE` and `INACTIVE`).

**Dispatch** is the workflow, introduced with **ADR-0047**, by which the admin hands a specific **Trip** — carrying its structured haulage **order** (a **MaterialType** + optional `materialNote`, pinned pickup and drop-off **Site**s, the **consignee**, expected load count, special instructions, and a free-text docket number) — to a chosen **driver + vehicle**, and the driver receives a request in the phone app and **accepts** it before driving it. It is modelled on a ride-hailing driver interface but with **hard-assign, not marketplace matching** (ADR-0047 c2): FleetCo is one company with employee drivers, so the dispatcher *chooses* the driver — there is no auto-matching, no offer-cascade, no bidding, and no acceptance countdown. The flow is **accept-only**: the driver **Accepts** (`OFFERED → ACCEPTED`, stamping `acceptedAt`), and there is *no in-app decline* — a driver who cannot take a trip is handled out-of-band and the admin **reassigns** by editing the OFFERED/ACCEPTED trip's driver/vehicle (the existing PATCH re-assign path; an `OFFERED → PLANNED` reassign-back exists too). Because dispatch:trip is 1:1, the order attaches as **nullable columns on the Trip** (ADR-0047 c3, the ADR-0036 columns-on-Trip precedent) rather than a separate `Dispatch` aggregate; extracting a `Dispatch` child aggregate is the named revisit if the acceptance lifecycle later grows (decline-with-reason, re-offer, timeouts). The admin dispatch UI and the driver's order view can draw the route line + an ETA estimate through a provider-agnostic **`RoutingProvider`** seam (a no-network Mock in dev/test/CI; a live Google/OSM impl is M1-gated), while the driver's *authoritative* turn-by-turn is a **Navigate** deep-link to the device's Google Maps (ADR-0047 c9, no API key, no cost). RBAC is the existing spine: dispatch + `Site` CRUD = ADMIN + OFFICE_STAFF (`trips:*` create already held by both, new `sites:*`); accept + progress = the driver PATCHing their **own** trip (no new driver-side capability). See the **Trip** / **trip.status** / **Site** / **MaterialType** / **consignee** / **milestone timestamps** entries and ADR-0047 (which *extends, does not supersede,* ADR-0003).

**Driver** is an employed or contracted person who operates vehicles on trips. Drivers have license metadata (`licenseNumber`, `licenseClass`, `licenseExpiresAt`), Tier 2 PII (`fullName`, `phone`, optional `dateOfBirth`), and an employment status (`driver.status`). Drivers are participants in trips; they are not the central aggregate. Iter 6 shipped the read path (schema, list and detail API, list and detail UI). Iter 7 shipped the write path (`POST` / `PATCH` / `DELETE /api/v1/drivers`, plus create / edit / delete UI under `apps/web/src/app/(app)/drivers/`). As of iter 13, the Driver detail page surfaces lifetime stats (completed trip count, total km logged across COMPLETED trips, and the vehicle paired with the most recent trip) via `GET /api/v1/drivers/:id/stats` — the symmetric mirror of the iter-12 lifetime-stats surface on the Vehicle detail page. As of 2026-07-05, a Driver row can be linked to its mobile-app login (`POST` / `DELETE /api/v1/drivers/:id/login-link`, gated by `users:manage`, not the shared `drivers:*` floor) — the write path ADR-0034 c8 left open, closing the gap where `Driver.userId` had no writer anywhere in the app. See the **DRIVER role** entry for the mechanism.

**driver.status** is the enum tracking a driver's employment state. Values are `ACTIVE` (default; eligible to be assigned to trips), `ON_LEAVE` (temporarily unavailable but still employed), `SUSPENDED` (administratively withheld from trips pending a decision), and `TERMINATED` (no longer employed). Transition into `TERMINATED` populates the driver's `terminatedAt` date via the **Termination transition** rule, mirroring the **Retirement transition** rule on vehicles.

**engine-hours** is the hour-meter dimension for the earthmoving / equipment half of the fleet (excavators, loaders, graders, generators, cranes), introduced with **ADR-0036** (Program B). Where trucks and tippers wear by distance (the **Odometer**), equipment wears by how long its engine runs — an excavator parked on a site digging for ten hours travels a few hundred metres while every wear-bearing component runs hard — so the number that drives its utilization, preventive-maintenance triggers, rental billing, and resale is engine-hours (the manufacturer's "Service Meter Reading" / SMR), not kilometres. Engine-hours are the **Odometer machinery, rotated**: nullable integer columns on `Vehicle` (`engineHoursStart`, the baseline at acquisition, and `engineHoursCurrent`) and `Trip` (start + end, captured at the same `→IN_PROGRESS` / `→COMPLETED` status transitions as the odometer readings), with a Trip→Vehicle bump that advances `engineHoursCurrent` to a completed trip's end-hours inside the **same** `COMPLETED`-transition `$transaction` as the odometer bump, under the identical monotonic "once forward, stays forward" rule. Readings are stored as **integer tenths of an hour** (deci-hours) — never a float — so `1234.5 h` stores as `12345`: the CLAUDE.md never-floats rule extended to a physical quantity exactly as `FuelLog.litersMl` extended it to volume; display divides by ten (`formatHours` in `apps/web/src/lib/units.ts`), cross-field math stays in integers. Which reading the trip-stop UI prompts for is governed by the vehicle's **meterType**; the per-vehicle lifetime `totalHoursLogged` stat mirrors `totalKmLogged`. Capture is **operator-entered** at trip stop on both the web edit form and the driver app (`apps-mobile/driver/`); an automated CAN-bus / telematics feed is a future revisit (the integer-tenths storage is chosen partly to receive it cleanly). See ADR-0036, the **meterType** / **Odometer** entries, and — the consumer of the engine-hours trigger — the **ServiceSchedule** entry.

**Expense log** is a single record of a fleet-attributable cost on one date. The persisted ExpenseLog row carries **nullable** `vehicleId` (FK Restrict to Vehicle — nullable because a vehicle-agnostic expense like the company's quarterly insurance premium or office stationery is a valid ledger row that does not belong under any specific vehicle's cost basis; the iter-23 cost report will surface the `vehicleId IS NULL` bucket via its own endpoint), **nullable** `tripId` (FK Restrict to Trip — like Fuel logs, an Expense may or may not be paired with a specific trip; mutable on PATCH so the operator can pair / unpair after the fact), `date` (UTC), `category` (the enum **`ExpenseCategory`**: `MAINTENANCE` / `REPAIR` / `TOLL` / `PARKING` / `INSURANCE` / `PERMIT` / `FINE` / `OTHER` — the eight values cover the categories the CEO already uses on the paper expense ledger; `OTHER` is the safety valve for genuine outliers, and adding a ninth category is a single-line migration), integer `amountPaisa` (NPR minor units per CLAUDE.md; **unlike Fuel logs there is no derived total — `amountPaisa` is the authoritative entered value, not a product of two factors**), nullable free-form `vendor` (the shop / mechanic / toll booth name — not normalized into a Vendor aggregate in iter 21, that lands later), nullable `receiptNumber`, nullable `notes`, and `createdById` (FK Restrict to User, populated from the authenticated session per ADR-0021). The list endpoint sorts by `date` desc by default; the schema's `(date desc)` partial index makes that the cheap natural sort, and `amountPaisa` is on the sortable-column whitelist (a notable difference from Fuel logs, which keeps `litersMl` off the sort list) because the per-vehicle cost report wants "biggest expense first" as a routine query and we'd rather expose it via the list endpoint than add a dedicated reports route this iter. Tier-1 operational data per ADR-0013 (no PII; `vendor` and `notes` stay Tier 1 unless the operator starts using them for contact info). As of Phase 1 iter 22, Expense logs ship the full read + write path: `model ExpenseLog` + `ExpenseCategory` enum, `GET /api/v1/expense-logs` (filters: vehicleId / tripId cuid, category enum, startDate / endDate inclusive; sortable columns: date / amountPaisa / createdAt; pagination), `GET /api/v1/expense-logs/:id` returning the nested Vehicle (nullable) and Trip (nullable), `POST /api/v1/expense-logs` (201 + created row with the operator-entered `amountPaisa` stored verbatim), `PATCH /api/v1/expense-logs/:id` (200 + updated row), `DELETE /api/v1/expense-logs/:id` (204), each guarded by `AuthGuard`. **`vehicleId` is immutable on PATCH** — the iter-22 `UpdateExpenseLogSchema` omits the field and `.strict()` rejects any client-sent value with HTTP 400 (mirror of the Fuel-log iter-20 rule; once an expense is attributed to a vehicle, changing the FK silently rewrites the per-vehicle cost report's basis — re-create the expense against the correct vehicle when the operator realises the original was mis-attributed). `vehicleId` IS optional on POST so a vehicle-agnostic expense (the company's quarterly insurance premium, office stationery) is a first-class ledger row. `tripId` is optional+nullable and **mutable on PATCH** so the operator can pair / unpair a trip after the fact; the service runs a **trip-vehicle consistency** check on both create (when `vehicleId` and `tripId` are both present) and update (against the merged shape, i.e., the patched `tripId` against the stored `vehicleId`); the error surfaces as HTTP 400 with the body "Trip <reg> belongs to vehicle <reg>; cannot pair with an expense log for vehicle <reg>." Unlike Fuel logs the service writes `amountPaisa` verbatim — there is **no derivation, no preview**, and the create / edit forms have a single `amount` (NPR rupees) input that the action layer converts to integer paisa via `Math.round(rupees * 100)`. Prisma `P2003` on a stale `vehicleId` / `tripId` / `createdById` is mapped to HTTP 400 with "Vehicle/Trip/User <id> does not exist." (see `docs/runbook/api-error-mapping.md` — the iter-22 row in the table). Create / edit / delete UI lives under `apps/web/src/app/(app)/expense-logs/new/` and `apps/web/src/app/(app)/expense-logs/[id]/edit/` plus `[id]/delete-expense-log-dialog.tsx`. The create form's vehicle picker has a leading "— no vehicle —" option (and the trip picker disables when no vehicle is selected); the edit form has no vehicle picker (immutable) and shows the current binding as a read-only row. The form schemas live at `apps/web/src/lib/expense-logs-schema.ts`. The remaining Phase-1 slice is Reports v1.

**Fuel log** is a single record of fuel dispensed into one vehicle on one date. The persisted FuelLog row carries `vehicleId` (FK Restrict to Vehicle, required, **immutable post-create** — a fuel log records a fact about which vehicle was fueled; rewriting that FK silently rewrites history, so the iter-20 PATCH schema's `.strict()` rejects `vehicleId` and the edit form renders the vehicle registration as a read-only display row), nullable `tripId` (FK Restrict to Trip — a fill happens whether or not it can be attributed to a specific trip; the operator pairs a fill with a trip when the receipt and the trip line up, otherwise the fill stands alone; **mutable on PATCH** so the operator can pair / unpair after the trip is created), `date` (UTC), integer `litersMl` (milliliters — a mechanical extension of the money-as-minor-units rule to volume, so per-liter prices stay exact integer math), integer `pricePerLiterPaisa`, integer `totalCostPaisa` (a **derived field**, server-computed by `FuelLogsService` as `Math.round(litersMl * pricePerLiterPaisa / 1000)` on every create and on every update that touches either factor — the rounding is half-up, not banker's, because it matches the operator's mental model for cash receipts in Nepal and the printed receipts they audit against; the worked example documented on the service is `12345 mL × 11055 paisa/L = 136473.975 → 136474 paisa`, demonstrating that `.5`-and-above rounds up; the iter-20 write schemas' `.strict()` rejects any client attempt to set `totalCostPaisa` directly), nullable integer `odometerReadingKm` (the vehicle's odometer at the time of fill — captured for a future per-fill view without coupling fuel logs to the Vehicle odometer auto-update path; note the **shipped** per-vehicle fuel-efficiency report (Reports v2) deliberately derives distance from completed trips, the system of record, NOT this reading; odometer readings are **non-monotonic by design** (a RECORDED DECISION as of the P3 polish sweep, not an open deferral) — there is no service-layer check that a fill's odometer reading is ≥ the previous fill's against the same vehicle, because backdated corrections, odometer swaps / replacements (a broken-odometer swap makes the new reading genuinely lower), and the future bulk-import path all make a hard write-time reject wrong; the soft signal that matters — a per-vehicle km/L outlier — belongs with the per-vehicle fuel-efficiency report (Reports v2, now **shipped**; it derives km/L from completed-trip distance, not this odometer field), not this write path. See the **Odometer** and **Per-vehicle fuel-efficiency report** entries and `docs/tech-debt.md` (Paid-off)), nullable free-form `station` (the fuel station's name; not normalized into a Vendor table in iter 19 — Vendor as an aggregate is a later slice), nullable `receiptNumber`, nullable `notes`, and `createdById` (FK Restrict to User, populated from the authenticated session per ADR-0021). The list endpoint sorts by `date` desc by default; the schema's `(date desc)` partial index makes that the cheap natural sort. As of Phase 1 iter 20, Fuel logs ship the full read + write path: `model FuelLog`, `GET /api/v1/fuel-logs` (filters: vehicleId / tripId cuid, startDate / endDate inclusive; sortable columns: date / createdAt; pagination), `GET /api/v1/fuel-logs/:id` returning the nested Vehicle (always present) and Trip (nullable), `POST /api/v1/fuel-logs` (201 + created row with derived `totalCostPaisa`), `PATCH /api/v1/fuel-logs/:id` (200 + updated row; recomputes `totalCostPaisa` when either factor is touched), `DELETE /api/v1/fuel-logs/:id` (204), each guarded by `AuthGuard`. A **trip-vehicle consistency** check at the service layer rejects pairing a fuel log against a trip whose `vehicleId` does not match — both on create (when `tripId` is set) and on update (against the merged shape, i.e., the patched `tripId` against the stored `vehicleId`); the error surfaces as HTTP 400 with the body "Trip <reg> belongs to vehicle <reg>; cannot pair with a fuel log for vehicle <reg>." Prisma `P2003` on a stale `vehicleId` or `tripId` is mapped to HTTP 400 with "Vehicle <id> does not exist." / "Trip <id> does not exist." (see `docs/runbook/api-error-mapping.md` — the iter-20 row in the table). Create / edit / delete UI lands under `apps/web/src/app/(app)/fuel-logs/new/` and `apps/web/src/app/(app)/fuel-logs/[id]/edit/` plus `[id]/delete-fuel-log-dialog.tsx`. The create form fetches active vehicles (required picker) and recent trips (optional picker, client-side-filtered to the selected vehicle so swapping vehicles auto-clears the trip pick); both forms render a read-only **total-cost preview** computed identically to the API's `deriveTotalCostPaisa` so the operator can spot a typo before submitting; the edit form pre-fills every editable field, PATCHes only the diff against initial values, and renders the vehicle registration as a read-only row. The form schemas live at `apps/web/src/lib/fuel-logs-schema.ts` and accept decimal-liters / decimal-rupees inputs which the action layer multiplies by 1000 / 100 and `Math.round`s to the integer mL / paisa the wire expects — same rounding rule on both sides of the wire means the preview is bit-for-bit the persisted value. The remaining Phase-1 slices are Expense logs and Reports v1.

**Geofence** is a polygon on the map representing a meaningful region: the depot, a customer site, a restricted zone, or a route corridor. GPS events crossing the boundary of a geofence are recorded and may trigger logic (alerts, automatic state transitions). The point geometry that geofence checks run against is the GpsPing aggregate's `geometry(Point, 4326)` column (ADR-0029 T2, the first PostGIS geometry in FleetCo). **Geofencing is now live (ADR-0029 T5):** the derived read layer's `GET /api/v1/telematics/vehicles/:vehicleId/geofence-status` endpoint (gated `gps:read-derived`) classifies a vehicle's latest fix against a caller-parameterized geofence via parameterized `$queryRaw` — point-in-polygon by `ST_Contains(ST_GeomFromText(<wkt>, 4326), geometry)` and proximity by `ST_DWithin(geometry::geography, <center>::geography, <meters>)` (the geography cast makes the radius meter-accurate; an SRID-4326 geometry `ST_DWithin` would measure in degrees). The response is a single boolean (a genuinely lower-resolution, Tier-3 derived status — not the raw trail relabeled, per ADR-0027 c6's anti-circumvention clause). Geofence-**polygon storage** is now a first-class aggregate (**ADR-0030**): `model Geofence` + a `GeofenceType` enum (DEPOT / CUSTOMER_SITE / ROUTE_CORRIDOR), Tier 3 per ADR-0027 c6, storing the boundary with the **hybrid** representation that extends ADR-0029's Point hybrid to a Polygon — a canonical Prisma-native `boundaryWkt` text column (a closed WGS84 ring as `POLYGON((lon lat, …))` WKT) plus a database-`GENERATED` `geometry(Polygon, 4326)` GIST-indexed column derived from it via `ST_GeomFromText("boundaryWkt", 4326)` (so it cannot drift), FleetCo's first PostGIS Polygon. The schema + hand-authored migration landed in G1; the CRUD API (`/api/v1/geofences`, service / controller / Zod schemas mirroring `customers/`) landed in **G2** with: the read/write RBAC split (`geofences:read` for ADMIN + OFFICE_STAFF, `geofences:write` for ADMIN only); the boundary accepted in the SAME `lon,lat;…` vertex representation the T5 query parses, via a **shared** WKT builder (`apps/api/src/common/wkt.ts`) both telematics and geofences import, so a stored fence and a query-param fence are byte-identical WKT (ADR-0030 c1); a service-level `ST_IsValid` pre-write gate that rejects a self-intersecting (bowtie) ring as 400 (ADR-0030 c2); and a type/ownership refine (CUSTOMER_SITE requires a `customerId`, DEPOT / ROUTE_CORRIDOR forbid one; the owning Customer's existing `P2003 → 409` delete-blocker arm covers a referenced fence for free). **G5 wires a stored fence into the T5 geofence-status query** (the "one-line change" ADR-0029 T5 anticipated): `GET /api/v1/telematics/vehicles/:vehicleId/geofence-status` now accepts a three-way **circle XOR polygon XOR `geofenceId`** choice (validated at the schema layer — two-or-more, or none, is a 400), and supplying `geofenceId` loads the stored fence through the `GeofencesService` public interface (**404** if missing) and classifies the vehicle's latest fix against its `boundaryWkt` via the SAME `ST_Contains(ST_GeomFromText("boundaryWkt", 4326), "geometry")` query — the stored WKT is byte-identical to a query-param polygon's, so a stored fence and an equivalent query-param polygon are proven to classify a fix **identically** (ADR-0030 c1's coherence guarantee). The gate **stays `gps:read-derived`** (ADMIN + OFFICE_STAFF): this reads geofence STATUS (a Tier-3 derived boolean), NOT the geofence CONFIG, so it is emphatically not `geofences:*`. The response echoes the resolved fence's id + type, never its coordinates (Tier-5 egress discipline holds). **With G5 the five-ticket geofence storage program (ADR-0030, G1–G5) is COMPLETE** — geofencing reads against stored depot / customer-site / route-corridor fences end-to-end. See ADR-0030, ADR-0029, and the **PostGIS** entry.

**GPS ping (telematics)** is one raw GPS fix reported for a vehicle — the atomic unit of FleetCo's telematics data, and expected to be the highest-row-count table in the system. Persisted as `model GpsPing` (ADR-0029 T2, the first PostGIS geometry in FleetCo): `vehicleId` (FK Restrict, required), nullable `tripId` (FK Restrict — collection is scoped to on-trip windows per ADR-0027 c2, but a fix captured at a shift boundary may carry none), the coordinate/movement fix `latitude` / `longitude` / `altitude` / `speed` / `heading` (every one **Tier 5** per ADR-0027 — a raw location trail, the most sensitive class: never logged via the pino `redact` list, never exported in a span — the ADR-0026 scrub seam now exists at `apps/api/src/observability/span-scrub.ts` (the `GpsSpanScrubProcessor` deletes the GPS denylist keys from every span before OTLP egress, the egress twin of the `redact` GPS paths) — pruned after the retention window), a `timestamp` (the device fix time — travels with the Tier-5 fix but is not itself independently sensitive, so it is not redacted), and `createdById` (FK Restrict to User, the authenticated submitter per ADR-0021). Coordinates use the **hybrid** storage of ADR-0029 commitment 8: native Float `latitude`/`longitude` as the canonical, Prisma-readable columns plus a database-generated `geometry(Point, 4326)` column (GIST-indexed, derived so it cannot drift) that spatial queries use — see the **PostGIS** entry. As of ADR-0029 T3, pings arrive through an authenticated **batch** endpoint `POST /api/v1/telematics/pings` (owned by the telematics feature module, `apps/api/src/modules/telematics/`): it accepts `{ pings: [ ... ] }` (a single ping is the degenerate batch-of-one; each ping carries its own `vehicleId`), validates minimally (coordinate ranges, cuid ids, ISO timestamp, `.strict()` unknown-key rejection → HTTP 400), is gated by `@RequirePermission("gps:ingest")` (ADMIN-only today, DRIVER-held later — ADR-0029 c11) on the composed AuthGuard + RolesGuard chain, then **enqueues onto the `gps-ingest` BullMQ queue and returns 202 fast** — it does NOT block on the database write, and `createdById` comes from the session, never the body. The `gps-ingest` worker (a `@Processor` + WorkerHost running in-process per ADR-0029 c5) dequeues and bulk-inserts each batch via `createMany` (the database derives the geometry; Prisma supplies only the Float columns); the batch-processing job is the unit of tracing (ADR-0026 c4), with no GPS coordinates in span attributes. The producer is the future React Native driver app, which does not exist yet, so the endpoint and worker are built and tested with synthetic batches posted as an authenticated principal — exactly as auth and RBAC were tested before any office-staff account existed. Retention is enforced by the `traces-prune` job (ADR-0029 T4): a daily keyed repeatable that hard-deletes pings whose fix `timestamp` is older than the provisional 90-day window — see the **Retention (telematics)** entry. Pings are read back through the RBAC-gated raw/derived split (ADR-0029 T5, ADR-0028 c6 / ADR-0027 c7): `GET …/vehicles/:vehicleId/pings` (the full-resolution raw trace, paginated + time-bounded, Prisma-native Float reads — never the geometry column) is gated `gps:read-raw` (ADMIN-only, the most-privileged data access in the system); `GET …/vehicles/:vehicleId/location` (the single latest fix for the live-location map) and `GET …/vehicles/:vehicleId/geofence-status` (the first PostGIS geofencing) are gated `gps:read-derived` (ADMIN + OFFICE_STAFF). Tier-5 coordinates travel to an authorized caller over the wire but never log and never enter a span. **This completed the five-ticket GPS telematics ingestion program (ADR-0029, T1–T5).** See ADR-0029, ADR-0027 (the Tier-5 classification), and the **BullMQ** / **PostGIS** / **Geofence** / **Capability** entries.

**IMEI** (International Mobile Equipment Identity) is the 15-digit identity of a cellular device — for FleetCo, the identity of a hardware GPS tracker unit (ADR-0042). It is the number the Traccar gateway matches an incoming device connection by, and therefore the join key between the physical unit and the **Tracker device** row (`TrackerDevice.imei`, unique, Tier 3 per ADR-0013 — an asset identifier like a registration number, not a person identifier). ADR-0042 c9 names the residual risk that the IMEI is also the ONLY thing the open tracker TCP port authenticates by, which is why the gateway registers known IMEIs only and the ingest adapter drops unknown-IMEI forwards. The write surface validates exactly 15 digits (`apps/api/src/modules/telematics/trackers.schemas.ts`) so a mistyped IMEI fails at registration rather than silently never matching a forward.

**Job** is a contract or engagement with a customer. A job consists of one or more trips. Example: "Haul aggregate from quarry X to construction site Y for two weeks" is a job; each individual round-trip is a trip on that job. The persisted Job row carries `jobNumber` (unique, of the form `JOB-YYYY-NNNNN`; the generator landed with iter 18 — server-side, derived from the current UTC year and the highest existing per-year sequence + 1, retried up to 3 times on `P2002` so a concurrent insert cannot collide deterministically; `jobNumber` is immutable post-create and the `PATCH` schema rejects it via `.strict()`), `customerId` (FK Restrict to Customer — deleting a customer that has any Job is blocked at the DB with Postgres 23503 / Prisma `P2003`, mapped to HTTP 409, the live exercise of the forward-compatible delete-blocker the Customers iter-16 write path wired; `customerId` is also immutable post-create — the `PATCH` schema rejects it), free-text `description`, `status` (the enum `job.status`: `PLANNED` / `IN_PROGRESS` / `COMPLETED` / `CANCELLED`), nullable `scheduledStartDate` / `scheduledEndDate` / `actualStartDate` / `actualEndDate` (each pair enforces end ≥ start as a service-layer cross-field rule that re-runs against the merged shape on `PATCH`), nullable `notes`, and `createdById` (FK Restrict to User, populated from the authenticated session — never accepted from the request body). Iter 17 shipped the read path (schema, migration, `GET /api/v1/jobs`, `GET /api/v1/jobs/:id`, list and detail UI); iter 18 closed the aggregate with the write path (`POST` / `PATCH` / `DELETE` + the jobNumber generator + create/edit/delete UI under `apps/web/src/app/(app)/jobs/new/` and `apps/web/src/app/(app)/jobs/[id]/edit/`) and swapped the Customer delete-dialog placeholder link to the now-live `/jobs?customerId=…` deep-link. A Trip will reference its Job by id in a later slice.

**Lease-taker** is a party who has rented or leased a vehicle for a period and operates it independently of FleetCo's direct dispatch. We bill lease-takers on a contract basis, not on a per-trip basis. Lease-takers are distinct from customers.

**licenseClass** is the enum classifying a driver by the category of Nepali driving licence they hold, as defined by Nepal's Department of Transport Management (DoTM). Phase 1 values cover the four categories relevant to heavy-construction haulage: `LMV` (Light Motor Vehicle — light four-wheelers up to ~7.5 tonnes GVW such as pickups and small utility trucks), `HMV` (Heavy Motor Vehicle — medium trucks above the LMV threshold, the common cargo class), `HTV` (Heavy Transport Vehicle — articulated trucks and tractor-trailers used for the heaviest cargo loads), and `HPMV` (Heavy Passenger Motor Vehicle — buses and other large passenger vehicles, kept in scope because the same operator endorsement applies). Categories outside this set (HGMV, motorcycle classes, special-equipment endorsements) may be added in a later slice; introducing them requires an ADR because they widen the operator's compliance surface.

**MaterialType** is the Prisma enum, introduced with **ADR-0047** (trip dispatch, c5), naming *what a haulage trip carries*. It seeds the common Nepal aggregate-haulage materials — `SAND`, `AGGREGATE`, `STONE`, `BOULDER`, `GRAVEL`, `SOIL`, `BRICKS` — plus an `OTHER` escape hatch that pairs with a free-text `materialNote` (the same curated-enum-with-`OTHER` shape as **vehicle.kind**). It is stored as a nullable `materialType` column on the **Trip** and, with a pickup and drop-off **Site**, is **required at the `→ OFFERED` transition**, unconstrained before it (mirroring how meter readings are required only at `IN_PROGRESS`/`COMPLETED`). An enum (not a managed lookup aggregate) keeps v1 lean and is trivially extended by a later migration; if the material list ever needs per-row attributes (density, a default rate) it graduates to its own aggregate — a named revisit, not a v1 concern. See the **Dispatch** / **Trip** entries and ADR-0047 c5.

**meterType** is the discriminator on `Vehicle` (ADR-0036) saying which meter(s) an asset has: `ODOMETER_KM` (the **default** — trucks and tippers, the distance-metered half), `ENGINE_HOURS` (excavators, loaders, graders, generators, cranes — the hour-metered half; see **engine-hours**), or `BOTH` (the occasional asset where both matter, e.g. a truck-mounted crane whose chassis runs on km but whose boom runs on hours). It makes the rest of the system unambiguous — which reading the trip-stop UI captures, which stat the Vehicle detail page shows, which meter a **ServiceSchedule** keys on — **independent of whether a reading has been captured yet**: a brand-new excavator with no SMR keyed in is still an `ENGINE_HOURS` asset ("tracks hours" and "has a reading yet" are different facts, the same distinction the nullable compliance columns draw). The default is `ODOMETER_KM` so the migration is backward-compatible — every existing row keeps its km-only behaviour with no data change, and the operator flips equipment to `ENGINE_HOURS` / `BOTH` as the fleet is classified. An explicit column is preferred over deriving the meter from **vehicle.kind** (a chassis taxonomy with an `OTHER` escape hatch that cannot express `BOTH` and breaks for `OTHER` assets — overloading it with a metering policy is exactly the implicit coupling that drifts). The meter-consistency rule (ADR-0037 c3) reads this column: an `ENGINE_HOURS` ServiceSchedule is rejected on an `ODOMETER_KM`-only vehicle, since it has no `engineHoursCurrent` to measure against. See ADR-0036 and the **engine-hours** / **ServiceSchedule** entries.

**milestone timestamps** are the six nullable `DateTime` columns on the **Trip** that record *dispatch progress as timestamps, not statuses* — the single most important anti-over-engineering decision in **ADR-0047** (c1/c8), and consistent with FleetCo's ISO/UTC-dates convention. Two are **server-stamped** at their transitions: `offeredAt` (at `→ OFFERED`, when the trip is dispatched) and `acceptedAt` (at `→ ACCEPTED`, when the driver accepts). The other four are the intra-trip **progress taps** the driver makes while the trip is `IN_PROGRESS` — `arrivedPickupAt`, `loadedAt`, `arrivedDropoffAt`, `deliveredAt` — each a one-field PATCH (no status change) on the driver's own-trip path, and each **monotonic / non-decreasing** so a done milestone reads as its label plus its time, "a timestamp, not a toggle". Modelling en-route/loaded/delivered stages as timestamps *within one `IN_PROGRESS` order* — rather than as a fine-grained status explosion (`EN_ROUTE_TO_PICKUP`, `LOADING`, …) — keeps the transition matrix, every list filter, and every consumer simple, and matches how mature dispatch systems track stop-level progress (ADR-0047 §Alternatives). Dates are stored ISO/UTC and rendered Bikram Sambat where a calendar date is shown (the driver app's progress rows show a device-local time-of-day, which is not a Y/M/D date, so no BS is involved). See the **Dispatch** / **Trip** / **trip.status** entries and ADR-0047 c1/c8.

**Odometer** is the integer kilometer reading on a vehicle's distance counter. FleetCo tracks two odometer values per vehicle: `odometerStartKm` (the reading at acquisition, used as the baseline for lifetime distance) and `odometerCurrentKm` (the most recent reading). Lifetime kilometers for a vehicle is `current − start`. Odometer values are stored as integers (no fractional kilometers); fractional distances belong on trip-level fields where precision matters. As of Phase 1 iter 11, `odometerCurrentKm` is updated automatically when a Trip transitions into `COMPLETED`: the service-layer `TripsService.update` runs the trip write plus a conditional `vehicle.update({ odometerCurrentKm })` inside a single Prisma interactive transaction, so the two rows commit or roll back together. The bump fires only when the trip's `endOdometerKm` is strictly greater than the vehicle's current value — a backdated correction trip whose reading is lower than the current value records the trip's history but does NOT move the vehicle's odometer backwards. Operator manual editing of `odometerCurrentKm` on the Vehicle edit form remains supported as the compensating action when (a) a vehicle's odometer needs to be corrected independently of trips and (b) when "rolling back" the odometer after a mistakenly-completed trip — the auto-update path is one-way (once forward, stays forward); the system does not unwind the bump if the COMPLETED trip is later deleted or its status is unwound by an operator. The previous behavior — `odometerCurrentKm` only updated by hand on the Vehicle edit form — applied through iter 10; iter 11 closed the gap that iter-10's cross-slice "Recent trips" sections had made visible (a vehicle's detail page showed it had trips, but its odometer field drifted behind the underlying odometer-end events). Separately, **fuel-log odometer readings are non-monotonic by design** — a recorded decision discharged in the P3 polish sweep. `FuelLog.odometerReadingKm` is stored as entered and is NOT required to be ≥ the prior fill's reading on the same vehicle; there is intentionally no write-time guard in `FuelLogsService`, because backdated corrections, odometer swaps / replacements, and the future bulk-import path all make a hard reject wrong. An out-of-order reading is tolerated, never rejected: the **shipped** per-vehicle fuel-efficiency report (Reports v2) measures km/L from completed-trip distance — the system of record (ADR-0003) — precisely so a non-monotonic fuel-log odometer cannot corrupt it, while the reading stays captured for a future per-fill view. See the **Fuel log** entry, the **Per-vehicle fuel-efficiency report** entry, and `docs/tech-debt.md` (Paid-off).

**Retirement transition** is the rule that links a Vehicle's `vehicle.status` to its `retiredAt` date. When a vehicle's status transitions into `RETIRED` or `SOLD`, `retiredAt` is set automatically (to the date the client passed, if any, otherwise to `now()`). When the status transitions back out of `RETIRED`/`SOLD` to `ACTIVE` or `IN_MAINTENANCE`, `retiredAt` is cleared. An explicit `retiredAt` value supplied by the client always wins over the derived value, supporting unusual but legitimate cases such as backfilling a historical retirement date or recording a sold-then-bought-back vehicle. The rule is applied in `VehiclesService.update` at `apps/api/src/modules/vehicles/vehicles.service.ts`. Iter 2 introduced this rule alongside the PATCH endpoint.

**ServiceRecord** is a completed service event — the maintenance history — one row per service performed on a vehicle (ADR-0037, the second of the preventive-maintenance aggregate's two tables; the **ServiceSchedule** is the recurring plan, the ServiceRecord is the immutable completed fact). The persisted row carries `vehicleId` (FK Restrict to Vehicle), nullable `serviceScheduleId` (FK Restrict to the **ServiceSchedule** it satisfies — null = an ad-hoc / one-off service with no schedule), nullable `expenseLogId` (the **cost-link**, ADR-0037 c6 — null = a warranty service with no cost, or the invoice is not keyed in yet), `performedAt` (the service date, UTC), nullable integer `odometerKm` and `engineHours` (the meter reading[s] captured at service — integer km / integer tenths-of-an-hour per **engine-hours**), nullable `notes`, and `createdById` (FK Restrict to User, from the session per ADR-0021). It carries **no `amountPaisa`**: the authoritative cost of a maintenance spend lives in exactly one place — an **Expense log** row with `category` `MAINTENANCE` or `REPAIR` — and the ServiceRecord *references* it rather than duplicating the money (the per-vehicle cost report keeps reading `ExpenseLog` directly; the link is additive context, "this MAINTENANCE expense was the 250-hour service on schedule X"). Recording a ServiceRecord against a schedule **advances that schedule's last-service anchor** forward — `lastServiceAt` plus the meter anchors — inside the **same** Prisma `$transaction` as the record insert, under the monotonic "once forward, stays forward" rule (ADR-0037 c5, the rotation of the Trip→Vehicle odometer bump), so recording a service resets "next due" forward by one interval; a backdated / lower-reading correction records history without rolling the anchor backward, and a PATCH never re-advances (editing a recorded service is the manual-correction path). Two service-layer consistency checks (the rotation of the Fuel/Expense trip-vehicle check): a set `serviceScheduleId` must point at a schedule on the **same** vehicle, and a set `expenseLogId` must point at a `MAINTENANCE`/`REPAIR` Expense log on the **same** vehicle — each a clean 400. Exposed as a CRUD aggregate at `/api/v1/service-records` (`AuthGuard`, the standard `{ items, total, skip, take, sortBy, sortDir }` list shape, `P2003` → 400 / `P2025` → 404; there is no unique constraint so no `P2002`, and nothing FKs into a ServiceRecord so no delete-when-referenced 409), with the web service-history surface under `apps/web/src/app/(app)/service-records/`. See ADR-0037, the **ServiceSchedule** / **Expense log** / **engine-hours** entries, and `docs/CURRENT_PHASE.md` (Program B).

**ServiceSchedule** is a recurring maintenance interval for a vehicle — "service this excavator every 250 engine-hours", "this tipper every 5,000 km", "this inspection every 365 days" — the first of the preventive-maintenance aggregate's two tables (ADR-0037; the **ServiceRecord** is the completed-service history that satisfies it). A vehicle has **many** schedules at once (an excavator might run a 250-hour engine service, a 500-hour hydraulic service, and an annual inspection simultaneously) and an unbounded growing history, which is why maintenance is a dedicated aggregate rather than columns on `Vehicle` — the deliberate cardinality contrast with the cardinality-one **engine-hours** columns. The persisted row carries `vehicleId` (FK Restrict), `name` (free-text, `@@unique([vehicleId, name])` so one schedule is not defined twice on a vehicle), nullable `description`, an `intervalType` discriminator — `DISTANCE_KM` (measured against the vehicle's `odometerCurrentKm`), `ENGINE_HOURS` (against `engineHoursCurrent`), or `CALENDAR_DAYS` (against the wall clock) — plus one integer `intervalValue` in that dimension's **minor units** (kilometres / integer tenths-of-an-hour per ADR-0036 / days; a 250-hour interval stores as `2500`, never a float), a `status` (`ACTIVE` / `INACTIVE`), and a **denormalized last-service anchor** (`lastServiceAt` + nullable `lastServiceOdometerKm` / `lastServiceEngineHours`) from which "next due" is **derived, not stored** (`nextDueKm = lastServiceOdometerKm + intervalValue`, and so on) — the same denormalization `Vehicle.odometerCurrentKm` already makes. The anchor is **seeded** at creation from the vehicle's current reading for the schedule's dimension when the operator omits it (ADR-0037 c4) and **advanced** transactionally when a **ServiceRecord** is recorded (c5). A **meter-consistency rule** binds `intervalType` to the vehicle's **meterType** (an `ENGINE_HOURS` schedule needs an hour-metered vehicle, ADR-0037 c3, re-validated against the merged shape on update). Exposed as a CRUD aggregate at `/api/v1/service-schedules` (`AuthGuard`, the standard list shape, the `@@unique([vehicleId, name])` `P2002` → 409, a stale `vehicleId` `P2003` → 400, and the delete-when-referenced `P2003` → 409 when a ServiceRecord still points at the schedule), with the web surface under `apps/web/src/app/(app)/service-schedules/` (list / detail / new / edit + the `/service-schedules/due` due-list). "Service due soon / overdue" is classified by the pure `serviceScheduleState` helper — and rolled up across many schedules by `worstServiceState` (the per-vehicle worst-of badge) and `rollUpServiceSchedules` (the Home-dashboard counts) — in `apps/web/src/lib/maintenance.ts` / `dashboard.ts`, painting the shipped amber/red `<Badge>`; the per-dimension due-soon windows (`DEFAULT_SERVICE_DUE_SOON_WINDOWS` — 500 km / 250 tenths-of-hour / 30 days) are **provisional** and the PO's to finalize (flagged in `docs/tech-debt.md`, the same owner-level-number pattern as the GPS-retention window). See ADR-0037, the **ServiceRecord** / **Service due/overdue badge** / **engine-hours** / **meterType** entries.

**Site** is the reusable, pinned pickup/drop-off location aggregate introduced with **ADR-0047** (trip dispatch, c4) — a crusher, a pit, a delivery site, or a depot that FleetCo hauls from/to repeatedly, defined once and reused forever (the PO chose ride-sharing-style **map pins**, not free text and not a polygon **Geofence**). A `Site` reuses the **ADR-0029 GpsPing hybrid Point pattern**: canonical native `latitude`/`longitude` `Float` columns plus a database-generated `geometry(Point, 4326)` column with a GIST index, hand-authored in the migration (`ST_SetSRID(ST_MakePoint(longitude, latitude), 4326)` — X = lon, Y = lat), `Unsupported`-declared so Prisma never selects it. It is FleetCo's first PostGIS **Point** aggregate that is a first-class *managed* entity (a GpsPing is a telematics fact; a Geofence is a **Polygon**, ADR-0030). Fields: `name`, `kind` (the **`SiteKind`** enum: `CRUSHER` / `PIT` / `DELIVERY_SITE` / `DEPOT` / `OTHER`), `latitude`, `longitude`, optional `address`, and optional `contactName` / `contactPhone` (a site-level contact — **Tier-2 PII** like the **consignee**, same redaction). A dispatch picks a saved Site from a list or drops a new pin on the map (ad-hoc → saved as a Site). A Site referenced by a trip's `pickupSiteId` / `dropoffSiteId` (both `onDelete: Restrict`) cannot be silently deleted — P2003 → 409, the house delete-blocker. `Site` CRUD is a new `sites:*` capability (ADMIN + OFFICE_STAFF, the Customer/Geofence module pattern). Deliberately **not** the polygon **Geofence** (wrong grain — a boundary, no crusher type, customer-owned); an optional future `Site.geofenceId` link (ADR-0030) for automatic arrival detection is explicitly deferred (see `docs/tech-debt.md`). See the **Dispatch** / **PostGIS** / **Geofence** / **GPS ping** entries and ADR-0047 c4.

**Termination transition** is the rule that links a Driver's `driver.status` to its `terminatedAt` date. When a driver's status transitions into `TERMINATED`, `terminatedAt` is set automatically (to the date the client passed, if any, otherwise to `now()`). When the status transitions out of `TERMINATED` back to `ACTIVE`/`ON_LEAVE`/`SUSPENDED`, `terminatedAt` is cleared. An explicit `terminatedAt` value supplied by the client always wins over the derived value, supporting unusual but legitimate cases such as backdating a termination or correcting a historical record. The rule is applied in `DriversService.update` at `apps/api/src/modules/drivers/drivers.service.ts` and mirrors the **Retirement transition** rule on vehicles. Iter 6 introduced this rule at the service layer; iter 7 exposed it over the `PATCH /api/v1/drivers/:id` endpoint.

**Tipper** is a truck with a hydraulic dump body, common in construction haulage. Tippers are a vehicle type within the fleet.

**Tracker device** is a hardware GPS unit (Teltonika FMC-series class, ADR-0042 c1) registered in FleetCo's tracker register — persisted as `model TrackerDevice` (ADR-0042 c6, M3): `imei` (unique, Tier 3 — see **IMEI**), nullable-unique `vehicleId` (FK Restrict; null = spare, set = the ONE mounted tracker on that vehicle; a mounted tracker blocks vehicle deletion until unassigned), `label`, `simMsisdn` (Tier 3 — the SIM's phone number, where configuration SMS commands go), `status` (`ACTIVE` / `SPARE` / `RETIRED`), and `installedAt` (the mount date on the CURRENT vehicle; the service resets it when the assignment changes without a new date). The M4 CRUD API lives in the telematics module (`/api/v1/telematics/trackers`, `trackers:read` = ADMIN + OFFICE_STAFF, `trackers:write` = ADMIN-only) with a `/trackers` admin surface. There is deliberately NO delete: a device that existed keeps its row; unassign frees the vehicle slot and RETIRED ends the lifecycle (a RETIRED tracker may not hold an assignment — unassign before retiring). The ingest adapter (M5) maps an incoming position to a vehicle only when the IMEI's row is `ACTIVE` **and** assigned; everything else is dropped fail-safe. See **Traccar** and **IMEI**.

**Trip** is the central aggregate of the FleetCo domain. One trip is one contiguous use of one vehicle by one driver for one job, with a start odometer reading, an end odometer reading, a start time, an end time, and route data accumulated during the trip. See ADR-0003 for why the Trip is the central aggregate. The persisted Trip row carries `vehicleId` (FK Restrict to Vehicle), `driverId` (FK Restrict to Driver), `status` (the enum `trip.status`), nullable `startedAt` and `endedAt`, nullable integer `startOdometerKm` and `endOdometerKm`, free-form `notes`, and `createdById` (FK Restrict to User). The FK delete policy is Restrict on all three references: deleting a vehicle or driver that participates in any Trip is blocked by the database with Postgres error 23503 (Prisma `P2003`); this is the iter-8 baseline that the iter-9 write path will translate into a friendly HTTP 409. Iter 8 shipped the read path (schema, migration, `GET /api/v1/trips`, `GET /api/v1/trips/:id`, list and detail UI). Iter 9 ships the write path (`POST` / `PATCH` / `DELETE`, plus create / edit / cancel UI). As of **ADR-0047** (trip dispatch, 2026-07) the Trip carries the structured haulage **order** it exists to execute — the reshape ADR-0003 explicitly reserved ("the first place we will ADR-revise") — as new nullable columns: `materialType` (a **MaterialType**) + `materialNote`, `pickupSiteId` / `dropoffSiteId` (FK → **Site**, `onDelete: Restrict`), `consigneeName` / `consigneePhone` (the **consignee**, Tier-2), `expectedLoadCount`, `specialInstructions`, `docketNumber`, and the six **milestone timestamps**; and its `status` gains the **`OFFERED`** and **`ACCEPTED`** dispatch states (see **trip.status** and the **Dispatch** entry). The order is **required at the `→ OFFERED` transition** (material + pickup + drop-off) and **one trip = one load** (a multi-load job is many trips). This *extends, does not supersede,* ADR-0003 — the Trip stays the central aggregate and the executor of a Job's work; a first-class `Trip.jobId` link and trip-level pricing remain distinct **deferred** items (their own ADR-0003-touching decisions — see `docs/tech-debt.md`), so the order attaches per-dispatch on the Trip, not via a Job.

**trip.status** is the enum tracking the lifecycle stage of a single Trip. Values are `PLANNED` (the trip is recorded but not yet started — common when dispatching ahead of time; `startedAt` and the odometer fields remain null), `IN_PROGRESS` (the trip has started and is currently underway; `startedAt` and `startOdometerKm` are set, `endedAt` and `endOdometerKm` are null), `COMPLETED` (the trip finished normally; all four start/end fields are set and `endOdometerKm >= startOdometerKm`), and `CANCELLED` (the trip was abandoned at any stage; the start fields may or may not be set depending on whether the cancellation happened before the trip started). The iter-9 write path will validate legal transitions (no jumping from `PLANNED` directly to `COMPLETED` without going through `IN_PROGRESS`) in service-layer logic mirroring the **Retirement transition** and **Termination transition** rules. As of **ADR-0047** (trip dispatch, 2026-07) the enum gains two states *between* `PLANNED` and `IN_PROGRESS`: **`OFFERED`** (the trip has been dispatched to a specific driver + vehicle carrying its order, awaiting the driver) and **`ACCEPTED`** (the driver acknowledged it; not yet started). The full machine is now `PLANNED → OFFERED → ACCEPTED → IN_PROGRESS → COMPLETED`, with `CANCELLED` still reachable from any non-terminal state, an `OFFERED → PLANNED` reassign-back, and the `PLANNED → IN_PROGRESS` direct start kept for back-compat. The driver **Accepts** (`OFFERED → ACCEPTED`); there is *no in-app decline* (see **Dispatch**). Intra-trip progress after `IN_PROGRESS` is tracked as **milestone timestamps**, not further statuses (ADR-0047 c1). The enum lives in **five lock-step mirrors** that must move together: the Prisma enum (`schema.prisma`), `TRIP_STATUSES` + `TRIP_STATUS_TRANSITIONS` (`apps/api/src/modules/trips/trips.schemas.ts`), the web `types.ts` vocabulary, and the driver-app union (`apps-mobile/driver/src/trips.ts`).

**Vehicle** is a registered asset in the fleet: a truck, a tipper, an excavator, a loader, a grader, or other heavy-construction equipment. Vehicles have a Nepali commercial registration number (unique across the fleet), a kind (the enum `vehicle.kind`), a make and model, a year of manufacture, a status (the enum `vehicle.status`), odometer readings (see `odometer`), an acquisition date, and an optional retirement date. Bluebook, insurance, and route-permit metadata are persisted on Vehicle as of Phase 1 iter 14 (all nullable columns); Bishesh Anumati remains deferred to a later slice. Vehicle records are classified Tier 3 (operational business data) per ADR-0013. Vehicle is the foundation aggregate for Trips per ADR-0003 — every Trip references one Vehicle by id.

**vehicle.kind** is the enum classifying a vehicle by chassis type. Phase 1 values are `TRUCK`, `TIPPER`, `EXCAVATOR`, `LOADER`, `GRADER`, and `OTHER`. `OTHER` exists so an unusual asset can be registered without forcing a schema change; future slices may promote frequent `OTHER` values to their own enum entries.

**vehicle.status** is the enum tracking a vehicle's operational state. Values are `ACTIVE` (default; available for trips), `IN_MAINTENANCE` (off-road for service), `RETIRED` (permanently out of service), and `SOLD` (no longer owned by FleetCo). Transition to `RETIRED` or `SOLD` populates the vehicle's `retiredAt` date.

**Vendor** is a party that FleetCo pays: a fuel station, a repair shop, a parts supplier. Vendors are distinct from customers (whom FleetCo bills) and from lease-takers (whom FleetCo also bills, but on a contract basis).

## Compliance vocabulary (Nepal-specific)

**BS** is the abbreviation for Bikram Sambat, the Nepali calendar. The Nepali fiscal year runs roughly from mid-July to mid-July (Shrawan to Ashadh). FleetCo stores dates internally in ISO 8601 (Gregorian) and renders Bikram Sambat dates for users where appropriate. As of ADR-0031 (the deferred Phase-1 daily-use polish), BS rendering is implemented by the **NepaliDate** component / `formatNepaliDate` formatter — Latin transliteration, BS prominent with a muted Gregorian parenthetical (e.g. `2083 Jestha 6 (2026-05-20)`), converting the verified `nepali-date-converter` library at render time only; dates are never stored as BS (DESIGN.md anti-pattern #12). BS *entry* is symmetric as of ADR-0032: the **NepaliDatePicker** widget lets operators pick dates from a BS month grid, emitting the same ISO string the native input did. See the **NepaliDate** / **NepaliDatePicker** entries and ADR-0031 / ADR-0032.

**Compliance badge** is the amber "Expiring soon" / red "Expired" indicator shown beside each vehicle compliance-document expiry date (bluebook, insurance, route permit) on the Vehicle detail page, introduced with ADR-0031 (the deferred Phase-1 daily-use polish). The classification is the pure `complianceBadgeState(expiryIso, now, windowDays = 30)` helper in `apps/web/src/lib/compliance.ts`: an expiry whose UTC calendar day is before today is `expired` (red); one within the window — today through day 30 inclusive — is `expiring-soon` (amber); one further out shows no badge; a null / unparseable date shows the date alone. The 30-day window is the PO-ratified threshold (ADR-0031 commitment 5). Like `formatNepaliDate`, the comparison truncates both dates to their UTC calendar day, so the boundary is deterministic regardless of server timezone. The same badge also surfaces as a worst-of-three roll-up column on the vehicles **list** (ADR-0031 §E / "Revisit when"), computed by the pure `worstComplianceState(expiries, now, windowDays = 30)` sibling helper in the same module — the most urgent of a vehicle's three document states by the precedence `expired` > `expiring-soon` > `ok` > `none`, built strictly on `complianceBadgeState` and never re-deriving the window / UTC rule (the Home dashboard's per-vehicle `vehicleComplianceState` delegates to the same helper, so the precedence lives in exactly one place). The badge is a passive, read-time visual only — compliance reminder *delivery* (email/SMS, the "reminder delivery" SLI) is Phase 3 and explicitly out of scope. See ADR-0031, the **Badge** entry, and the **BS** entry.

**Service due/overdue badge** is the amber "Service due soon" / red "Service overdue" indicator shown beside each ACTIVE service schedule on the Vehicle detail page (ADR-0037 c7, Program B / B4 — the maintenance twin of the **Compliance badge**, deliberately rotating that pattern). The classification is the pure `serviceScheduleState(schedule, reading, now, windows?)` helper in `apps/web/src/lib/maintenance.ts`, which returns the same four-state shape: `overdue` (red), `due-soon` (amber), `ok` (no badge — rendered "On track"), and `none` (no badge — the schedule's next-due cannot be derived yet, e.g. an `ENGINE_HOURS` schedule on a vehicle whose hour-meter reading is still null; "tracks hours" ≠ "has a reading"). A schedule is measured in one of three dimensions: for **DISTANCE_KM** / **ENGINE_HOURS** the next-due is the schedule's last-service meter anchor plus its interval, compared against the vehicle's current reading (`remaining = nextDue − current`, classified strict-`< 0` overdue / inclusive-`≤ window` due-soon); for **CALENDAR_DAYS** the next-due is `lastServiceAt + interval` days and the case literally IS `complianceBadgeState` — it reuses the shipped helper's UTC-calendar-day date math rather than re-deriving it. All three sit on one shared `thresholdState` core extracted into `compliance.ts`, so the boundary semantics cannot drift between the compliance and maintenance surfaces. The per-dimension "due soon" windows are **provisional** — 500 km / 250 tenths-of-an-hour (25 h) / 30 days (`DEFAULT_SERVICE_DUE_SOON_WINDOWS`) — and are the PO's to finalize (ADR-0037 c7 / "Revisit when"; flagged in `docs/tech-debt.md`, the same owner-level-number pattern as the GPS-retention window). The badge is a passive, read-time visual only — service-reminder *delivery* (email/SMS) is DEC3/ADR-0038, Phase 3, out of scope; INACTIVE schedules are excluded from the surface. The worst-of-N roll-up (the rotation of `worstComplianceState`) **shipped in B5**: `worstServiceState` (the sibling of `serviceScheduleState`, same module) powers the `/service-schedules/due` due-list's per-vehicle worst-of badge, and `rollUpServiceSchedules` (`apps/web/src/lib/dashboard.ts`) feeds the Home dashboard's "Services due" card (overdue + due-soon counts across the up-to-200 ACTIVE schedules it scans). See ADR-0037, the **Compliance badge** entry, the **ServiceSchedule** entry, and the **Badge** entry.

**ICAN** is the Institute of Chartered Accountants of Nepal, the regulator for accountants and the source of accounting standards relevant to FleetCo's eventual accounting integrations.

**IRD** is Nepal's Inland Revenue Department, where tax filings happen. FleetCo's accounting outputs are designed to be compatible with IRD requirements (this is a Phase 4 concern, not a Phase 0 or Phase 1 concern).

**Insurance (तेस्रो पक्ष / comprehensive)** in the Nepali context refers to either third-party insurance (mandatory) or comprehensive insurance (optional). Both have expiries. FleetCo tracks both kinds and triggers reminders before expiry. As of Phase 1 iter 14, persisted on Vehicle as `insurer`, `insurancePolicyNumber`, `insuranceType` (enum `THIRD_PARTY` | `COMPREHENSIVE`), and `insuranceExpiresAt` — a single-row model (one active policy per vehicle); a vehicle carrying both types simultaneously is rare in Nepali heavy-fleet practice and not in scope. Phase 3 reminders will key off `insuranceExpiresAt`.

**OCR** is the Office of the Company Registrar, where company-level filings happen. Distinct from IRD.

**PAN / VAT** are Permanent Account Number and Value Added Tax registration, both required for businesses in Nepal. (This entry is the registration *numbers*; for the VAT *tax* an invoice computes, see the **VAT** entry in the Money vocabulary, and for FleetCo's own supplier PAN as an operator-configured invoice field, see the **Invoice** entry — ADR-0039.)

**Route Permit** is permission for a goods or passenger vehicle to operate on specific routes. Route permits are time-bound and trigger compliance reminders when expiry approaches. As of Phase 1 iter 14, persisted on Vehicle as `routePermitNumber` and `routePermitExpiresAt` (both nullable). Phase 3 reminders will key off `routePermitExpiresAt`.

## Money vocabulary

**NPR / रू** is the Nepalese Rupee, FleetCo's default currency. All money is stored as integer paisa (1 NPR = 100 paisa) to avoid floating-point errors. Display formatting renders rupees from paisa.

**credit note** is the ONLY way to correct an issued invoice (ADR-0039 c5; Program D). Because an issued tax invoice is immutable — its number is a permanent, gapless IRD anti-tamper anchor that is never edited or deleted in place — a mistake (a wrong amount, cancelled work, a return) is corrected by issuing a *separate* document that offsets the original, not by changing it. In FleetCo a credit note is the SAME `Invoice` aggregate with `documentType = CREDIT_NOTE` and an `originalInvoiceId` self-FK back to the invoice it corrects; `InvoicesService.createCreditNote(originalId)` (`apps/api/src/modules/invoices/invoices.service.ts`) copies the original's customer, job, service type, discount, and lines into a new DRAFT (a full reversal by default), and issuing it draws a number from its OWN gapless fiscal-year series — `CRN-2082-83-00001`, independent of the `INV-…` series because the counter is keyed by document type (see **fiscal-year invoice numbering**). Only an ISSUED INVOICE can be credited (a DRAFT, or a credit note itself, → 409). The exact credit-note semantics — full vs. partial, the sign convention, which lines to reverse — are an accountant-verified detail (ADR-0039 c9); v1 ships the seam + the independent numbering. See ADR-0039 and the **Invoice** / **fiscal-year invoice numbering** entries.

**fiscal-year invoice numbering** is FleetCo's gapless, never-reused, anti-tamper invoice / credit-note number, scoped to the Nepali (Bikram Sambat) fiscal year and assigned only at issue (ADR-0039 c4; Program D / D3). The format is `INV-<BS-fiscal-year>-NNNNN` (e.g. `INV-2082-83-00001`) for invoices and `CRN-…` for credit notes; the `2082-83` segment is the BS fiscal year the issue date falls in — Shrawan→Ashadh, mid-July to mid-July (see **BS**), derived via `bsFiscalYear` in `@fleetco/shared`, NOT the Gregorian year. The number is handed out from a dedicated counter ROW (`model InvoiceNumberSequence`, composite PK `(documentType, bsFiscalYear)` + `lastValue`) read `SELECT … FOR UPDATE` inside the same `$transaction` that flips the invoice to ISSUED (`InvoiceNumberingService.nextNumber` / `formatInvoiceNumber`, `apps/api/src/modules/invoices/`). A counter row is deliberately chosen over a Postgres `SEQUENCE`: a SEQUENCE advances even on a rolled-back transaction, which would burn a number and leave a permanent gap, whereas the counter only advances when the issue commits — so the series is gapless even across rollbacks (the IRD anti-tamper expectation). Each `(documentType, fiscal year)` is an independent series, so invoices and credit notes never share a number and a new fiscal year restarts at `00001`. This HARDENS — does not reuse — the `JobsService.nextJobNumber` find-highest+1 scheme (which leaves holes on delete and keys by the Gregorian year). The Nepal numbering rules themselves remain operator/accountant-verify before real billing (ADR-0039 c9). See ADR-0039 and the **Invoice** / **credit note** / **BS** entries.

**Invoice** is FleetCo's first revenue-side aggregate — a compliant Nepal VAT tax invoice built from the Customer → Job → Trip chain (ADR-0039; Program D, a Phase-4 "Money" pull-forward under the ADR-0025 ahead-of-gate pattern). Where every prior money surface tracks what the fleet COSTS (**Fuel log**, **Expense log**), the Invoice is the first that BILLS a customer. Persisted as `model Invoice` in `apps/api/prisma/schema.prisma` (the header) owning many **InvoiceLine** rows: a required `customerId` FK (the buyer + their PAN), a nullable `jobId` FK (the work reference), a `documentType` discriminator (`INVOICE` / `CREDIT_NOTE`), an `InvoiceStatus` (`DRAFT` / `ISSUED` / `CANCELLED`), a nullable `serviceType` (`VEHICLE_HIRE` / `GOODS_TRANSPORT`, which selects the **TDS** rate), an optional `discountPaisa`, the gapless `number` (null until issue), the frozen tax-snapshot columns (subtotal / VAT / gross / TDS / net + the two rate columns — all null until issue), `issuedAt`, the `pdfR2Key`, and the self-FK `originalInvoiceId` (set on a **credit note**). The lifecycle (ADR-0039 c5): a DRAFT is fully mutable and unnumbered; **issuing** is a one-way transition that — in a single `$transaction` — validates (≥ 1 line, a serviceType, FleetCo's supplier PAN configured, R2 configured), computes and FREEZES the **VAT** + **TDS** snapshot, assigns the gapless **fiscal-year invoice number**, renders the PDF and stores it in Cloudflare R2 (the first in-app R2 use), and flips to ISSUED; an ISSUED invoice is immutable and corrected only by a credit note (never an edit or delete). Money is integer paisa throughout (CLAUDE.md), dates render Bikram Sambat (ADR-0031), storage stays ISO/UTC. **Built, not yet billing**: the VAT / TDS rates, the service-type→rate mapping, FleetCo's own supplier PAN, and the exact IRD-required field set are all PROPOSED pending operator/accountant verification before real use (ADR-0039 c9), and a real issue also needs the still-owed production deploy + the operator's R2 / supplier-PAN env. Code: `apps/api/src/modules/invoices/`; web surface `apps/web/src/app/(app)/invoices/`. See ADR-0039 and the **InvoiceLine** / **VAT** / **TDS** / **credit note** / **fiscal-year invoice numbering** entries.

**InvoiceLine** is one billable line on an **Invoice** (ADR-0039 c2; Program D / D4). Persisted as `model InvoiceLine` with a required `invoiceId` FK, nullable `tripId` / `jobId` provenance FKs (which executions the line bills — a manual line, e.g. a flat mobilization fee, references neither), a free-text `description` (BS-dated when built from a job, e.g. "Haul aggregate Kalimati → Pokhara, 2083 Shrawan 3"), an integer `quantity`, an integer `unitPricePaisa`, and a derived integer `lineAmountPaisa = quantity * unitPricePaisa`. `lineAmountPaisa` is ALWAYS derived server-side (`InvoicesService.deriveLineAmountPaisa`, int4-ceiling-guarded → 400) and never client-set (the `.strict()` schema rejects it). The line is the FIRST place a SELLING price is captured — the `Trip` carries no price (ADR-0039 Phase-mismatch §4) — so a line is NOT a read-and-sum of any upstream COST figure (`FuelLog.totalCostPaisa` / `ExpenseLog.amountPaisa` are a different number for a different purpose and are neither read nor restated here). Lines are mutable only on a DRAFT (`addLine` / `updateLine` / `removeLine` / `buildFromJob`; a write on an ISSUED / CANCELLED invoice → 409). Because the schema has no `Trip → Job` link, `buildFromJob` is operator-selected trips + amounts, NOT a server-side traversal (the gap is recorded in `docs/tech-debt.md`). At issue the frozen `subtotalPaisa` is Σ `lineAmountPaisa`. Code: `apps/api/src/modules/invoices/invoices.service.ts`. See ADR-0039 and the **Invoice** entry.

**Lease income** is money earned from lease-takers on contract.

**Per-vehicle cost report** is the read-only aggregation over Fuel logs and Expense logs that surfaces, for a given inclusive date range, each vehicle's total fuel paisa, total expense paisa, combined total paisa, and the row-counts that produced each sum. As of Phase 1 iter 23, exposed as `GET /api/v1/reports/per-vehicle-cost?from=YYYY-MM-DD&to=YYYY-MM-DD&vehicleId=<cuid>` (auth-gated, the `vehicleId` filter is optional), implemented in `apps/api/src/modules/reports/reports.service.ts` as two Prisma `groupBy` aggregations (`fuelLog.groupBy({by: ["vehicleId"], _sum: {totalCostPaisa: true}})` and `expenseLog.groupBy({by: ["vehicleId"], _sum: {amountPaisa: true}})`) merged in memory keyed by vehicleId, joined to the Vehicle table for the registration number, sorted by `totalPaisa` desc with `registrationNumber` ascending as the tiebreaker, and rendered at `/reports/per-vehicle-cost` with a date-range picker that defaults to the current calendar month UTC. The response carries a separate `companyLevel: { expensePaisa, expenseLogCount }` block populated by an `expenseLog.aggregate` over rows where `vehicleId IS NULL` — the iter-22 vehicle-agnostic expenses (quarterly insurance premiums, office stationery) belong in the operator's bill of costs but not under any one vehicle's bucket, so they live in their own row beneath the totals on the report page. Three rules carry through from Expense logs iter 22 unchanged: (1) `amountPaisa` is summed verbatim — no derivation, mirroring how the write path stored it; (2) vehicle-agnostic expenses route to `companyLevel`, never into a per-vehicle row, even when the `vehicleId` filter is set (the company-level block is independent of the filter so the operator sees the context regardless of which vehicle they're inspecting); (3) the date filter applies to the user's reporting `date` column on both FuelLog and ExpenseLog, not `createdAt` — a fill recorded today but dated three weeks ago lands in the window that includes the dated day, not the recorded day. The endpoint uses inclusive-through-end-of-day semantics: `from=2026-02-01&to=2026-02-28` matches every row whose `date` is in `[2026-02-01T00:00:00.000Z, 2026-02-28T23:59:59.999Z]`. Zero-activity vehicles do NOT appear as zero-filled rows; the rows array contains only vehicles with at least one matching FuelLog or ExpenseLog in the window. This is the last Phase-1 slice; Phase-1 content is complete once iter 23 ships.

**Per-vehicle fuel-efficiency report** is the read-only aggregation, sibling to the per-vehicle cost report, that surfaces for a given inclusive date range each vehicle's distance travelled, fuel consumed, and the two efficiency ratios that matter — **km/L** (distance per litre) and **NPR/km** (fuel cost per kilometre) — with a per-vehicle **flag** comparing this window's km/L against the same vehicle's prior equal-length window so a sudden drop reads at a glance. It completes the "fuel efficiency" deliverable named for Reports v1 in `docs/product/roadmap.md` (§"Phase 1 — The Spine") that iter-23 left unshipped (iter-23 shipped the cost half only), landing as **Reports v2** in the Phase-2 window — deferred Phase-1 daily-use polish, the same category as the Home dashboard and the BS-date work, not new Phase-2 scope. Exposed as `GET /api/v1/reports/per-vehicle-efficiency?from=YYYY-MM-DD&to=YYYY-MM-DD&vehicleId=<cuid>` (auth-gated, `vehicleId` optional, reusing `ReportsQuerySchema` verbatim — no new capability and no `ReportsModule` change), implemented in `apps/api/src/modules/reports/reports.service.ts` as four Prisma `groupBy` aggregations (completed-trip distance and fuel consumption, each for the current window and the prior-equal-length baseline) merged in memory by vehicleId and joined to the Vehicle table for the registration number, and rendered at `/reports/per-vehicle-efficiency` with the same date-range picker (defaulting to the current calendar month UTC) and single-vehicle filter the cost report uses. **Distance is the system-of-record figure** — Σ(`endOdometerKm` − `startOdometerKm`) over COMPLETED trips, attributed to the window by `endedAt` (ADR-0003: the Trip aggregate owns distance, and the Trip → Vehicle odometer auto-update already maintains it) — deliberately NOT the fuel-log `odometerReadingKm`, which is nullable and non-monotonic by recorded decision (see the **Fuel log** / **Odometer** entries and `docs/tech-debt.md`, Paid-off) and would inject noise. **Consumption** is Σ `litersMl` and Σ `totalCostPaisa` over in-window FuelLogs, filtered by the operator's reporting `date` (not `createdAt`) on the same inclusive-through-end-of-day semantics as the cost report. **km/L and NPR/km are display ratios** computed at the response edge and never stored (integers stay integers in storage and in the flag math); km/L is null on `insufficient-data`, NPR/km null only when distance is 0 (no divide-by-zero). The **flag** is the `EfficiencyFlag` union `degraded | improved | normal | insufficient-data` (a ≥15% deviation from the baseline km/L flags degraded / improved; a 50 km / zero-litre floor flags insufficient-data), rendered through the shipped `<Badge>` — `degraded` → red "Efficiency down", `improved` → green "Improved", `insufficient-data` → neutral "Not enough data", `normal` → no badge (the absence is the signal). Unlike the cost report there is **no `companyLevel` block**: both inputs (completed trips, fuel logs) are always vehicle-bound. **Honest framing** (carried in the page's coverage note and the PR): with a nullable / non-monotonic fuel odometer and ~15–20% dashboard drift this is a reliable fleet / period-level km/L **trend + exception flag, not a forensic per-fill meter** — a `degraded` flag means "investigate", not "proven theft" — and it self-sharpens once the driver app makes odometer-at-fill routine. See the **Per-vehicle cost report** entry (the sibling it mirrors) and DESIGN.md §"Surfaces" → "Per-vehicle fuel-efficiency report".

**Per-vehicle profit and loss** is the profit and loss attributable to one specific vehicle for a specified period: the trip-revenue for that vehicle minus all costs (fuel, expenses, allocated maintenance, allocated overhead) for that vehicle.

**TDS** (Tax Deducted at Source) is Nepali withholding tax: the customer paying a FleetCo invoice withholds a small percentage of the taxable amount and remits it to the **IRD** on FleetCo's behalf, so FleetCo receives the bill total minus that withholding. On a FleetCo invoice TDS is a MEMO, not a charge — it does NOT change the gross billed to the customer; it is captured so the operator sees the **net receivable** (`netReceivablePaisa = grossPaisa − tdsPaisa`), the cash actually expected once the customer withholds. The rate is selected by the invoice's `serviceType`: **PROPOSED 1.5% (`tdsRateBp = 150`) for vehicle / equipment hire, 2.5% (`250`) for goods transport** (ADR-0039 c3 / c9). It is computed in integer paisa, half-up — `Math.round(taxablePaisa * tdsRateBp / 10000)` — and frozen on the invoice with its rate at issue (the `computeInvoiceTax` snapshot, `apps/api/src/modules/invoices/invoice-tax.ts` / `invoice-tax.constants.ts`). ⚠️ The rates AND the service-type→rate mapping are PROPOSED pending operator / accountant verification against current IRD rules before any real billing (ADR-0039 c9) — the agent does not assume Nepal tax specifics. Whether and how TDS prints on the invoice vs. appears only in FleetCo's cash-reconciliation view is likewise an accountant detail. See ADR-0039, the **VAT** / **Invoice** entries, and the **IRD** / **PAN / VAT** compliance entries.

**Trip revenue** is money billed to a customer for trips on a job.

**VAT** (Value Added Tax) is Nepal's standard consumption tax, added to the taxable value of a FleetCo invoice and billed to the customer (ADR-0039 c3; Program D). Unlike **TDS** (a memo), VAT increases what the customer owes: `grossPaisa = taxablePaisa + vatPaisa` is the amount legally billed. The rate is **PROPOSED 13% (`vatRateBp = 1300`)**, the standard Nepal VAT rate; `vatPaisa = Math.round(taxablePaisa * vatRateBp / 10000)` — integer paisa, half-up, the `FuelLogsService` rounding precedent generalized to a rate (`apps/api/src/modules/invoices/invoice-tax.ts` / `invoice-tax.constants.ts`). The rate is stored in basis points and FROZEN on the invoice at issue, so a future statutory change never rewrites an already-issued document (the anti-tamper / freeze principle, ADR-0039 c3). ⚠️ The 13% rate is PROPOSED pending operator / accountant verification against current IRD rules before any real billing (ADR-0039 c9). Distinct from the **PAN / VAT** entry in the compliance vocabulary, which is about the VAT *registration number* a business holds; this entry is about the VAT *tax* an invoice computes. See ADR-0039, the **TDS** / **Invoice** entries, and the **IRD** / **PAN / VAT** entries.

## Technical vocabulary

**ADMIN** is the most-privileged FleetCo user role (ADR-0028): held by the CEO/owner, an ADMIN can perform every operation OFFICE_STAFF can plus every privileged/sensitive one — raw-GPS-trace query/export (`gps:read-raw` / `gps:export-raw`), GPS-telematics ingestion (`gps:ingest` — ADMIN-only today as a placeholder so the ingestion endpoint can be built and tested before the driver app exists, granted to the **DRIVER role** when it is defined with the driver-app slice, ADR-0029 commitment 11), the observability surface (`observability:read`), and user/role administration (`users:manage` / `roles:assign`). In Phase 2 there is exactly one ADMIN; the role name (rather than literally "CEO") is chosen so a second owner-level account is a data change, not a schema change. Stored as the Prisma `UserRole` enum value `ADMIN` on the better-auth-managed User and ridden on `session.user.role`; assigned only by an explicit privileged action (the `apps/api/scripts/create-user.ts` path, or the migration backfill that promoted the one pre-existing CEO admin), never through the public sign-up / update API (`input: false`). See ADR-0028 and the **RBAC** entry.

**ADR** stands for Architecture Decision Record. An ADR is a short, append-only document capturing one architectural decision: its context, the decision itself, alternatives considered, consequences, and the signal that would prompt revisiting the decision. ADRs live in `docs/architecture/decisions/`. See `template.md` for the format.

**Action card** is the compact card the `/chat` transcript renders for each tool the agent ran — tool name, a status badge (succeeded / failed / denied / flagged), latency, an args-detail disclosure, and, once the write tools populate them, one contextual deep-link to the affected record. Its defining property is that it is rendered from the SERVER's `AgentAction` record, never from the model's text (ADR-0043 commitment 4c — the honesty rule): the model cannot misreport a write it actually executed, because the transcript's account of what ran is the audit row, not the model's prose. That guarantee is narrower than it once read: a wholly fabricated write (no tool call at all) leaves no action card to contradict it, which is exactly what the **Ungrounded-claim guard** entry closes — see ADR-0043's 2026-07-05 append-only annotation for the incident. The A8 `/agent/activity` ledger renders the same `AgentAction` rows in a cross-user table. See ADR-0043, the **AI agent (FleetCo)**, **Pre-image**, and **Ungrounded-claim guard** entries, and `docs/design/DESIGN.md` §"Agent chat" / §"Agent activity".

**Agent attachment** is the image a user attaches to an agent chat message (ADR-0044): one photo per turn (JPEG/PNG/WEBP, ≤10 MB, client-downscaled), stored in R2 under `agent-attachments/<conversationId>/…` through the shared `ObjectStorage` port and rowed as `AgentAttachment`. An attachment is TRANSCRIPT content, not a document archive: it is Tier-2-handled, dies with the 180-day transcript prune (row cascade + object delete), and never links to the entity it helped create — the structured row the user confirms into existence is the durable record (ADR-0045 rule 2). Extraction happens on a separate no-tools vision seam; see the **Document extraction** entry.

**AI agent (FleetCo)** is the in-process chat agent ADR-0043 commits to: a curated, typed tool registry over the existing module services (list/get reads, the two report tools, and a bounded set of reversible creates/updates), driven by a provider-agnostic LLM client and surfaced as a chat page in the admin web. Three structural facts define it. It is **in-process** — tools call the domain services directly through their exported public interfaces (the same cross-module seam CLAUDE.md sanctions), so there is no HTTP hop and validation, RBAC, and row-scoping are inherited rather than re-implemented. It **executes as the requesting user** (ADR-0043 commitment 1) — every tool run carries the human's real identity: `createdById` is the human's user id, the capability ceiling is the human's role, and agent-ness is recorded in the audit spine (the `AgentAction` model), never hidden behind a synthetic machine actor. And its registry is **curated, not the whole API** — structurally absent are every delete, invoice issue/cancel/credit-note, raw GPS trace access, and user/role management (commitment 3). Access is gated by the `agent:use` capability, ADMIN-only in v1. See ADR-0043 and the **DeepSeek** entry.

**Badge** is FleetCo's status-indicator UI primitive (ADR-0031; `apps/web/src/components/ui/badge.tsx`): a CVA-driven `<span>` — status, never an action (DESIGN.md anti-pattern #2; for inline actions use a low-emphasis button) — mirroring `button.tsx`'s `cva`/`cn` shape. Its variants (`warning` amber, `error` red, `success` emerald, `info` blue, `neutral` zinc) bind only to the `color.status.*` / surface / text tokens already declared in DESIGN.md → `globals.css` `@theme`, so the Badge introduces no new design token and the design-token-drift test is untouched. Per DESIGN.md §"Status badges" the hue is always paired with a text label — the hue is recognition, the label is meaning. Its first consumer is the **compliance badge** on the Vehicle detail page. See ADR-0031 and the **Compliance badge** entry.

**better-auth** is the TypeScript-first authentication library FleetCo uses across the NestJS API and Next.js admin web. Sessions are database-stored and opaque-cookie-identified rather than JWT-based. The native driver app authenticates with better-auth's **bearer** plugin (ADR-0034): the same database session, presented as an `Authorization: Bearer <token>` header (the token stored in `expo-secure-store`) instead of a cookie — an additive change, the web's cookie flow is unchanged, and `AuthGuard` resolves both through one `getSession({ headers })` call. See ADR-0015 and ADR-0034.

**BullMQ** is a Redis-backed job queue library for Node.js. FleetCo uses BullMQ for background work that should not block the API: GPS ingestion (in Phase 2), notifications, report generation. It is wired into the NestJS API via `@nestjs/bullmq` through a thin `@Global()` `QueueModule` (`apps/api/src/modules/queue/queue.module.ts`) that owns a dedicated ioredis connection — `maxRetriesPerRequest: null`, deliberately separate from `RedisService`'s `:1` health-probe connection — and the default job options (bounded retry with exponential backoff; the `failed` set as the de-facto dead-letter); named queues are owned by their feature modules — `gps-ingest` (the telematics ingestion path, ADR-0029 T3) and `traces-prune` (the GPS-ping retention prune, ADR-0029 T4: a daily keyed repeatable that hard-deletes pings past the retention window — see the **Retention (telematics)** entry). See ADR-0020 (substrate choice) and ADR-0029 (integration shape).

**Capability (permission token)** is the unit of authorization in FleetCo's RBAC (ADR-0028 commitment 4): a coarse, operation-class token — e.g. `vehicles:*`, `reports:read`, `gps:ingest`, `gps:read-raw` — that a controller declares it requires via `@RequirePermission(...)`, decoupled from role names through the single hardcoded `ROLE_CAPABILITY_MAP`. Capabilities are deliberately coarse ("may this role perform this class of operation?"), never per-record (row-level scoping is deferred to the DRIVER role, ADR-0028 c9). The `*` in an operational token like `vehicles:*` is part of the token NAME — the whole vehicles operation-class as one token — not a glob expanded at match time; enforcement is exact set-membership. The closed `Capability` union and the map live in `apps/api/src/modules/auth/permissions.ts`, so a typo'd token is a compile error, not a silently-open route. See ADR-0028 and the **RolesGuard** entry.

**CNG (Continuous Native Generation)** is Expo's posture, adopted for the driver app by ADR-0033 c1/c2: the native `android/` project is *generated* from `app.json` + config plugins by `npx expo prebuild`, is gitignored and disposable, and is never hand-edited — the mobile mirror of "the Prisma schema is truth; applied migrations are never edited." A hand-edit would be silently overwritten by the next prebuild, so native configuration changes land only in `app.json`/config plugins, where they diff cleanly. See ADR-0033 and `docs/runbook/dev-setup.md` §"Building the driver app (local prebuild)".

**Command palette** is FleetCo's ⌘K keyboard-navigation surface (ADR-0040; `apps/web/src/components/ui/command.tsx`, mounted in `components/app-shell/app-shell.tsx`): a modal, type-to-filter list of the app's destinations, opened with `⌘K` / `Ctrl+K` (a global keydown listener) or the quiet top-bar "Search… ⌘K" affordance, that `router.push`-es to the selected page and closes. It is backed by **cmdk** — the one new top-level dependency the Phase-1 T7 ticket adds (React-19 verified at add time), the library shadcn-ui's `Command` wraps — re-pointed from shadcn's dead `:root` aliases to FleetCo's `@theme` semantic utilities exactly as `popover.tsx` documents (the consumption guard enforces it), with its modal wrapper built on the already-installed `radix-ui` Dialog (no separate dialog dependency). **DeepSeek** is the hosted LLM provider behind FleetCo's AI agent (ADR-0043). The agent's chat completions run against DeepSeek's OpenAI-compatible API via raw `fetch` — zero new dependencies, behind a provider-agnostic client seam — with the model name env-configured (`DEEPSEEK_MODEL`, defaulting to `deepseek-v4-flash`). See ADR-0043 and the **AI agent (FleetCo)** entry.

**Document extraction** is the ADR-0044 vision step that turns an agent attachment into typed fields: a single schema-constrained pass on the `VisionExtractor` seam — run by a **self-hosted OCR/document-parsing model on FleetCo's own infrastructure** (Box B; candidate `baidu/Unlimited-OCR` as a version-pinned sidecar container, the Traccar precedent), so **document images never leave the VPS**. The pass deliberately carries NO tool registry, so text inside an image can never reach a tool-calling context (the structural prompt-injection defense), and returns a `.strict()` `DocumentExtraction` (document type, date with its AD/BS calendar, vendor, integer-paisa amounts, receipt number, bounded raw text). A pure mapping helper converts the extraction into candidate create-tool args; the agent then PROPOSES the record field-by-field and the write fires only on the user's confirming next turn, through the existing create tools — the extraction's content reaching DeepSeek only as normal Tier-2 turn text, like dictation. An unset extractor endpoint binds a mock extractor — the feature's kill switch, mirroring `DEEPSEEK_API_KEY`. See ADR-0044 and the **Agent attachment** entry.

**DRIVER role** is the third FleetCo `UserRole` enum value. It was reserved-but-undefined in the RBAC slice (ADR-0028 c1) and is **defined as of ADR-0034** (the driver-app auth slice) as the login identity a driver authenticates with on the mobile app, via better-auth **bearer** tokens (see **better-auth**). Its capability set is granted **incrementally** per ADR-0034 c5's hard rule (no DRIVER _write_ capability enters the `ROLE_CAPABILITY_MAP` without its row-level own-record predicate landing in the **same change**): D1 shipped it **empty** (a DRIVER could authenticate and read `/me` but was inert against every route), and **D2 grants exactly `trips:*` + `fuel-logs:*`** atomically with the **service-layer own-record predicate** (`DriverScopeService.resolveOwnDriverId`) that scopes a driver to their own data — a driver reads / starts / stops only their **own** trips (reused `PATCH /trips/:id`) and logs fuel only against their **own** trip (`POST /fuel-logs`), and may **not** create or delete either; a foreign record 404s (existence-hiding) and an unlinked DRIVER session fails **closed** (403). Because the Phase-1 trips/fuel-logs routes are auth-guarded but NOT RolesGuard-gated, that predicate is the **sole** enforcement (ADR-0034 c7), not a route gate — so the two caps record the grant rather than guarding the route. The remaining two caps of c6's lean set, `gps:ingest` (D4/D5, ADR-0035) and `gps:read-derived` (D6), stay **deferred** until their own scopes land. A driver login links 1:1 to the fleet **Driver** aggregate via the nullable-unique `Driver.userId` FK (ADR-0034 c4), the binding the own-data scoping reads through. Distinct from the **Driver** aggregate (the person / operational record): the DRIVER role is who-may-do-what, the Driver aggregate is data, and `userId` joins them. Created via `apps/api/scripts/create-user.ts … DRIVER` (admin-only; the `input:false` privilege-escalation defense preserved) — that script creates the login only; **as of 2026-07-05**, `POST` / `DELETE /api/v1/drivers/:id/login-link` (`DriversService.linkLogin`/`unlinkLogin`, gated by `users:manage`) is the write path that actually sets `userId`, closing ADR-0034 c8's open item ("the exact UX of linking an existing Driver to a new login is the slice's to pin") — until this, the FK existed and `DriverScopeService` read it, but nothing in the app ever wrote it. See ADR-0028 and ADR-0034.

**Idempotent** describes an operation that produces the same result whether called once or many times. Idempotency matters for retries: an idempotent operation can be safely retried after a network failure without risking duplicate effects.

**Legal status transition** is the rule, enforced at the service layer rather than at the database, that constrains which status changes are permitted on a state-machine field. As of Phase 1 iter 9, FleetCo applies legal-status-transition rules in two places: (a) the Vehicle aggregate's `status` field follows the "retirement transition" rule documented in `apps/api/src/modules/vehicles/vehicles.service.ts` (status→RETIRED/SOLD auto-stamps `retiredAt`; status→ACTIVE/IN_MAINTENANCE clears it); (b) the Trip aggregate's `status` field follows the transition matrix `PLANNED → IN_PROGRESS → COMPLETED`, with `CANCELLED` reachable from any non-terminal state and self-transitions permitted (the matrix lives in `apps/api/src/modules/trips/trips.schemas.ts:TRIP_STATUS_TRANSITIONS` and the predicate in `isLegalTripStatusTransition`). An illegal transition is rejected by the service with `BadRequestException` (HTTP 400) before the write reaches Prisma. Cross-field rules layered on top — IN_PROGRESS requires `startedAt` and `startOdometerKm`; COMPLETED requires all four of `startedAt`, `endedAt`, `startOdometerKm`, `endOdometerKm` with `endedAt >= startedAt` and `endOdometerKm >= startOdometerKm` — are validated against the merged shape (current row + patch) inside the service, because Zod's `superRefine` only sees the partial PATCH body. The matrix is not encoded as a database constraint because the constraint is a small per-aggregate detail that changes faster than schema migrations are comfortable to ship; future aggregates (e.g., a Maintenance ticket) that introduce their own state machines will mirror this pattern.

**Mailer** is FleetCo's thin transactional-email seam (ADR-0038 commitment 1), at `apps/api/src/modules/notifications/mailer.ts`: a one-method `Mailer` abstract class — `send({ to, subject, text, html }): Promise<MailerSendResult>` — that is the ONLY place in the codebase that knows the email vendor, exactly as `nepali-date.ts` wraps `nepali-date-converter` and `common/wkt.ts` centralizes the WKT builder. Two implementations sit behind it: **`ResendMailer`** (`resend.mailer.ts`) calls the pinned `resend` SDK over a single outbound HTTPS request (the provider chosen at ADR-0038 acceptance; the API key `RESEND_API_KEY` is a Tier-1 secret read through the typed `env`, never logged); **`MockMailer`** (`mock.mailer.ts`) records sends in an in-memory array and never touches the network. The `NotificationModule` DI factory resolves `Mailer` to `ResendMailer` only when `RESEND_API_KEY` is set (production with a verified sending domain) and to `MockMailer` everywhere else (dev / test / CI), so the API never reaches the network outside production and tests assert against the recorded sends. A later swap to Postmark or SMTP touches this one implementation, never the scan or the digest. See ADR-0038 and the **NotificationModule** / **reminder digest** entries.

**NepaliDate** is FleetCo's web-side Bikram Sambat date renderer (ADR-0031): the pure `formatNepaliDate(iso, { format })` formatter in `apps/web/src/lib/nepali-date.ts` and the thin `<NepaliDate iso=… format=… />` server component that wraps it (`apps/web/src/components/nepali-date.tsx`), mirroring the `money.ts`/`formatNpr` display-formatter pattern. It converts a stored ISO/UTC date to a BS-prominent Latin-transliteration string with a muted Gregorian parenthetical — `format="both"` (default) → `2083 Jestha 6 (2026-05-20)`, `"bs"` → BS only (Gregorian carried in a `title` tooltip), `"en"` → Gregorian only — em-dashing null / unparseable input exactly like `formatNpr`. Conversion VALUES come from the verified `nepali-date-converter` library (never fabricated — DESIGN.md anti-pattern #12; where DESIGN.md's illustration and the library disagree, e.g. the BS year of 2026-05-20, the library wins and the doc discrepancy is noted), but the 12 month names render from a FleetCo-owned `BS_MONTHS` array so a library update cannot silently change a spelling (the library differs at three: Baisakh→Baishakh, Asar→Ashadh, Aswin→Ashwin). Two correctness invariants: the conversion is done on the **UTC calendar day** (the converter is handed `new Date(y, m, d)` built from the ISO's UTC components — this library reads a Date via server-local getters, so a deadline's BS day stays independent of server timezone), and BS is **render-time only** — never stored, never round-tripped through the API (DESIGN.md §"BS calendar"). The BS date-picker INPUT widget is the sibling **NepaliDatePicker** component (ADR-0032), which reuses this module's `BS_MONTHS` array and UTC-calendar-day rule; compliance-reminder delivery remains Phase 3. See ADR-0031 and the **BS** / **NepaliDatePicker** entries.

**NepaliDatePicker** is FleetCo's web-side Bikram Sambat date-PICKER input (ADR-0032; `apps/web/src/components/nepali-date-picker.tsx`), the input twin of **NepaliDate**: a hand-built Radix-Popover month grid over the already-installed `nepali-date-converter` (no new top-level dependency — ADR-0032 commitment 1), built on the in-tree `popover.tsx` shadcn primitive. Its trigger shows the current BS + Gregorian date (via `<NepaliDate format="both">`); the popover renders a BS month grid with each day's Gregorian date underlaid in muted corner text, arrow controls to switch BS month/year, today outlined in `color.accent.primary`, and the selected day filled (DESIGN.md §"BS calendar"). The load-bearing contract (ADR-0032 commitment 3): it accepts and emits the EXACT ISO/UTC date string (`YYYY-MM-DD`) the native `<input type="date">` did — `<NepaliDatePicker value={iso|null} onChange={(iso|null)} />` — so it is a drop-in at the input layer and the form Zod schemas, server actions, and API contracts are untouched. The conversion/grid math is the pure, unit-tested helpers in `apps/web/src/lib/nepali-date-picker.ts` (the component is the thin shell); it reuses the **NepaliDate** module's `BS_MONTHS` anti-drift array and the **UTC-calendar-day rule** (`bsDayToIso` reads AD calendar integers off the converter, so a picked day never drifts by server/local timezone). First adopted on the Vehicle compliance date inputs (B1); rolled across the remaining forms, filters, and the report date-range in B2/B3. See ADR-0032 and the **NepaliDate** / **BS** entries.

**NetInfo** (`@react-native-community/netinfo`) is the React Native community library the driver app uses to observe device connectivity (sanctioned by name in ADR-0035 commitment 3). The D5 sync runtime (`apps-mobile/driver/src/sync-runtime.ts`) subscribes to its listener and treats the offline→online transition as a drain trigger: the moment the network comes back, buffered fixes in the **outbox** are pushed without waiting for the periodic tick. Its `isConnected` can be `null` before the first determination — FleetCo treats only an explicit `false` as offline (optimism costs one failed POST and a backoff; pessimism would strand the queue). It reports link-level connectivity, not API reachability — a failed drain still classifies and backs off on its own (`src/sync.ts`). See the **Outbox** entry and ADR-0035.

**NotificationLog** is FleetCo's reminder-delivery dedup ledger and audit trail (ADR-0038 commitment 5), `model NotificationLog` in `apps/api/prisma/schema.prisma` (Tier 3 operational data). It records every reminder the daily scan dispatched, keyed by the unique tuple **`(subjectType, subjectId, reminderKind, state, occurrenceKey)`** — the send-once-per-lapse mechanism: a lapse is emailed once when it crosses into `expiring-soon` / `expired` (compliance) or `due-soon` / `overdue` (maintenance), never re-emailed daily, and a renewal (a new `occurrenceKey` = the new due date) re-arms the reminder. `subjectType` / `reminderKind` / `state` are stored as OPEN STRINGS (not enums) precisely so C3's `SERVICE_SCHEDULE` subject and `SERVICE` kind extended the ledger with no migration. The row also carries the `recipient` (Tier-2 PII, comma-joined when more than one; never logged), the `sentAt` instant, and the provider's `providerMessageId`. It doubles as the ADR-0013 audit value — "we notified about that lapse on date X" — surfaced READ-ONLY by the C4 reminder-history view (`GET /api/v1/notification-logs`, `apps/web/src/app/(app)/notification-logs/`, ADMIN-only via the `notifications:read` capability); the ledger is append-only, written ONLY by the scan→send worker, never by an HTTP client. See ADR-0038 and the **NotificationModule** / **reminder digest** entries.

**NotificationModule** is FleetCo's compliance/maintenance reminder-DELIVERY subsystem (ADR-0038, `apps/api/src/modules/notifications/`), modelled on the ADR-0029 T4 RetentionModule. It owns the `notifications` BullMQ queue (per-feature ownership, ADR-0029 commitment 2) and a single keyed `upsertJobScheduler` daily scan (`NotificationService implements OnApplicationBootstrap`, cron `0 1 * * *` ≈ 06:45 Nepal, registered idempotently at boot) whose concurrency-1 `@Processor` worker, on each run, classifies the due/overdue sources via the SHARED `@fleetco/shared` classifiers (the SAME `complianceBadgeState` (ADR-0031) and `serviceScheduleState` (ADR-0037) the web badges use — the load-bearing drift guard, ADR-0038 c6), diffs them against the **NotificationLog** (send-once), and enqueues one SEND job per recipient (the scan→send split, so a transient provider blip retries the one send, not the whole scan). Each send delivers one §Voice Bikram-Sambat **reminder digest** via the **Mailer** to the ADMIN users' emails plus the `NOTIFICATION_RECIPIENTS` override (ADR-0038 c7). It exposes NO HTTP surface — it is a scheduled job (ADR-0038 c2); the read/audit surface lives in the sibling `NotificationLogsModule` (`apps/api/src/modules/notification-logs/`). See ADR-0038 and the **Mailer** / **NotificationLog** / **reminder digest** / **reminder_delivery** entries.

**OFFICE_STAFF role** is the second live FleetCo user role (ADR-0028): a trusted employee who does operational data entry. It holds the operational capability floor — the Phase-1 CRUD aggregates (`vehicles:*`, `drivers:*`, `customers:*`, `jobs:*`, `trips:*`, `fuel-logs:*`, `expense-logs:*`), `reports:read`, and the **derived** GPS views (`gps:read-derived`: live-location map, geofence status, trip route summary) — but NOT raw-GPS-trace export, NOT GPS-telematics ingestion (`gps:ingest` is ADMIN-only today, DRIVER-held later — ADR-0029 commitment 11), NOT observability, and NOT user/role administration (those are ADMIN-only). The threat model is "limit the blast radius of a compromised office-staff session", not "office staff is hostile". OFFICE_STAFF is the least-privileged LIVE role and the default for every new account (the `defaultValue` on the better-auth `role` field); ADMIN is granted only by an explicit privileged action. Stored as the Prisma `UserRole` enum value `OFFICE_STAFF`. See ADR-0028 and the **RBAC** entry.

**Outbox** is the driver app's durable on-device buffer for captured GPS fixes (ADR-0035 commitment 2, the D5 slice): a single **SQLCipher**-encrypted SQLite table (`outbox_pings` — native shell `apps-mobile/driver/src/outbox-sqlite.ts`, pure policy `src/outbox.ts`) that every captured fix is written into BEFORE any network send — capture never POSTs directly; the sync runtime is the single sender. It drains oldest-first in bounded chunks (`DRAIN_BATCH` ≤ the server's 1000-ping batch cap, a tested cross-constant pin) and deletes rows only after the server's 202 — **at-least-once** delivery (a crash between the 202 and the delete re-sends a harmless duplicate; ADR-0035 c4 declined dedup). A server-judged 4xx drops that chunk with a count-only warning (retrying rejected rows forever is wrong; 401/429 stay retryable), while transport failures retain rows under a capped, jittered exponential backoff. Two provisional bounds keep it finite: a 48 h age-out on capture time and a 20 000-row cap dropping oldest first. Inserts are per-row autocommit — deliberately NO wrapping transaction, after the D5 E2E showed a transaction left open by a paused backgrounded JS thread wedging every later statement on the shared connection. Coordinates never appear in logs (ADR-0027 c5, mobile mirror). See **SQLCipher**, **NetInfo**, and ADR-0035.

**Phone link (WhatsApp)** is FleetCo's phone-number → user mapping for the WhatsApp agent channel (ADR-0046), `model AgentPhoneLink` in `apps/api/prisma/schema.prisma`. It maps a canonical E.164 phone (`@unique`) to a `User` — the identity every WhatsApp-driven agent turn executes as (`createdById` is that human, the capability ceiling is their role; ADR-0021 / ADR-0043 c1) — plus a nullable `conversationId` pointer (FK → `AgentConversation`, `onDelete: SetNull`, set on the first turn) that makes "the user's WhatsApp conversation" deterministic (the conversation model has no channel discriminator), a `verifiedAt`, and an active/deactivated state for opt-out. It is the **authorization chokepoint, not mere plumbing**: the inbound webhook has no `RolesGuard` and `AgentService.runTurn` trusts its caller-supplied actor, so the resolver + processor enforce `agent:use`/ADMIN **at turn time** and a privileged provisioning script (no public endpoint) refuses to link a non-ADMIN user; an unmapped or deactivated number fails **closed** (no turn). `phoneE164` is Tier-2 PII. Day-1 the table holds the PO's number → the PO's ADMIN user; widening `agent:use` beyond ADMIN is ADR-0043's own separate decision. See ADR-0046 and the **WhatsApp channel** / **Twilio** entries.

**PostGIS** is a PostgreSQL extension for geospatial data and queries. FleetCo uses PostGIS for geofences, GPS pings, and any spatial queries. The `postgis` extension was enabled in the baseline migration (ADR-0020), but no table held a geometry until the GpsPing aggregate landed it (ADR-0029 T2): each ping stores native Float `latitude`/`longitude` as the canonical, Prisma-native columns plus a `GENERATED ALWAYS AS (ST_SetSRID(ST_MakePoint(longitude, latitude), 4326)) STORED` `geometry(Point, 4326)` column with a GIST index, derived by the database (so it cannot drift from the floats) and used only by spatial queries. The geometry column is declared `Unsupported(...)` in the Prisma schema (Prisma tracks it for migrate-diff but never selects/inserts it); the generated-column SQL and the GIST index are hand-authored in the migration because Prisma expresses neither. The first spatial **queries** against that geometry landed with ADR-0029 T5 — geofencing by `ST_Contains` / `ST_DWithin` in parameterized `prisma.$queryRaw`, confined to the derived read layer (`gps:read-derived`); ordinary coordinate reads stay on the type-safe native-Float Prisma path, which is the whole point of the hybrid representation. FleetCo's first `geometry(Polygon, 4326)` landed with the Geofence aggregate (**ADR-0030**): the SAME hybrid pattern with one structural change — a polygon has no scalar canonical form, so its geometry is `GENERATED ALWAYS AS (ST_GeomFromText("boundaryWkt", 4326)) STORED` from a canonical TEXT column rather than from scalar floats (`ST_GeomFromText(text, 4326)` is IMMUTABLE, which is what lets it drive a `GENERATED … STORED` column). A service-level `ST_IsValid` gate rejects a self-intersecting ring that the generated column would otherwise accept and `ST_Contains` would misclassify. See ADR-0029 (the hybrid storage representation, commitment 8), ADR-0030 (the polygon extension), and the **Geofence** entry.

**Pre-image** is the prior state of a row, captured as JSON into `AgentAction.previousJson` immediately BEFORE an agent update executes (ADR-0043 commitments 3/4b) — what makes every agent update cheaply reversible. It is the RAW row, not the model-redacted shape: a pre-image exists to be restored from, so it must be faithful, and redaction is a model-context boundary (what the LLM may see), not an operator boundary (what the ADMIN may read). It is **Tier 2** — never logged (the pino `*.previousJson` redact path), never placed on a span, never sent to the model; it surfaces only to the authorized ADMIN in the `/agent/activity` Details disclosure. It is null for reads, creates, and failed updates (a failed update changed nothing, so there is nothing to undo). The undo procedure that reads it lives in `docs/runbook/agent-operations.md`. See ADR-0043 and the **AI agent (FleetCo)** / **Action card** entries.

**RBAC (Role-Based Access Control)** is a permission model in which permissions are grouped into roles, and users are assigned roles. RBAC was deferred until Phase 2 in FleetCo, when office staff are introduced as a user role distinct from the CEO. As of Phase 2 (ADR-0028, implemented) the concrete model is two live roles — **ADMIN** (the CEO/owner; full access) and **OFFICE_STAFF** (operational data entry plus derived views) — with **DRIVER** reserved-but-undefined (no capabilities yet). Roles map to coarse **capability** tokens through a single hardcoded in-tree table (`apps/api/src/modules/auth/permissions.ts`); the role rides `session.user.role` (a better-auth `additionalFields` field, `input: false`); and a **RolesGuard** composing the existing `AuthGuard` enforces `@RequirePermission(...)` / `@RequireRole(...)` decorators (403 for an authenticated-but-unauthorized caller, distinct from `AuthGuard`'s 401). Enforcement is opt-in restriction — an undecorated route stays open to any authenticated user. See ADR-0028.

**reminder digest** is the single §Voice email the reminder scan sends (ADR-0038 commitment 7), rendered by the pure `renderReminderDigest(items)` function in `apps/api/src/modules/notifications/digest.ts` (digest data → `{ subject, text, html }`, unit-testable like every FleetCo formatter). It batches the day's newly-due / overdue / expiring items — grouped by domain (vehicle compliance, service schedules) — into one factual message rather than one email per item (the anti-spam choice, ADR-0038 "A daily digest vs. per-event emails"), with every date rendered in Bikram Sambat via `formatNepaliDate` (`@fleetco/shared`) so the operator reads the calendar they think in, in the DESIGN.md §Voice register: state the fact, no exclamation, no apology (subject e.g. "FleetCo — 3 items need attention", never "⚠️ URGENT!!!"). A scan that finds nothing due renders and sends nothing — no "all-clear" email. See ADR-0038 and the **NotificationModule** / **Mailer** entries.

**reminder_delivery** is FleetCo's reminder-send-success SLI (ADR-0038 commitment 8, the indicator ADR-0011's "Revisit when" anticipated by name), defined in `apps/api/src/common/sli.ts` (`SLI_REMINDER_DELIVERY` + `ReminderDeliverySignal` + `isReminderDeliveryGood` + `buildReminderDeliverySignal`), mirroring `trip_creation_success` exactly. The VALID EVENT is a reminder **send attempt** — a `Mailer.send` call against a real recipient — so an idle scan that finds nothing to send is NOT a valid event and does not inflate the denominator (the "count attempts, not non-attempts" discipline the trip SLIs follow). The send worker logs one line per attempt tagged `sli: "reminder_delivery"` with `sli_good` (true on the provider's ack, false on a thrown-and-rethrown send error) and, on failure, `error_kind` = the exception CLASS NAME only — never `err.message`, which can embed the Tier-2 recipient address. Target 99.0% over the rolling 28-day window (ADR-0011's catalog); the constant only instruments it, and there are no sends to measure until the channel runs in production. See ADR-0038, ADR-0011, and the **NotificationModule** entry.

**@RequirePermission / @RequireRole** are the two NestJS route decorators that declare a route's RBAC requirement (ADR-0028 commitment 5), defined in `apps/api/src/modules/auth/decorators.ts` via `SetMetadata` (metadata keys `fleetco:rbac:require-permission` / `fleetco:rbac:require-role`). `@RequirePermission(capability)` is the preferred form — capability-based, so adding a role does not require editing every controller; `@RequireRole(role)` is the coarser shortcut for genuinely role-scoped surfaces (observability, user admin). The RolesGuard reads the metadata via NestJS's `Reflector` and allows or denies. A route with neither decorator stays open to any authenticated user (opt-in restriction). See ADR-0028 and the **RolesGuard** entry.

**Retention (telematics) / `traces-prune`** is FleetCo's data-lifecycle enforcement for raw GPS pings. ADR-0027 commitment 3 classifies a raw, full-resolution GPS trail as Tier 5 (surveillance-grade) and requires a **short** raw-trace retention window — provisionally 90 days, the PO's number to finalize at ADR-0027 acceptance — after which raw pings are pruned (hard-delete, the default) or aggregated to a Tier-3 derived summary (the preferred end-state, a later ticket). As of ADR-0029 T4 the window is enforced by the `traces-prune` BullMQ job, owned by a dedicated **RetentionModule** (`apps/api/src/modules/retention/`): a single keyed repeatable (bullmq's `upsertJobScheduler`, id `traces-prune-daily`, cron `0 3 * * *`) registered idempotently at boot, whose `@Processor` worker (concurrency 1) hard-deletes `GpsPing` rows whose fix `timestamp` is older than the window (`GPS_PING_RETENTION_DAYS`, currently 90). The prune basis is the device fix `timestamp` (not storage `createdAt`; switching is a one-line change for late offline flushes); the run is traced with only safe attributes (deleted count, cutoff, window) and never any GPS coordinate — and the ADR-0026 span-scrub seam (`apps/api/src/observability/span-scrub.ts`, the OTLP-egress twin of the pino `redact` GPS denylist) now backstops every exported span by deleting the GPS denylist keys regardless of which span set them. See ADR-0027 (the Tier-5 retention policy), ADR-0029 (the job shape, commitment 12), and the **BullMQ** / **GPS ping (telematics)** entries.

**RolesGuard** is FleetCo's RBAC enforcement guard (ADR-0028 commitment 5), at `apps/api/src/modules/auth/roles.guard.ts`. It is a NestJS `CanActivate` that **composes** the existing `AuthGuard`: controllers use `@UseGuards(AuthGuard, RolesGuard)` in that order, so `AuthGuard` runs first and attaches the session, and RolesGuard reads `session.user.role` already on the request with no extra `getSession` call. It resolves the route's required capability/role from the `@RequirePermission` / `@RequireRole` metadata (via `Reflector.getAllAndOverride`), checks it against the role's set in `ROLE_CAPABILITY_MAP`, and: no decorator → allow (opt-in restriction); decorator satisfied → allow; otherwise → `ForbiddenException` (HTTP 403, deliberately distinct from `AuthGuard`'s 401 `UnauthorizedException`). It realizes ADR-0021's prediction that `AuthGuard` would "grow into `RolesGuard` or compose with one" — the integration shape is untouched; only the guard surface grows. See ADR-0028.

**Session-based authentication** is an authentication model in which a server issues a session record (stored in a database or cache), and the client holds an opaque cookie that references the session. Distinct from token-based authentication (such as JWT), in which the client holds a self-contained signed token. FleetCo uses session-based authentication because the admin web does not need cross-domain token-based auth, sessions are revocable without a key-rotation event, and the server-side session record is the natural place to attach role and permission state as Phase 2's RBAC arrives. As of Phase 2 (ADR-0028), that attach point is realized: the user's RBAC `role` rides `session.user.role` (a better-auth-managed `additionalFields` field), so the RolesGuard reads it at enforcement time with no extra query. See ADR-0015 and ADR-0028.

**shadcn-ui** is the base design system FleetCo customizes from: a copy-paste-not-install component library built on Radix UI primitives and Tailwind CSS, shipping a Vercel-derived aesthetic. Component implementations are copied into `apps/web` and owned in-tree rather than imported as a versioned dependency; FleetCo customizations (NPR formatting, BS-calendar widgets, Devanagari fallback, ERP density) layer on top. See ADR-0016 and ADR-0007 for the design-folder discipline that consumes from it.

**Soft delete** is the pattern of marking a row as deleted (typically via a `deletedAt` timestamp) and filtering it out of queries, rather than physically removing it. Soft delete preserves referential integrity for related rows and supports recovery from accidental deletion. As of Phase 1 iter 9, FleetCo does NOT implement soft delete: the Vehicle, Driver, and Trip DELETE endpoints hard-delete the row. Iter 8 introduced the Trip aggregate, which references Vehicle and Driver by id with FK delete policy `Restrict`; iter 9 closed the loop by mapping the resulting Prisma `P2003` (FK constraint violation) to `ConflictException` (HTTP 409) with a friendly per-aggregate message naming the count of referencing trips (e.g. `"Cannot delete vehicle: 3 trips reference this vehicle."`). Iter 10 added the operator-facing affordance: when `DeleteVehicleDialog` / `DeleteDriverDialog` surface that 409, the inline error block also renders a "View N trips" deep-link to the filtered trips list (`/trips?vehicleId=<id>` or `/trips?driverId=<id>`) so the operator can pivot from "I can't delete this" to "let me see what's blocking" without leaving the dialog mental model. The block-when-referenced policy is the project's chosen approach over true soft delete: a vehicle or driver with historical trips is a different entity from a never-existed one, and the audit value of `deletedAt` is lower than the simplicity of "the row exists or it doesn't." A follow-up that adds `deletedAt` to any aggregate would be a separate ADR, prompted by an explicit audit-trail or recovery requirement.

**SQLCipher** is the encrypted-SQLite build protecting the driver app's **outbox** database at rest (ADR-0035 commitment 2: GPS traces are Tier-2 data on a phone that leaves the yard). FleetCo enables it through expo-sqlite's `useSQLCipher: true` config-plugin option (a gradle property baked at prebuild — part of why D5 requires the custom dev build, ADR-0033 c7) and keys each connection with `PRAGMA key = "x'<hex>'"` as the first statement — the RAW-key form (32 random bytes from expo-crypto, hex-encoded), not a passphrase, so opens skip PBKDF derivation and no text-encoding ambiguity exists. The key is generated once and lives in expo-secure-store (Android-Keystore-backed — the same store as the auth session credential). Key loss is handled by honest reset: if the stored key no longer opens the file, the outbox is deleted and recreated with a count-unknown warning (it is a buffer, not a source of truth — the cost is a trail gap, ADR-0035 c1's gap tolerance). Proven at rest in the D5 E2E: the database file pulled off the device is refused by plain `sqlite3` ("file is not a database"). See **Outbox** and ADR-0035.

**Traccar** is the open-source (Apache-2.0) GPS tracking server FleetCo runs as its device gateway (ADR-0042 c2, ticket M6): a version-pinned official Docker image (`traccar/traccar` in `docker-compose.prod.yml`) that terminates the native binary tracker protocols (Teltonika Codec 8 on TCP 5027 — the one internet-facing device port; Caddy proxies HTTP only), decodes positions into its own separate `traccar` database in the existing Postgres container (c3, created once per `docs/runbook/traccar.md`), and forwards each decoded position as JSON (`forward.type=json`, payload `{position, device}` — the shape the M5 adapter contract test pins) to the API's machine-authenticated ingest endpoint `POST /api/v1/telematics/ingest/traccar` (`IngestKeyGuard` + the Tier-1 `INGEST_API_KEY` header, ADR-0042 c5). FleetCo treats Traccar as infrastructure, not a second app: its committed config is the non-secret `deploy/traccar.xml` with secrets env-injected (`CONFIG_USE_ENVIRONMENT_VARIABLES`), its web UI binds to loopback only (SSH tunnel), it auto-registers nothing (known IMEIs only, c9), and its own store and application log are Tier-5-bearing — pruned to the ≤90-day window by `deploy/traccar-prune.sh` and bounded by docker log caps per c8 (Traccar 6.x has no built-in retention). See the TrackerDevice register (ADR-0042 c6 / M4) and **GPS ping (telematics)**.

**Twilio** is the WhatsApp Business Solution Provider (BSP) behind FleetCo's WhatsApp agent channel (ADR-0046): the intermediary that delivers a user's WhatsApp message to FleetCo's inbound webhook and relays the agent's reply back. FleetCo integrates it **zero-dependency** — inbound `X-Twilio-Signature` verification (`base64(HMAC-SHA1(TWILIO_AUTH_TOKEN, canonical-URL + ascending-key-sorted params))`, compared with `timingSafeEqual`, verified against the configured `TWILIO_WEBHOOK_URL` constant, never a header-reconstructed URL — ADR-0046 c2) and outbound sends (a raw `fetch` POST to the Messages API, Basic auth) both use `node:crypto` + `fetch`, exactly as the agent talks to DeepSeek (ADR-0043 c2), behind the `WhatsAppSender` seam. `TWILIO_ACCOUNT_SID` / `TWILIO_AUTH_TOKEN` are Tier-1 secrets and OPTIONAL: unset ⇒ `MockWhatsAppSender` + the `TwilioSignatureGuard` fails closed (503) ⇒ the channel is off (the `RESEND_API_KEY` kill-switch idiom). Twilio — and Meta/WhatsApp behind it — is a **content sub-processor** that sees both inbound message content and outbound replies: the new bidirectional egress class ADR-0046 names in its ADR-0013 amendment. The official `twilio` SDK was declined (it needs the same configured webhook URL, so it buys little). See ADR-0046, the **WhatsApp channel** entry, and — for the machine-ingress guard template — **Traccar**.

**Ungrounded-claim guard** is the deterministic, same-turn check (`apps/api/src/modules/agent/ungrounded-claim-guard.ts`) that closes a gap ADR-0043 commitment 4c's "the model cannot misreport what it did" guarantee left open: that guarantee holds for a REAL write with wrong details (a genuine `AgentAction` row contradicts it), not for a wholly fabricated one (no row exists to contradict anything). Found 2026-07-05 when a live model claimed a demo driver was created — a fabricated name, phone, dates, and a fake-but-plausible `/drivers/<id>` link — with zero matching tool call; confirmed against the dev database (zero driver rows, zero `create_driver` actions ever). The guard checks a turn's final reply (the one with no more `tool_calls`) against that turn's own `AgentAction` history before the reply reaches the user: write-completion language (a curated verb list, negation-guarded so "couldn't add" doesn't count) combined with either zero successful creates/updates this turn (Rule A) or a cited id that matches none of them (Rule B) gets flagged — a server-authored system notice is appended (the model's original text survives verbatim; the transcript must not ventriloquize the model any more than the model may misreport the server) and a `status: "flagged"` row lands on the same `AgentAction` ledger, visible on `/agent/activity` immediately. It is a regex heuristic, not comprehension — see the module's own doc comment for what it does not close. See ADR-0043's 2026-07-05 append-only annotation, and the **Action card** / **AI agent (FleetCo)** entries.

**Vertical slice** is the development pattern FleetCo uses: each unit of work is one user-facing workflow built end-to-end (schema, migration, service, API, UI, test) rather than one layer of the system across all features. See ADR-0006.

**WhatsApp channel** is FleetCo's two-way WhatsApp interface to the AI chat agent (ADR-0046; the `whatsapp` API module lands across tickets W2–W5): a user messages the company's WhatsApp number, **Twilio** POSTs an inbound webhook, and the existing agent (ADR-0043) runs the turn **as the resolved human behind the verified number** (the **phone link**) and replies in the same thread — reusing the agent's tool registry, audit spine, autonomous-write posture, budgets, and redaction contract with **no new capability**. Structurally it is a dedicated delivery module on the `NotificationModule` precedent: its own `TwilioSignatureGuard` (the Traccar machine-ingress posture — fail-closed 503, `timingSafeEqual`, guard-only, 202-fast), its own `whatsapp-inbound` BullMQ queue + in-process processor (dedup by `MessageSid` → resolve the phone link → enforce `agent:use`/ADMIN → get/create the user's WhatsApp conversation → `runTurn` → render → send), its own `WhatsAppSender` seam (the `Mailer` pattern; Twilio / Mock), and its own `WhatsAppMessageLog` delivery + idempotency ledger (`@@unique` on the inbound `MessageSid`, the retry/replay defense). The reply is a **plain-text rendering of the server's `AgentAction` records** (the ADR-0043 c4c honesty rule over text — the ADMIN's only in-thread catch for a bad autonomous write), never the model's prose. **Text now; receipt photos are built but inert** behind the paused image intake (`AGENT_OCR_URL` unset). All production use gates on the M1 deploy (the webhook needs a public 24/7 HTTPS endpoint). See ADR-0046 and the **AI agent (FleetCo)** / **Phone link (WhatsApp)** / **Twilio** entries.

## Memory architecture vocabulary

**Episodic memory** in the project's memory architecture is memory of specific events and decisions. It lives in `docs/architecture/decisions/`.

**Operational memory** is memory of measurements and operational records taken on regular cadences. It lives in `docs/operations/`. The DORA metrics file is updated weekly; SLO compliance reports are computed monthly.

**Perceptual memory** is memory of what things should look and feel like. It lives in `docs/design/`.

**Procedural memory** is memory of how things are done. It lives in `docs/runbook/` and `BOOTSTRAP.md`.

**Prospective memory** is memory of intentions held for future execution. It lives in `docs/product/roadmap.md`, `docs/CURRENT_PHASE.md`, and `docs/tech-debt.md`.

**Semantic memory** is memory of what words mean. It lives in this glossary.

**Substrate** in the memory architecture refers to the medium in which memory is stored. FleetCo's substrate is the filesystem under git version control, with files organized in a hierarchical directory structure. See `docs/architecture/memory-architecture.md`.

## Delivery operating model vocabulary

**API availability SLI** is the first of FleetCo's two SLIs (ADR-0011): the percentage of HTTP requests to the API that return a 2xx or 3xx status within 500 milliseconds, measured against a 99.0 percent objective over a rolling 28-day window. As of Phase 1 (ticket T_SLI1) the signal is instrumented at the pino-http request-completion layer in `apps/api/src/app.module.ts`: every completed request's log line carries `sli: "api_availability"`, `http_status`, `response_time_ms`, and `sli_good` (true when `http_status < 400 && response_time_ms <= 500`), which a future 28-day report aggregates into the indicator. The threshold and the good/bad predicate live in `apps/api/src/common/sli.ts` as `SLI_LATENCY_BUDGET_MS` (500) and the pure `isAvailabilityGood(status, ms)` helper, shared with the performance budget (T_PERF) so the committed budget and the emitted signal cannot drift. The signal is emitted at the pino-http layer rather than a Nest interceptor so it covers 100 percent of requests, including the raw better-auth handler at `/auth/*` that an interceptor would miss; it is wired as both `customSuccessObject` (2xx/3xx/4xx) and `customErrorObject` (5xx and thrown errors) because pino-http routes server errors down a separate path. It is Tier-4 metadata per ADR-0013 — status code, duration, and a boolean only, never the URL or query string (which can carry Tier-2 PII that the pino `redact` list does not cover for `req.url`). See ADR-0011 and the **SLI** / **SLO** entries below.

**Change failure rate** is one of the four canonical DORA metrics. It is the percentage of production deployments that result in a customer-impacting failure (detected via Sentry error spike or manual rollback within 24 hours of the deploy). FleetCo's Phase 1 target is under 15 percent measured monthly. See ADR-0010.

**Deployment frequency** is one of the four canonical DORA metrics. It is the rate at which the team deploys to production, measured weekly as deploys per working day. FleetCo's Phase 1 target is at least one production deploy per working day on average.

**DORA** stands for DevOps Research and Assessment, the research program (now part of Google) that produces the annual State of DevOps reports and that defined the four canonical delivery metrics plus the rework rate. See ADR-0010 for FleetCo's commitment to measure these metrics.

**Error budget** is the quantity of permitted bad events over an SLO measurement window, computed as 100 percent minus the SLO. For FleetCo's 99.0 percent availability SLO over a 28-day rolling window, the error budget is approximately 7 hours of unavailability per 28 days. See ADR-0011.

**Failed deployment recovery time** is one of the four canonical DORA metrics. It is the time between a detected deploy failure and the deploy that restored service. FleetCo's Phase 1 target is under 2 hours at the 90th percentile.

**Lead time for changes** is one of the four canonical DORA metrics. It is the median time from PR merge to production deploy completion. FleetCo's Phase 1 target is under 24 hours.

**Rework rate** is the fifth DORA metric, formalized in 2025. It is the percentage of merged PRs that have to be revisited within 14 days because they did not work as intended. FleetCo's Phase 1 target is under 25 percent measured monthly.

**SBOM** stands for Software Bill of Materials. It is a structured list of all dependencies in a software artifact, used for supply-chain security analysis and vulnerability response. FleetCo generates a CycloneDX SBOM for every production build. See ADR-0012.

**SEV1, SEV2, SEV3** are the three incident severity levels in FleetCo's classification. SEV1 is a customer-impacting incident affecting core business operations (trip creation, fuel logging). SEV2 is a customer-impacting incident affecting non-core modules or a degraded but functional core module. SEV3 is a non-customer-impacting incident (logging gap, backup verification failure, minor UI defect). Each level has defined response expectations. See ADR-0011 and `docs/runbook/incident-response.md`.

**SLI (Service Level Indicator)** is a measurement of something users care about, typically expressed as the ratio of good events to valid events. FleetCo's two SLIs are API availability (the percentage of HTTP requests returning 2xx or 3xx within 500 milliseconds) and trip-creation success (the percentage of trip-creation operations completing end-to-end without user-visible error). See ADR-0011.

**SLO (Service Level Objective)** is a target on an SLI over a defined time window. FleetCo's SLOs are 99.0 percent for both indicators over a rolling 28-day window. See ADR-0011.

**SRE (Site Reliability Engineering)** is the body of practice developed at Google for running production software systems with measurable reliability targets and operational discipline. FleetCo adopts a minimum viable SRE framework appropriate to a one-person operation. See ADR-0011.

**trip-creation success SLI** is the second of FleetCo's two SLIs (ADR-0011): the percentage of trip-creation operations that complete without a user-visible error, measured against a 99.0 percent objective over a rolling 28-day window. As of Phase 1 (ticket T_SLI2) the signal is emitted in `TripsController.create` (`apps/api/src/modules/trips/trips.controller.ts`): the create call is wrapped in try/catch, and every attempt logs a line carrying `sli: "trip_creation_success"` and `sli_good` — `true` on success, `false` on a thrown error (which is then rethrown so the HTTP response is unchanged) — plus, on the failure path, `error_kind` set to the exception's class name. A future 28-day report filters log lines where `sli` equals `"trip_creation_success"` and computes the indicator as the share with `sli_good === true`. The `sli` tag value lives in `apps/api/src/common/sli.ts` as `SLI_TRIP_CREATION_SUCCESS`, alongside the API-availability vocabulary, so the two signals share one source of truth. The SLI is scoped to the post-validation create operation: `ZodValidationPipe` rejects a malformed body with HTTP 400 before the controller method body runs, so request-shape rejections are not counted against this indicator. It is Tier-4 metadata per ADR-0013 — only the exception's class name is logged as `error_kind`, never `err.message` (the trips foreign-key errors deliberately embed the literal vehicle or driver id, which is Tier-3 operational data). See ADR-0011 and the **SLI** / **SLO** / **API availability SLI** entries above.

## Data and security vocabulary

**Data classification tier** is a categorization of data by sensitivity. FleetCo uses four tiers per ADR-0013. Tier 1 is administrative and authentication data. Tier 2 is personally identifying information. Tier 3 is operational business data. Tier 4 is non-sensitive metadata. The tier determines logging, encryption, and access rules.

**Recovery Point Objective (RPO)** is the maximum acceptable data loss in a disaster, measured as a time window. FleetCo's RPO is 24 hours, meaning the maximum acceptable data loss is one day's worth of data.

**Recovery Time Objective (RTO)** is the maximum acceptable time for service restoration after a disaster. FleetCo's RTO is 4 hours for the API and admin web, and 24 hours for the driver app (in Phase 2).

**Secrets scanning** is the practice of scanning code (and git history) for credential patterns that should never be committed. FleetCo uses GitHub native secrets scanning per ADR-0012.

**Static Application Security Testing (SAST)** is automated analysis of source code for security vulnerabilities. FleetCo uses Semgrep with the OWASP and security-audit rulesets per ADR-0012.

**Software Composition Analysis (SCA)** is the practice of identifying and tracking known-vulnerable dependencies in a project. FleetCo uses Dependabot for SCA per ADR-0012.

**Tier 1 / Tier 2 / Tier 3 / Tier 4** see "Data classification tier" above.
