# Architecture Overview

This document describes the shape of the FleetCo system. It is one of the two core architecture documents in the repository. The other is `memory-architecture.md`, which describes how the repository holds the project's memory across time. Together they answer the two architectural questions that matter most for FleetCo: how does the running software fit together, and how does the project itself persist as a continuous entity. A reader who internalizes one without the other has only half the picture.

## Shape

FleetCo is a modular monolith. This means it is a single deployable application with a single database, but the code inside it is organized into strict modules with explicit contracts between them. The reasoning for choosing this architecture over microservices and over an unstructured monolith is recorded in ADR-0001. The short version is that microservices add distributed-system overhead that a one-person team cannot absorb and that AI coding agents handle poorly across service boundaries, while an unstructured monolith decays into unmaintainability fast, especially when AI agents are doing most of the writing. The modular monolith captures the deployment simplicity of a monolith with the architectural discipline of services. If any single module ever genuinely outgrows the shared deployment, it can be extracted into its own service later, with the data we will then have to justify the extraction.

The system has three logical layers plus two satellite producers. The top layer is `apps/web`, the Next.js admin interface — built through Phase 2, with the first production deploy still owed (the current gate state lives in the as-of block of `docs/CURRENT_PHASE.md`). The middle layer is `apps/api`, the NestJS backend, organized into modules under `apps/api/src/modules/`; each module owns its own data and exposes a public service interface to other modules. The bottom layer is the data plane: PostgreSQL with PostGIS holding all primary data, Redis holding sessions, cache, and the BullMQ job queues, and Cloudflare R2 holding object storage — whose first real objects are issued invoice PDFs (ADR-0039), with document scans and vehicle photos anticipated. The two satellites: the driver app exists at `apps-mobile/driver` as a standalone Expo project deliberately **outside** the pnpm workspace (ADR-0033) — it cannot import `packages/shared`, so API shapes are hand-mirrored — and hardware GPS trackers feed the api through a self-hosted Traccar gateway container (ADR-0042). In production the whole system is one Docker Compose stack on a single VPS: api, web, postgres, redis, caddy, traccar (`docker-compose.prod.yml`, ADR-0014).

## The central aggregate: Trip

The most consequential domain decision in FleetCo is that the Trip is the central aggregate. This is recorded in ADR-0003. A vehicle without a trip is a record on a shelf; a vehicle's trips are its operational life. Fuel attaches to a trip or to a vehicle for non-trip fuel. Driver compensation is computed from trips. Customer revenue is invoiced from trips on jobs (with one honest caveat: the `Trip`→`Job` foreign key does not exist yet, so invoice lines are operator-selected trips rather than job-traversed — a tracked entry in `docs/tech-debt.md`). Maintenance triggers are driven by trip-accumulated kilometers and engine hours. GPS evidence belongs to a trip. Get the Trip schema right and most other modules become CRUD operations on top of it. Get the Trip schema wrong and every subsequent module accumulates compensating complexity. This is why the build order treated Trip not as a CRUD entity to bang out quickly but as the spine that the rest of the system hangs from.

## Modules

The authoritative module list is the directory `apps/api/src/modules/` — as of Phase 2 it holds seventeen domain modules and four infrastructure modules. Domain: `auth`, `vehicles`, `drivers`, `customers`, `jobs`, `trips`, `fuel-logs`, `expense-logs`, `reports`, `geofences`, `telematics` (GPS ingestion + the tracker register), `maintenance` (service schedules + records), `invoices`, `notifications` (reminder delivery), `notification-logs` (its audit ledger), `retention` (data-lifecycle prune jobs), and `agent` (the AI chat agent, ADR-0043). Infrastructure: `prisma`, `redis`, `queue`, `health`.

Two per-module notes earn their lines. The `auth` module owns sessions, the current-user concept, and live role-based access control (ADR-0028): three roles — ADMIN, OFFICE_STAFF, DRIVER — mapped to coarse capability tokens in the single hardcoded table at `apps/api/src/modules/auth/permissions.ts`, enforced by an AuthGuard + RolesGuard chain on every operational controller, with service-layer own-record scoping for drivers. The `reports` module is read-only and produces aggregates over the others; it never writes.

## Module rules

A module exposes a service interface (such as `VehiclesService`). Nothing else is public. Other modules import the service interface only; they never reach into another module's repositories, schema files, or internal helpers. Cross-module writes happen via service calls through NestJS dependency injection. Read-only cross-aggregate consumers (reports, retention, notifications) query other modules' tables directly through the global Prisma service — aggregation is their whole job, and forcing every read into a service round-trip would add cost without protection. An event bus is a future option, not a present mechanism; nothing in the system reacts to events today. The Prisma schema is one file, but each module owns its own tables.

## Stack choices

The choices, with their justifying ADRs, are summarized here for a reader who wants the picture at a glance. TypeScript end-to-end with strict mode is the language choice (ADR-0005), because one language across the stack maximizes shared types between API and UI and AI agents have particularly strong training mass on this combination. NestJS is the backend framework (ADR-0018) because its module pattern is built into the framework's core; we get the modular boundaries we want without inventing them. Next.js with the App Router (ADR-0019) is the frontend framework because it is mature, supports server-rendered components, and shares an ecosystem with the rest of the TypeScript world. PostgreSQL with the PostGIS extension is the primary database (ADR-0002) because the data is fundamentally relational, money is involved which makes ACID transactions non-negotiable, and PostGIS handles geospatial queries natively without requiring a separate store. Prisma is the ORM and migration tool (ADR-0020) because its generated types align with our TypeScript-everywhere choice and its migration discipline aligns with our refusal to edit the database by hand. Redis with BullMQ (ADR-0020) handles cache and background jobs because the API needs to stay stateless and responsive while slow work happens elsewhere. Cloudflare R2 handles object storage because it is S3-compatible and has zero egress fees. The authentication library is `better-auth` (ADR-0015), used across the NestJS API, the Next.js admin web, and the Expo driver app with database-stored sessions rather than JWTs. The base design system that `docs/design/DESIGN.md` derives from is `shadcn-ui` (ADR-0016), a copy-paste-not-install component library on Radix and Tailwind customized for FleetCo's NPR/paisa formatting, BS-calendar widgets, and ERP-density needs. Sentry handles error tracking and pino (via nestjs-pino) handles structured logging (ADR-0018), with OpenTelemetry tracing layered on Sentry's provider and an env-gated OTLP exporter (ADR-0024). Caddy terminates TLS and reverse-proxies the single VPS (ADR-0014). Traccar is the version-pinned GPS-tracker gateway (ADR-0042). The AI chat agent is an in-process tool layer over the module services, DeepSeek-first behind a provider-agnostic raw-fetch client (ADR-0043).

## Delivery and reliability posture

The project's delivery and reliability posture is anchored in three additional ADRs that complement the architectural choices above. ADR-0010 commits us to measuring the four canonical DORA metrics plus rework rate, with explicit Phase 1 targets that are realistic for a one-person shop with an AI agent (at least one production deploy per working day, lead time under 24 hours at the median, change failure rate under 15 percent monthly, failed deployment recovery time under 2 hours at the 90th percentile, rework rate under 25 percent monthly). ADR-0011 commits us to a minimum viable reliability framework with two SLIs (API availability and trip-creation success) at 99.0 percent over a 28-day rolling window, an error-budget policy that halts feature work when the budget is exhausted, and a three-level severity classification (SEV1, SEV2, SEV3) with defined response expectations. ADR-0012 commits us to a security baseline that runs on every PR (Dependabot, GitHub native secrets scanning, Semgrep with OWASP and security-audit rulesets, action-version pinning, CycloneDX SBOM generation). The measurement clocks start at the first production deploy, which is still owed — the SLI instrumentation and DORA scaffolding are in place, waiting for production traffic.

## Data posture

The data posture is anchored in ADR-0013, which established a four-tier data classification scheme, amended by ADR-0027 to five tiers. Tier 1 is administrative and authentication data that never appears in source code or logs. Tier 2 is personally identifying information of drivers and customers that is encrypted at rest and not logged by default (agent chat transcripts are classified here too, per ADR-0043). Tier 3 is operational business data (vehicles, trips, fuel, expenses, and derived location events like geofence status) that is logged only at the level of operational events. Tier 4 is non-sensitive metadata that may be logged in full. Tier 5 is raw GPS location data — the most sensitive class: short retention, never exported, stripped from logs and trace spans by twin redaction seams that must stay in sync (`apps/api/src/observability/log-redact.ts` and `apps/api/src/observability/span-scrub.ts`). Retention is set at the lifetime of the customer relationship plus three years for Tier 2 and Tier 3 data. The Recovery Point Objective is 24 hours; the Recovery Time Objective is 4 hours for the API. Fields are marked in the Prisma schema by tier and logging is configured so that Tier 2, 3, and 5 data is not captured by default.

## Scalability posture

We do not build for scale we do not have. An honest read of the foreseeable scale puts FleetCo at approximately 100 vehicles and approximately 500 trips per day even in a wildly successful version. A single PostgreSQL instance on a modest VPS handles that load for years. Premature scaling is the largest known killer of greenfield projects, and we deliberately avoid it. What we do build in from day one, because it is cheap, is statelessness in the API (so that running multiple instances behind a load balancer is a configuration change rather than a refactor), Redis-backed background jobs on four BullMQ queues (`gps-ingest`, `traces-prune`, `transcript-prune`, `notifications`), and connection pooling. The highest-volume table, `gps_ping`, is a plain table whose growth is controlled by short raw-trace retention — the `traces-prune` job hard-deletes pings past a provisional 90-day window (ADR-0027/ADR-0029) — rather than by partitioning. Read replicas, sharding, separate analytics warehouses, table partitioning, and TimescaleDB come only when measurements demand them, probably not in year one and possibly not at all.

## Stability posture

Several disciplines are non-negotiable from commit one. TypeScript strict mode runs everywhere. Migrations are versioned and never edited after applying. Tests cover module boundaries meaningfully (not 100 percent, but at the seams where modules contract with each other). CI runs on every PR with the security baseline jobs from ADR-0012. Sentry is wired up from Phase 0, before the first real bug happens. The encrypted-backup scripts are committed (`deploy/backup.sh`, `deploy/restore.sh`) and the restore procedure must be tested within two weeks of the first deploy per the RPO and RTO commitments in ADR-0013 — that drill is still owed alongside the deploy itself. Half-built surfaces stay unrouted and unlinked until they are ready; nothing ships behind a pretend switch. The DORA metrics from ADR-0010 are recorded weekly and the SLOs from ADR-0011 computed monthly once production traffic exists. None of these is optional. Skipping any of them weakens every other one, because the disciplines reinforce each other and a project that is "almost rigorous" is in practice a project that is rigorous nowhere.
