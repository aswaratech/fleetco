# Architecture Overview

This document describes the shape of the FleetCo system. It is one of the two core architecture documents in the repository. The other is `memory-architecture.md`, which describes how the repository holds the project's memory across time. Together they answer the two architectural questions that matter most for FleetCo: how does the running software fit together, and how does the project itself persist as a continuous entity. A reader who internalizes one without the other has only half the picture.

## Shape

FleetCo is a modular monolith. This means it is a single deployable application with a single database, but the code inside it is organized into strict modules with explicit contracts between them. The reasoning for choosing this architecture over microservices and over an unstructured monolith is recorded in ADR-0001. The short version is that microservices add distributed-system overhead that a one-person team cannot absorb and that AI coding agents handle poorly across service boundaries, while an unstructured monolith decays into unmaintainability fast, especially when AI agents are doing most of the writing. The modular monolith captures the deployment simplicity of a monolith with the architectural discipline of services. If any single module ever genuinely outgrows the shared deployment, it can be extracted into its own service later, with the data we will then have to justify the extraction.

The system has three logical layers. The top layer is `apps/web`, the Next.js admin interface that the CEO uses every day, with a React Native driver app planned for Phase 2 that will share types with the web app through the shared package. The middle layer is `apps/api`, the NestJS backend, organized into modules under `apps/api/src/modules/`. Each module owns its own data, exposes a public service interface to other modules, and emits events that other modules can subscribe to. The bottom layer is the data plane: PostgreSQL with PostGIS holding all primary data, Redis holding sessions and cache and the BullMQ job queue, and Cloudflare R2 holding object storage like Bluebook scans, insurance PDFs, and vehicle photos.

## The central aggregate: Trip

The most consequential domain decision in FleetCo is that the Trip is the central aggregate. This is recorded in ADR-0003. A vehicle without a trip is a record on a shelf; a vehicle's trips are its operational life. Fuel attaches to a trip or to a vehicle for non-trip fuel. Driver compensation is computed from trips. Customer revenue is invoiced from trips on jobs. Maintenance triggers are driven by trip-accumulated kilometers and hours. GPS evidence belongs to a trip. Get the Trip schema right and most other modules become CRUD operations on top of it. Get the Trip schema wrong and every subsequent module accumulates compensating complexity. This is why the Phase 1 build order treats Trip not as a CRUD entity to bang out quickly but as the spine that the rest of the system hangs from.

## Modules in the Phase 1 set

The modules built during Phase 1, in build order, are the following. The `auth` module owns sessions and the current-user concept; in v1 it has only one user, the CEO, but the module is structured to admit role-based access control later without a rewrite. The `vehicles` module owns vehicle records, including Bluebook metadata, insurance metadata, and registration. The `drivers` module owns driver records, including license metadata and employment status. The `customers` module owns customer records (companies and individuals who hire vehicles, distinct from lease-takers who rent vehicles for their own operation). The `jobs` module owns the contracts with customers, where one job consists of one or more trips. The `trips` module owns the central aggregate. The `fuel` module owns fuel logs that link to trips or vehicles. The `expenses` module owns expense logs (toll, repair, miscellaneous) that link to trips or vehicles. The `reports` module is read-only and produces aggregates over the others; it never writes.

## Module rules

A module exposes a service interface (such as `VehiclesService`) and events (such as `VehicleCreated`). Nothing else is public. Other modules import the service interface only; they never reach into another module's repositories, schema files, or internal helpers. Cross-module writes happen via service calls. Cross-module reactions happen via events. The Prisma schema is one file, but each module owns its own tables; other modules query other modules' tables only through that module's service. Reports are read-only and may query any table directly, because their job is aggregation and breaking the read-isolation rule for reports would force every read into a service round-trip with no benefit.

## Stack choices

The choices, with their justifying ADRs, are summarized here for a reader who wants the picture at a glance. TypeScript end-to-end with strict mode is the language choice (ADR-0005), because one language across the stack maximizes shared types between API and UI and AI agents have particularly strong training mass on this combination. NestJS is the backend framework (ADR-0018) because its module pattern is built into the framework's core; we get the modular boundaries we want without inventing them. Next.js with the App Router (ADR-0019) is the frontend framework because it is mature, supports server-rendered components, and shares an ecosystem with the rest of the TypeScript world. PostgreSQL with the PostGIS extension is the primary database (ADR-0002) because the data is fundamentally relational, money is involved which makes ACID transactions non-negotiable, and PostGIS handles geospatial queries natively without requiring a separate store. Prisma is the ORM and migration tool (ADR-0020) because its generated types align with our TypeScript-everywhere choice and its migration discipline aligns with our refusal to edit the database by hand. Redis with BullMQ (ADR-0020) handles cache and background jobs because the API needs to stay stateless and responsive while slow work happens elsewhere. Cloudflare R2 handles object storage because it is S3-compatible and has zero egress fees, which matters when serving Bluebook scans and vehicle photos to the admin web. The authentication library is `better-auth` (ADR-0015), used across the NestJS API and Next.js admin web with database-stored sessions rather than JWTs because the admin UI does not need cross-domain token-based auth. The base design system that `docs/design/DESIGN.md` derives from is `shadcn-ui` (ADR-0016), a copy-paste-not-install component library on Radix and Tailwind whose Vercel-derived aesthetic is customized for FleetCo's NPR/paisa formatting, BS-calendar widgets, Devanagari font fallback, and ERP-density needs. Sentry handles error tracking and pino (via nestjs-pino) handles structured logging (both per ADR-0018).

## Delivery and reliability posture

The project's delivery and reliability posture is anchored in three additional ADRs that complement the architectural choices above. ADR-0010 commits us to measuring the four canonical DORA metrics plus rework rate from the moment Phase 1 ships, with explicit Phase 1 targets that are realistic for a one-person shop with an AI agent (at least one production deploy per working day, lead time under 24 hours at the median, change failure rate under 15 percent monthly, failed deployment recovery time under 2 hours at the 90th percentile, rework rate under 25 percent monthly). ADR-0011 commits us to a minimum viable reliability framework with two SLIs (API availability and trip-creation success) at 99.0 percent over a 28-day rolling window, an error-budget policy that halts feature work when the budget is exhausted, and a three-level severity classification (SEV1, SEV2, SEV3) with defined response expectations. ADR-0012 commits us to a security baseline that runs on every PR (Dependabot, GitHub native secrets scanning, Semgrep with OWASP and security-audit rulesets, action-version pinning, CycloneDX SBOM generation). Together these three ADRs establish that the bootstrap is not just an architectural skeleton but a delivery operating model with measured throughput, defined reliability, and a security posture appropriate for the data it holds.

## Data posture

The data posture is anchored in ADR-0013, which establishes a four-tier data classification scheme. Tier 1 is administrative and authentication data that never appears in source code or logs. Tier 2 is personally identifying information of drivers and customers that is encrypted at rest and not logged by default. Tier 3 is operational business data (vehicles, trips, fuel, expenses) that is encrypted at rest and logged at the level of operational events. Tier 4 is non-sensitive metadata that may be logged in full. Retention is set at the lifetime of the customer relationship plus three years for Tier 2 and Tier 3 data. The Recovery Point Objective is 24 hours, the Recovery Time Objective is 4 hours for the API and 24 hours for the driver app. The privacy posture commits to data minimization, encryption at rest and in transit, and honoring data subject requests within 30 days. The Phase 1 schema work explicitly takes the data classification into account, with fields marked in the Prisma schema by tier and logging configured so that Tier 2 and Tier 3 data is not captured by default.

## Scalability posture

We do not build for scale we do not have. An honest read of the foreseeable scale puts FleetCo at approximately 100 vehicles and approximately 500 trips per day even in a wildly successful version. A single PostgreSQL instance on a modest VPS handles that load for years. Premature scaling is the largest known killer of greenfield projects, and we deliberately avoid it. What we do build in from day one, because it is cheap, is statelessness in the API (so that running multiple instances behind a load balancer is a configuration change rather than a refactor), Redis-backed background jobs (so that slow work like report generation does not block the API), connection pooling (so that the database is not surprised by connection spikes), and partitioned tables for the eventual time-series concerns (GPS pings in Phase 2 will land in tables partitioned by month). Read replicas, sharding, separate analytics warehouses, and TimescaleDB for time-series come only when measurements demand them, probably not in year one and possibly not at all.

## Stability posture

Several disciplines are non-negotiable from commit one. TypeScript strict mode runs everywhere. Migrations are versioned and never edited after applying. Tests cover module boundaries meaningfully (not 100 percent, but at the seams where modules contract with each other). CI runs on every PR with the security baseline jobs from ADR-0012. Sentry is wired up from Phase 0, before the first real bug happens. Daily encrypted database backups are configured and the restore procedure is tested within the first two weeks of operation per the RPO and RTO commitments in ADR-0013. Feature flags allow half-built modules to ship behind a switch. The DORA metrics from ADR-0010 are recorded weekly. The SLOs from ADR-0011 are computed monthly. None of these is optional. Skipping any of them weakens every other one, because the disciplines reinforce each other and a project that is "almost rigorous" is in practice a project that is rigorous nowhere.
