# Memory Architecture

This document explains how FleetCo holds its memory across time. It is a peer of `overview.md` in the architecture documentation, because the project's memory architecture is as much an architectural concern as the modular monolith, the database choice, or the central aggregate. A reader who understands the system architecture without understanding the memory architecture will be able to write code that runs but will not be able to participate in the project's continuity over time. Every long-running session in this repo, whether by a human or an AI agent, depends on the memory architecture functioning correctly.

## Why this document exists

FleetCo is built by one person, the company's CEO, working with AI coding agents that have no memory across sessions. The agents read the repo, do work, write code, and then forget everything. The CEO has human memory, which is unreliable across the months and years a project of this scope spans. Neither the CEO nor any individual agent session is, by itself, capable of holding the project's identity over time. The repo is the only candidate for that role. This document explains how the repo discharges that responsibility.

The argument that follows is grounded in real intellectual traditions, named here in case readers want to follow the references. The extended mind hypothesis, introduced by Andy Clark and David Chalmers in 1998, argues that an external artifact can be part of cognition rather than merely a tool used by cognition, when the artifact is reliably available, easily retrievable, and trusted. Distributed cognition, developed by Edwin Hutchins, argues that complex cognition often happens across people, instruments, procedures, and artifacts together rather than in any single mind. Organizational memory theory, developed by Walsh and Ungson in 1991, argues that organizations remember things through retention bins outside of individuals — through structure, processes, archives, culture — and that organizations whose retention bins are weak forget when personnel change. The mediating-artifacts strand running from Vygotsky through activity theory argues that artifacts do not merely store information; they shape the thought of those who use them. These four strands together establish that memory held outside the individual mind is real memory, that it has properties shaped by its substrate, and that the choice of substrate is a choice with real consequences.

## The substrate: filesystem under git

The repo holds project memory as plain text files (mostly markdown, with some YAML, JSON, and code) organized in a hierarchical directory structure inside a git repository. This substrate has properties that we depend on, listed here so that the rules in `CLAUDE.md` and elsewhere can be understood as derived from these properties rather than imposed arbitrarily.

The first property is radical durability. Plain text files in a known path on a filesystem are the most durable form of digital memory currently available. They do not depend on any particular software vendor, on any particular SaaS subscription, on any particular database product. If the company that makes our editor disappears, the memory is unaffected. If GitHub disappears, the memory is on every machine that has ever cloned the repo. If a markdown viewer becomes unavailable, the underlying text is still readable in any text editor. This independence from vendors is what makes long-term durability possible.

The second property is automatic version history. Because the substrate is git, every file is tracked, every change is attributed and timestamped, every state the memory has ever been in remains recoverable. We get the history for free as a side effect of normal work. We do not have to design a separate audit trail or backup mechanism to know what the project's memory looked like six months ago; git knows. The git history is itself a second-order memory — a memory of how the first-order memory has changed — and we treat it as such, which is why commit messages and PR descriptions are documents in their own right rather than throwaway labels.

The third property is co-location of code and documentation. They live in the same repo, move together, are reviewed in the same diff. This makes drift between them mechanically detectable: a reviewer looking at a PR sees both changes (or sees only one) and can demand the other before the PR merges. A documentation system stored elsewhere — in Notion, in a wiki, in any external tool — cannot offer this guarantee, because the writer of the code is on a different system from the writer of the docs (even when they are the same person), and the synchronized review that catches drift is impossible.

The fourth property is path-as-meaning. Where a file lives in the directory tree identifies what kind of memory it holds before you read a word of it. A file at `docs/architecture/decisions/0007-design-system-as-source-of-truth.md` is identifiable as an Architecture Decision Record from the path alone. A hypothetical file at `docs/runbook/incidents/<incident-name>.md` would be identifiable as an incident response procedure. A file at `docs/operations/dora-metrics.md` is identifiable as operational measurement memory. This makes navigation through the memory faster than search, which is one of the conditions Clark and Chalmers identified as necessary for an artifact to function as cognition.

The fifth property is tool-friendliness. Plain text in known locations can be searched (grep), analyzed (linters, scripts), transformed (any text-processing tool), and processed by AI agents reading the filesystem directly. This makes the memory active rather than just stored. Memory that cannot be processed by tools eventually drifts because no one notices when it becomes inconsistent. Memory that can be processed has a chance of being maintained because consistency can be checked automatically. We rely on this for memory hygiene: a docs-consistency check in CI (`scripts/check-doc-paths.mjs`) verifies that repo paths and relative links cited in the living documents still resolve. Richer checks — missing ADRs for new patterns, glossary terms used inconsistently — are not yet automated and remain review concerns until someone builds them.

The sixth property is human-first readability with secondary machine-friendliness. Markdown is prose with light structure. It is read by humans naturally and by AI agents naturally, because the agents are trained primarily on human-readable text. The choice of markdown over a more structured format (such as a knowledge graph or a database schema) reflects this dual-audience reality. We accept that this means some queries that would be elegant in a graph database are clumsy here. The trade is right for our scale.

The seventh property is composability. Files, directories, and repos compose freely. This means we can move pieces of memory between contexts, extract patterns from one project and seed them into another, and build templates that initialize new projects with the right memory shape. The lack of a fixed schema means the memory can grow, shrink, branch, and reorganize as the project requires.

The eighth property, which is more constraint than enabler, is that the substrate has a limited shape vocabulary: files and directories. We do not get rich relationships, queries, or transactional consistency. If we want to express a relationship between two pieces of memory, we do it by writing about it in the text or by linking through a path. If we want to query the memory, we do it through search or through scripts we write. The trade-off is real: we accept clumsier querying in exchange for durability, vendor independence, version control, tool-friendliness, co-location, and human readability. For a one-person project working with AI agents, this trade is overwhelmingly correct.

## The six categories of memory

Cognitive psychology distinguishes between several kinds of memory, and the distinction is useful because each kind has its own role and its own failure mode. FleetCo's memory architecture follows this map. A repo that holds only some of these categories will fail in characteristic, predictable ways.

**Episodic memory** holds specific decisions made at specific moments and the reasoning that surrounded them. In FleetCo this is held in `docs/architecture/decisions/` as Architecture Decision Records. An ADR is the project's memory of a single decision: what was the situation, what was decided, what alternatives were considered, what consequences were accepted, when should this decision be revisited. ADRs are numbered sequentially, never renumbered, and append-only; superseding a decision is done by writing a new ADR that explicitly supersedes the old one. Loss of episodic memory looks like the same decisions being re-litigated in every new session, with the original reasoning unavailable, leading to drift that no one notices. The starting set of ADRs in this repo (numbered 0001 through 0013) captures the foundational decisions made during project kickoff. Every subsequent meaningful decision adds to this set.

**Semantic memory** holds what words mean. In FleetCo this is held in `docs/glossary.md`. The glossary covers technical vocabulary (idempotency, vertical slice, ADR), domain vocabulary (Trip, Tipper, Lease-taker), and Nepal-specific regulatory and cultural vocabulary (Bluebook, Bishesh Anumati, BS calendar, IRD, NPR-as-paisa). Loss of semantic memory looks like participants using the same word with different meanings, agreeing in words while disagreeing in substance. The glossary is alphabetical, entries are not silently removed, and a term that changes meaning over time has the change documented in its entry rather than its history erased.

**Procedural memory** holds how things are done. In FleetCo this is held in `docs/runbook/` and `BOOTSTRAP.md`. Procedures cover deployment, rollback, restoring from backup, setting up a new development environment, responding to common alerts, handling incidents (general and security-specific), and maintaining business continuity in the founder's absence. In a large company, procedural memory often lives in the muscle memory of senior engineers who have done these things many times. FleetCo has no senior engineers; it has a fresh agent each session. Procedural memory therefore must be externalized into documents that any participant can read and execute. Loss of procedural memory shows up at the worst possible moment — during incidents, when the procedure is needed and the only person who knew it is unreachable. The runbook is testable: procedures known to be broken are flagged as broken rather than left in place.

**Prospective memory** holds intentions held for future execution. In FleetCo this is held in `docs/product/roadmap.md` (the long-range phasing), `docs/CURRENT_PHASE.md` (what phase is active right now), and `docs/tech-debt.md` (smaller items that have been deliberately deferred and must not be lost). Loss of prospective memory looks like a project full of energy but without follow-through: ideas surface, get briefly considered, and disappear because there is no place that holds onto them between sessions. The discipline here is that nothing accepted as a future commitment leaves the conversation that produced it without being written into one of these files.

**Perceptual memory** holds what things should look and feel like. In FleetCo this is held in `docs/design/`. The file `docs/design/DESIGN.md` defines the design system itself: tokens, typography, spacing, components, voice, anti-patterns. The folder `docs/design/slices/` holds locked HTML mockups for specific surfaces, committed when their visual design is approved. The folder `docs/design/slices/_archive/` holds historical mockups whose implementing code has merged. The Tailwind theme configuration derives from `docs/design/DESIGN.md`; drift between them is the failure mode. External design tools (Open Design, Figma, etc.) are iteration surfaces, not memory; the locked mockup in the repo is canonical. Loss of perceptual memory looks like visual identity drifting toward whatever defaults the most recent agent had, because there is no externalized reference for what the project should look like.

**Operational memory** holds measurements and operational records that accrue on regular cadences rather than in response to specific work. In FleetCo this is held in `docs/operations/`. The DORA metrics file is updated weekly with the four canonical delivery metrics plus the rework rate, per the targets in ADR-0010. SLO compliance reports are computed monthly per ADR-0011, though the format and home for those reports is to be settled when the first measurement window closes. Operational memory sits at the intersection of procedural memory (because operations are activities done over time) and prospective memory (because the records inform future planning), but it is distinct enough in character to deserve its own folder. Loss of operational memory looks like a project that is busy without knowing whether it is improving, because there is no cadenced measurement to detect trends or to surface failure modes early.

These six categories are not exhaustive — a sufficiently complex project might require additional categories — but they cover the territory FleetCo needs to cover. New categories require an ADR; we do not invent homes for memory casually.

## Why writing earns trust, and why trust is a precondition

Clark and Chalmers were specific about the conditions under which an artifact counts as part of cognition: it has to be reliably available, easily retrievable, and trusted. The first two conditions are about access; the third is about belief. The repo only functions as memory when its participants believe its contents enough to act on them without re-verification.

This produces a striking implication. Stale memory is worse than missing memory. Missing memory tells the reader to go figure it out. Stale memory tells the reader a plausible lie they may act on. A document that is sometimes wrong is not a partially useful document; it is a document that cannot be trusted to read at all, because every read becomes a verification problem. A repo with documentation that has a 30 percent chance of being stale is functionally equivalent to a repo with no documentation, because nobody reads documentation they cannot trust. This is why the discipline of keeping memory current is not aesthetic. It is what makes the entire system function. A code change that does not update the relevant memory is incomplete in the same sense that a code change without a test is incomplete: the work is not finished, even if the code happens to work right now.

The substrate helps here. Because filesystem memory is co-located with code and reviewed in the same diff, the discipline of synchronized maintenance is partially mechanized. A reviewer looking at a PR sees both the code change and the documentation change and can demand the latter before approving the former. A monolithic documentation system stored elsewhere makes this kind of synchronized review impossible, which is why teams that rely on such systems almost always end up with stale docs.

## The asymmetry between writers and readers

This is the structural reason memory discipline is hard to sustain even when participants intellectually agree with it. The participant who writes a piece of memory and the participant who later reads it are almost never in the same state of mind. The writer has just finished the work and has all the context fresh; the reader is starting cold, possibly months later, possibly in a different mental frame, and is often a different agent or a different version of the human PO. The writer feels the cost of writing — it takes time, it interrupts flow, it feels like overhead. The writer does not feel the benefit, because the benefit accrues to the reader, who is not there. The reader feels the benefit if writing was done well, or feels the cost of its absence if it was not, but cannot reach back in time to thank the writer or to demand more.

Because the writer never feels the cost of not writing and the reader cannot demand more after the fact, the natural equilibrium under immediate incentives is that every writer under-writes. The discipline of writing well must therefore be external — enforced by templates, PR checklists, and rules — rather than internal. The writer must write *for a future reader who is genuinely a different entity*. The future reader has no shared context, no memory of this session, and no way to ask a follow-up question. A note that means something to the writer because the writer was there does not mean anything to the reader who was not. The writer must write as though the reader is intelligent, attentive, and uninformed, because that is exactly what the reader is.

This is why committed documents do not contain phrases like "as we discussed" or "as agreed earlier." They contain the substance of what was discussed and agreed. The conversation was the iteration surface; the document is the memory.

## How the system protects itself

Several disciplines, taken together, keep the memory architecture working. None of these is optional. Skipping any of them weakens the entire system, because trust is binary: either the memory is reliable enough to act on without verification, or it is not, and there is no graceful degradation between those states.

Memory updates are part of the work, not after the work. Every PR that touches a domain concept, an architectural pattern, a procedure, a design surface, an operational measurement, or a roadmap item updates the corresponding memory artifact in the same diff. The PR template enforces this with explicit checkboxes covering all six memory categories. A PR that fails to update the relevant memory is not merged.

Memory artifacts have integrity invariants. Files have stable paths; renaming is breaking and requires deliberate redirect. Files are small and focused so they diff cleanly. Directories are shallow so they navigate well. Filenames are predictable so they can be guessed. Cross-references use relative paths so the memory is movable. Binary files are exceptions and are accompanied by text. ADRs are numbered sequentially and never renumbered. ADRs are append-only. The glossary is consistently ordered. Runbook procedures are testable. The design folder maintains its canonical-vs-iteration separation. Operational memory files document their update cadence at the top.

When memory and code disagree, the disagreement is treated as a defect. The project is in an inconsistent state. The correct response is not to silently update one to match the other but to figure out which is correct, fix the wrong one, and write down what happened. Because the substrate is filesystem-based and grep-able, consistency checks can be automated. The first such check runs in CI today: `scripts/check-doc-paths.mjs` fails a PR when a living document (the glossary, active tech-debt entries, runbooks, the design system, the roadmap, this file) cites a repo path or relative link that no longer resolves — exactly the drift class the 2026-07-02 audit found had accumulated after the `(app)` route-group move. Deeper semantic checks (glossary terms used in code with different meanings, runbook procedures invoking commands that no longer exist in the codebase) are not yet automated; they remain review concerns until built. Append-only historical records (ADRs, the CURRENT_PHASE log, paid-off tech-debt entries) are deliberately outside the checker's scope — they describe the repo as it was, and rewriting them to track renames would erase history.

External tools are iteration surfaces, not memory. Design tools, AI chat sessions, project management tools, and any other external system can be used freely for thinking, but the conclusions that emerge must be committed to the repo to count. The Open Design state on the laptop is not memory. The Claude.ai chat is not memory. The Notion page is not memory. Any of these can be where thinking happens; none of them is where thinking is preserved.

## The CEO's role in the memory architecture

The CEO of the company, who is also the sole product owner and developer of FleetCo, occupies a specific role in this architecture worth naming explicitly. The CEO does not need to write the memory artifacts personally; the AI agents working in the repo write them as part of their work. But the CEO is the only continuous human participant, and therefore plays two roles that no agent can play.

The first role is that of guardian of memory integrity. When an agent proposes a change, the CEO is the one who notices whether the proposal is consistent with prior decisions captured in ADRs, whether the change touches memory artifacts that should be updated, whether the agent has read the relevant memory before proposing. The agent has no memory of prior sessions; the CEO does, even if imperfectly. The CEO's review of agent proposals is partly a review of code and partly a review of whether the agent has correctly internalized the project's memory.

The second role is that of memory commissioner. The CEO is the one who notices when the repo's memory is failing — when a category is underdeveloped, when a particular kind of decision keeps getting re-litigated, when documentation has drifted from code. The CEO is the one who commissions the agent to fix these failures, to write the missing ADR, to expand the glossary, to refresh the runbook. The agent will not initiate this work; it will execute it once asked.

These two roles do not require the CEO to be a hands-on coder. They require the CEO to take the memory architecture seriously, to read the artifacts periodically, and to notice when things are slipping. This is exactly the role of a CEO with respect to any organizational system: not to do the work, but to ensure the work is being done well and to intervene when it is not.
