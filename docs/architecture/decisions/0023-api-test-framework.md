# ADR-0023: API test framework — Vitest with reset-between-tests against a real Postgres

- **Status:** Accepted
- **Date:** 2026-05-26
- **Decider:** Product owner (CEO)

## Context

`apps/api` has shipped four PRs of behavior — auth (`auth.guard`, `AuthController`), health probes (`HealthController`), the Vehicles read path, the Vehicles write path with the retirement-transition rule and the P2002 → 409 mapping, the Vehicles edit/delete controller surface, and the iter-4 `ListVehiclesQuerySchema` filter/sort/paginate contract — without any automated tests. The tech-debt entry "API has no test framework yet; ci.yml runs no `apps/api` tests" has been carried as active debt across all four iters. ADR-0006 (vertical-slice development) names "service-level unit tests" as one of the six steps of every slice; that step has been deferred slice-by-slice, and every Phase 1 slice from iter 5 onward assumes a working test framework. This ADR is being written together with the implementing PR (`feat/api-test-framework`) so future readers see the framework choice and its first wiring in one diff.

Two coupled decisions are open. The first is which framework: NestJS conventionally ships Jest, with `@nestjs/testing`'s `TestingModule` built around Jest's mock primitives and Jest's expect API; the broader NestJS ecosystem (StackOverflow corpus, package READMEs, example repos) defaults to Jest. The competing option is Vitest, which is the runner already used by `apps/web` (`apps/web/vitest.config.ts`, `apps/web/test/design-token-drift.test.ts`). `apps/web` chose Vitest because it composes with the Vite-powered Next.js toolchain; the question for the API is whether one-stack consistency outweighs the NestJS-Jest convention.

The second is the test-database strategy. The Prisma `Vehicle` model has a unique constraint on `registrationNumber` (`apps/api/prisma/schema.prisma:125`), which surfaces immediately: any test that creates a vehicle must clean up after itself, or the next test's create will hit P2002. Three strategies are available — Testcontainers (a real Postgres started per test suite via Docker), reset-between-tests against the existing dev/CI Postgres (truncate the affected tables in `beforeEach`), and schema-per-test (a separate Postgres schema per test file, isolated via the `?schema=` parameter on `DATABASE_URL`). Each has different startup cost, isolation strength, and CI-shape implications.

The constraints in play: TypeScript end-to-end with `strict: true` (ADR-0005); the API's `tsconfig.json` sets `module: CommonJS`, `experimentalDecorators: true`, and `emitDecoratorMetadata: true` (ADR-0018 wired NestJS as the runtime); auth guards are mandatory on every controller per ADR-0021, so the framework must be able to construct authed routes for testing without bypassing the guard surface; the pre-commit gate cycle (`pnpm format && pnpm format:check && pnpm lint && pnpm typecheck`) takes ~10s and CLAUDE.md's discipline rule names ~30s as the threshold for adding a test command to that gate cycle, so the test suite must run in under ~30s on a developer laptop. The data plane choice (ADR-0020) puts Prisma and Postgres at the heart of every integration test.

## Decision

The framework is **Vitest** with **`unplugin-swc`** for NestJS decorator-metadata emission, and the test-database strategy is **reset-between-tests against the existing dev/CI Postgres**, with a small `resetDb()` helper that truncates the affected tables in `beforeEach`. Six commitments land together in this PR (`feat/api-test-framework`):

1. **Runner: Vitest.** `apps/api/package.json` gains `"test": "vitest run"` and a `vitest.config.ts` at `apps/api/vitest.config.ts`. Test files live at `apps/api/test/**/*.test.ts` (mirroring `apps/web/test/`). The same `describe` / `test` / `expect` mental model is used in both apps; a single agent or developer switching between apps does not switch test mental models. The watch-mode command is left implicit (`vitest`) for local use and not added as a script.

2. **NestJS decorator support: `unplugin-swc`.** Vitest's default esbuild transformer does not emit TypeScript decorator metadata (which NestJS's DI relies on via `emitDecoratorMetadata`), so a `vitest.config.ts` that uses esbuild alone would fail to resolve `@Injectable()` providers at runtime. `unplugin-swc`'s `vite` plugin runs SWC with `decorators: true` and `transform.legacyDecorator: true` plus `transform.decoratorMetadata: true`, producing the metadata that `@nestjs/common`'s DI container reads. This is the canonical NestJS+Vitest wiring; the alternative (writing a custom Vite plugin that calls `tsc` per-file) is materially slower and more fragile.

3. **TestingModule via `@nestjs/testing`.** `@nestjs/testing` lands as a devDependency. Tests construct the NestJS app instance through `Test.createTestingModule({...}).compile()`, exactly as if they were a `main.ts` for the test process. The framework supports real `PrismaService`, real `AuthGuard`, and real validation pipes; individual tests still choose how much to wire up (e.g., the `auth.guard` unit test overrides the `AUTH` token with a stub; the `vehicles.service` integration test uses the real `PrismaService` against the test database).

4. **Database strategy: reset-between-tests, real Postgres.** Tests connect to the same Postgres instance as dev (`postgres://fleetco:fleetco@localhost:5432/fleetco_test` — note the `_test` suffix on the database name to keep test data out of the dev DB). A `apps/api/test/db.ts` helper exports `resetDb()` which runs `TRUNCATE "vehicle", "user", "session", "account", "verification" RESTART IDENTITY CASCADE;` and is called from each integration test file's `beforeEach`. The test DB is created and migrated once per test run by a `apps/api/test/setup.ts` global setup hook that runs `prisma migrate deploy` against `DATABASE_URL` if `NODE_ENV === "test"`. Locally this is the dev Postgres container with a second database; in CI it is a `services: postgres` block on the GitHub Actions runner.

5. **`tsconfig.test.json` for test-only TS settings.** A `apps/api/tsconfig.test.json` extends `tsconfig.json`, adds `test/**/*` to `include`, and sets `noEmit: true` so the test typecheck is fast. The production `tsconfig.json` keeps `rootDir: ./src` so the build (`nest build`) does not include test files. `apps/api/package.json`'s `typecheck` script keeps the existing single `tsc -p tsconfig.json --noEmit` invocation — adding test typecheck to the main typecheck script is deferred until a test compiles only because of a `tsconfig.test.json` setting (no such case today; Vitest's SWC transformer typechecks transitively when tests run).

6. **CI wiring.** `.github/workflows/ci.yml` gains a `services: postgres` block (the project's existing Postgres image, `postgis/postgis:16-3.4-alpine` from `docker-compose.yml`, exposed on `5432`) and a "Migrate test database" step that runs `pnpm --filter @fleetco/api exec prisma migrate deploy` against the service-container URL, followed by a "API tests" step that runs `pnpm --filter @fleetco/api test`. The new test step is ordered after lint, format, typecheck, and Prisma validate (which are cheap and fail-fast) and before the existing web-test step (no functional reason; alphabetical ordering keeps the file readable). The test step is a required check for `main` by virtue of being part of the same `ci` job, the same way the existing checks are required.

## Alternatives considered

**Jest as the runner.** The NestJS-conventional choice. `@nestjs/testing` is built around Jest's mock primitives (`jest.fn()`, `jest.spyOn()`) and Jest's matcher API. The NestJS docs example for every testing pattern uses Jest. The case for Jest is the breadth of community examples for NestJS-specific patterns (testing guards, interceptors, middleware, queues). Rejected because (a) the project already runs Vitest in `apps/web` and one-stack consistency is a real maintenance benefit — agents and developers move between apps frequently and switching test mental models is friction with no upside; (b) Vitest's mocking surface (`vi.fn()`, `vi.spyOn()`, `vi.mock()`) is a near-drop-in replacement for Jest's, so the community examples are mechanically adaptable; (c) Vitest's startup is faster, which directly serves the ~30s pre-commit budget; (d) Vitest's typescript-native execution (via SWC here) avoids the `ts-jest` configuration that NestJS+Jest projects often carry. The cost we accept: occasionally a Jest-only example from the NestJS ecosystem will need a small translation. That cost is paid per-example and trends to zero.

**Testcontainers for the test database.** A real Postgres container started per test run via the `testcontainers` npm package, with the connection string injected into `DATABASE_URL` at runtime. This is the highest-fidelity option: every test run gets a guaranteed-fresh database, no inter-test contamination is possible, and CI does not need a `services: postgres` block (Testcontainers manages the container). Rejected because (a) container startup is ~5-15 seconds even with a warm image cache, which on a developer laptop pushes the test suite from ~5s to ~20s on every run, brushing the ~30s pre-commit budget; (b) Testcontainers requires Docker to be running on the developer's machine — which is true for the FleetCo dev environment today (docker-compose.yml is the dev posture) but adds a hard dependency we have so far been willing to consider optional; (c) CI runners cost rises (the GitHub Actions free-tier minutes per PR roughly double when a Docker daemon must build and start a container per workflow). The fidelity gain is real but the day-to-day cost outweighs it for a single-developer project at Phase 1 scale.

**Schema-per-test.** Each test file connects to a freshly-created Postgres schema (`?schema=test_${randomId}`) and drops it on teardown. Test isolation is total — no `beforeEach` truncation is needed because no two tests share a schema. Rejected because (a) Prisma's connection-string `?schema=` parameter does not interact cleanly with `@nestjs/testing` — the `PrismaService` is constructed once per `TestingModule.compile()`, and switching schemas across files within a single test run requires re-instantiating the service, which fights the DI container's caching; (b) the bookkeeping cost is real: creating a schema, running `prisma migrate deploy` into it, and dropping it on teardown is several seconds per file, and we have 4+ test files even in this PR; (c) the isolation benefit duplicates what `resetDb()` already provides at a fraction of the cost. Schema-per-test is the right pattern at a scale where dozens of integration tests run in parallel and `TRUNCATE`'s lock contention becomes visible; we are not at that scale and ordering tests with `--no-file-parallelism` (Vitest's default for integration tests in this PR) eliminates the contention question entirely.

**In-memory SQLite via a Prisma SQLite provider.** Drop-in replace Postgres with SQLite in tests for speed. Rejected because Prisma's Postgres provider exposes features SQLite does not (the `postgresqlExtensions` preview feature is on the generator; PostGIS will be a load-bearing extension once geospatial trips land; `RESTART IDENTITY CASCADE` and `$queryRaw` semantics differ). A test that passes against SQLite but fails against Postgres is exactly the silently-passing test we are paying for this framework to prevent. The "tests run faster" argument is also unconvincing once integration tests can no longer rely on Prisma error codes — P2002 surfaces differently in SQLite and the runbook's api-error-mapping table is Postgres-specific.

**Mock everything — no real Prisma in tests.** Build a `jest-mock-extended`-style mock of `PrismaService` and never touch a real database. Rejected because the bugs this framework is paying off are at the integration seams: the P2002 → 409 mapping in `vehicles.service.ts`, the `$transaction([findMany, count])` pattern in `list()`, the retirement-transition rule's interaction with the existing row read in `update()`. A mocked Prisma would make every test pass while preserving the seam bugs the test is supposed to catch. The kickoff explicitly directs "real `PrismaService`, real auth guards, and real validation pipes — not a mock of any of those — because the bugs that matter are at integration seams."

**Run the test command from the pre-commit gate cycle today.** Add `pnpm --filter @fleetco/api test` to the `format && format:check && lint && typecheck` gate cycle in this PR. Rejected because the test suite at this PR's landing is ~6 seconds locally (which fits the ~30s budget), but adding it to the gate cycle requires every contributor's pre-commit to start the Postgres container if it is not already running, which is a discoverability friction (the error message is a Postgres connection refused, not "start docker"). The right path is to land the test framework first, let it bed in for a slice or two, then move the test command into the gate cycle when the dev-setup runbook is updated to explicitly include `pnpm dev:up` before commit. This is recorded as a follow-up tech-debt entry.

## Consequences

**What this makes easier.** Every Phase 1 slice from iter 6 (Drivers read path) onward has a test framework waiting: `apps/api/vitest.config.ts` exists, `@nestjs/testing` is installed, the `resetDb()` helper is reusable across modules (extend the truncate list when new tables land), and the CI gate is already green. ADR-0006's "service-level unit tests" step of every vertical slice becomes a real step rather than a deferred-to-the-test-framework-ticket placeholder. Developers and agents who run `pnpm --filter @fleetco/api test` locally get fast feedback (the suite runs in well under 30 seconds), and the same command on CI is the required check, so local-vs-CI signal drift is eliminated. The `@nestjs/testing` `TestingModule` API is well-documented and ergonomic; future tests for new modules follow the patterns established in this PR.

**What this makes harder.** The dev environment now requires Postgres to be running for the test command to succeed, which it already did for `pnpm dev` but is now a hard requirement for `pnpm --filter @fleetco/api test` as well. The `dev-setup.md` runbook is updated to call this out. The `resetDb()` helper must be kept in sync with the Prisma schema — when a new table is added in a future slice, its name must be added to the truncate list; otherwise tests that write to it will see contamination from prior runs. This is a maintenance cost paid in the migration commit, not separately, and a comment in `resetDb()` flags the discipline.

The `unplugin-swc` dependency couples the test runtime to SWC's TypeScript transform, which is a third TypeScript runtime in the project (alongside `tsc` and esbuild). The three implementations agree on `strict: true` TypeScript and on the decorator metadata shape, so disagreement risk is low; an upgrade that broke decorator-metadata compatibility would be caught by the test suite itself (every NestJS DI test would fail). This is a real but small lock-in cost — the alternative (custom `tsc`-per-file Vite plugin) would be slower and more fragile.

The CI job's wall-clock time grows by ~10-15 seconds for the Postgres service container startup, migrate-deploy, and test execution. This is acceptable in a workflow that already takes ~3 minutes; the marginal cost per PR is small. If the test suite grows into the multi-minute range, we revisit by sharding tests across multiple jobs or by introducing parallelism via Vitest's `--file-parallelism` flag, which works trivially with `resetDb()`'s per-test isolation as long as test files do not share state.

**Costs we accept.** A non-trivial fraction of the NestJS testing ecosystem (e.g., specific @nestjs-internal mocking patterns published as Jest helpers) does not translate to Vitest without adaptation. We accept that cost because the consistency gain across the monorepo dominates and the adaptations are mechanical. The decision is reversible — swapping back to Jest costs every test file but the production code does not move — though the cost of reversal grows with the number of tests, which is why the choice is being made now rather than later.

The `resetDb()` strategy depends on FK-cascade behavior (`TRUNCATE ... CASCADE`) and the absence of long-running transactions that would block the truncate. Both hold today and will likely hold through Phase 1; if they break (e.g., if we ever introduce a long-lived test fixture that the truncate must not touch), the right move is to switch to schema-per-test for the affected suite, not to rewrite all tests.

## Revisit when

- (a) **The test suite's wall-clock time approaches 30 seconds locally.** This is the threshold below which the test command goes into the pre-commit gate cycle. If the suite slows below that bar, the right responses are sharding via Vitest's parallel runner, moving slow integration tests to a separate `test:integration` script run only in CI, or both. The pre-commit-inclusion decision is revisited when wall-clock time, not file count, says it should be.
- (b) **A test would have caught a P2002-or-similar Prisma-mapping bug and did not, traced to the mock or to a missing seam.** The fix is more integration tests, not a framework swap; but the signal tells us the framework's coverage shape is wrong and the kickoff for the next slice should explicitly call out where coverage is thin.
- (c) **NestJS upgrades to a version whose decorator-metadata emission diverges from SWC's.** Revisit by checking whether `unplugin-swc` has caught up or whether the right transformer is now `vite-plugin-swc` (a likely rename) or `@vitest/coverage-v8` (if Vitest itself ships native decorator-metadata). The test that flags this is every DI test failing on upgrade; the response is upgrade-coordinated, not standalone.
- (d) **Phase 2 introduces concurrent test execution at a scale where `TRUNCATE`'s lock contention is observable.** The signal is test-suite slowness that does not scale linearly with file count or test count, attributable in profiles to lock waits. The right response then is schema-per-test for the suites with the worst contention, or moving integration tests to a worker-pool model that explicitly partitions data. Both are larger changes than this ADR's scope; both are deferred.
- (e) **The orchestration loop introduces a "test on every PR" auto-merge gate that would benefit from sub-10-second feedback.** Today the loop already runs CI before auto-merging (ADR-0022), so the framework's CI cost is the dominant signal. If the loop ever runs the API test suite locally before opening the PR, that is the signal to optimize for sub-10-second total runtime.
