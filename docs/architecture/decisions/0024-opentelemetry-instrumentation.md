# ADR-0024: OpenTelemetry instrumentation — extend Sentry's TracerProvider, env-gated OTLP export, trace IDs in logs

- **Status:** Accepted
- **Date:** 2026-05-29
- **Decider:** Product owner (CEO)

## Context

The Phase-1 roadmap names a specific operational milestone: "OpenTelemetry is added to the NestJS API so that logs carry trace identifiers and the foundation for distributed tracing is in place before Phase 2 needs it" (`docs/product/roadmap.md` §"Phase 1 — The Spine"). The Phase-1 → Phase-2 gate checklist in `docs/retrospectives/phase-1.md` carries "OpenTelemetry in the API" as an unchecked item. This ADR is the proposal that closes the design half of that milestone; the implementing PR wires it into `apps/api`.

ADR-0018 (the API stack: NestJS, nestjs-pino, Sentry) deferred tracing explicitly. Its Consequences section reads: "No tracing yet. OpenTelemetry, distributed tracing, and `traceId` propagation are out of scope for Phase 0. nestjs-pino's `genReqId` produces request IDs but they are not OpenTelemetry trace IDs; a future ADR will introduce the trace-id discipline when measurement justifies it." ADR-0018's "Revisit when" repeats the signal: tracing arrives "when request volumes are high enough that point-in-time logs are insufficient for root-cause analysis, or when a Phase 2+ feature (driver app talking to API) introduces cross-process flows that need trace-level visibility." Phase 2 is exactly that cross-process flow, and the roadmap wants the substrate in place *before* Phase 2 needs it. This ADR is the future ADR that ADR-0018 promised. ADRs are append-only: ADR-0018 is not edited; this ADR follows it.

ADR-0011 (reliability SLOs) defines two SLIs — API availability and trip-creation success — and notes they are "measurable from data we are already collecting (Sentry for the error portion, the API logs for the latency portion)." The SLI-instrumentation tickets emit structured per-request and per-operation signals at the log layer; OpenTelemetry is the substrate that lets a future reader correlate one of those signals to the exact trace (and Prisma query spans) that produced it. The trace context is not a replacement for the SLI signals; it is the connective tissue that makes them debuggable.

The constraint that forces an ADR before code is `CLAUDE.md`'s rule: a new top-level dependency must be proposed first, with the reason. OpenTelemetry instrumentation adds several packages to `apps/api`. This ADR is that proposal, and it names the exact dependency set so the implementing ticket adds that set and no more.

The decisive technical fact that shapes everything below: **the API already runs an OpenTelemetry TracerProvider today, because Sentry's Node SDK v9 is itself built on OpenTelemetry.** `apps/api` depends on `@sentry/node@^9` (resolved to 9.47.1), and ADR-0018 wired `Sentry.init({ dsn, environment })` at the top of `bootstrap()` in `main.ts`. In v9 the Node SDK, unless `skipOpenTelemetrySetup: true` is passed, initializes the global OpenTelemetry TracerProvider, registers Sentry's span processor on it, installs the W3C trace-context propagator, and — when tracing is enabled — auto-instruments HTTP (`httpIntegration`), Prisma (`prismaIntegration`), and Redis/ioredis (`redisIntegration`) from its default integration set. The API therefore is not a blank slate onto which we add OpenTelemetry from scratch; it has a live, Sentry-owned OpenTelemetry setup. The real question is how to *expose and extend* that setup — to add an exporter that can ship spans to a future tracing backend, and to surface trace IDs in logs — without standing up a second, conflicting TracerProvider and without double-instrumenting the libraries Sentry already covers.

The other constraints in play: TypeScript end-to-end with `strict: true` (ADR-0005); the API's build is CommonJS (`module: CommonJS`, ADR-0018); the `main.ts` bootstrap order is load-bearing per ADR-0021 (`bodyParser: false` → better-auth mount at `/auth/{*splat}` → re-attach body parsers → CORS → listen), and the Sentry/OTel init must stay above all of that; the data plane is Prisma 6 + Postgres + Redis (ADR-0020); optional-URL env vars follow the `SENTRY_DSN` pattern in `apps/api/src/config/env.ts` (an empty string in `.env` coerces to `undefined` so a blank line does not trip URL validation); and trace identifiers must respect the ADR-0013 data classification.

## Decision

We add OpenTelemetry trace context to the API by **extending the OpenTelemetry setup that Sentry v9 already owns** — feeding it an env-gated OTLP/HTTP span exporter and adding a pino log-correlation mixin — rather than standing up a second, independent OpenTelemetry `NodeSDK`. Seven commitments define the shape the implementing ticket builds:

1. **Extend Sentry's TracerProvider; do not duplicate it.** Sentry v9's `Sentry.init()` already creates the global TracerProvider, registers its span processor, and auto-instruments HTTP, Prisma, and Redis. We add extra span processors through `Sentry.init`'s `openTelemetrySpanProcessors` option and (only if needed) extra instrumentations through its `openTelemetryInstrumentations` option — both are first-class, documented init options in `@sentry/node@9` (verified against the installed 9.47.1 types and SDK source: `init()` forwards `openTelemetrySpanProcessors` into Sentry's `initOpenTelemetry(client, { spanProcessors })`). We do **not** call `new NodeSDK(...).start()` and we do **not** register a standalone `NodeTracerProvider`. OpenTelemetry's global API permits exactly one global TracerProvider; a second registration is silently dropped (or, depending on timing, detaches Sentry's processor and breaks Sentry's trace/error correlation).

2. **Dependency set (added by the implementing ticket; proposed here).** The implementing ticket adds exactly:
   - `@sentry/opentelemetry` — Sentry's OpenTelemetry compatibility layer, pinned as a direct dependency (it is already a transitive dependency of `@sentry/node`) so its version is explicit and its span-processor / propagator helper types are importable from the FleetCo seam.
   - `@opentelemetry/api` — the vendor-neutral API used by the pino mixin to read the active span (and the surface for any future manual spans). Already transitively present via `@sentry/node`; pinned directly because our own code imports it.
   - `@opentelemetry/exporter-trace-otlp-http` — the OTLP-over-HTTP span exporter.
   - `@opentelemetry/sdk-trace-node` — supplies `BatchSpanProcessor`, which wraps the OTLP exporter; the processor (not the raw exporter) is what is handed to Sentry's `openTelemetrySpanProcessors`.
   - `@opentelemetry/instrumentation-ioredis` — added **only if** the implementing ticket verifies that Sentry v9's default `redisIntegration` does not already cover ioredis spans. Sentry's `redisIntegration` wraps this same instrumentation, so the expected outcome is that this package is *not* needed; it is named here so the implementing ticket has an explicit yes/no to resolve rather than an open question.

   Explicitly **not** added: `@opentelemetry/sdk-node` (its presence invites the second-`NodeSDK` anti-pattern this ADR forbids), `@prisma/instrumentation` (Sentry's `prismaIntegration` already instruments Prisma 6 — see commitment 4), and any standalone `NodeTracerProvider`.

3. **OTLP export is env-gated and no-ops when unset.** A new `OTEL_EXPORTER_OTLP_ENDPOINT` env var (optional URL, empty-string→`undefined`) is added to `apps/api/src/config/env.ts`, mirroring the existing `SENTRY_DSN` field exactly (`z.preprocess(emptyStringAsUndefined, z.string().url().optional())`), and documented in `apps/api/.env.example` alongside the `SENTRY_DSN` line. When the endpoint is unset, no OTLP exporter and no `BatchSpanProcessor` are constructed, and the API ships no OTLP telemetry — the same no-op-in-local-dev posture as the `SENTRY_DSN` guard in `main.ts`. When the endpoint is set, an OTLP/HTTP `BatchSpanProcessor` is added to Sentry's `openTelemetrySpanProcessors`, so spans fan out to both Sentry and the configured OTLP endpoint.

4. **Prisma tracing needs no schema change.** Prisma 6's tracing is **GA** (the `tracing` feature was promoted out of preview in Prisma 6). Therefore: **no** `previewFeatures = ["tracing"]` in `schema.prisma`, **no** schema edit, **no** migration. Sentry's `prismaIntegration` consumes Prisma's GA tracing automatically. This is stated explicitly because a reader carrying the Prisma-5-era recipe would reach for the preview flag plus `@prisma/instrumentation` and generate a needless migration — both wrong under Prisma 6.

5. **Log correlation via a pino `mixin`.** A `mixin` is added to the `LoggerModule.forRoot` `pinoHttp` config in `app.module.ts`. It reads the active span from `@opentelemetry/api` (`trace.getActiveSpan()` → `spanContext()`); when a valid span context exists it injects `trace_id` and `span_id` into every log line, and it returns `{}` when there is no valid span (e.g. logs emitted outside any request, such as the bootstrap log line). This is what literally satisfies the roadmap's "logs carry trace identifiers." The nestjs-pino `genReqId` request ID (ADR-0018) is unchanged and continues to appear; `trace_id`/`span_id` are added alongside it, not in place of it.

6. **Init ordering is preserved, and the init guard widens.** The Sentry/OTel init stays at the very top of `bootstrap()`, **before** `NestFactory.create` and **above** the better-auth mount, per ADR-0021's load-bearing ordering — adding span-processor wiring must not reorder the body-parser dance. The init guard widens from `if (env.SENTRY_DSN)` to "init when **either** `SENTRY_DSN` **or** `OTEL_EXPORTER_OTLP_ENDPOINT` is set" (passing `dsn: undefined` when the DSN is absent), so the OpenTelemetry substrate can run in an environment that exports traces to an OTLP backend without also sending errors to Sentry.

7. **Deferred to Phase 2: the tracing backend vendor and OTel metrics.** Phase 1 commits the *substrate only* — trace context, trace-IDs-in-logs, and an env-gated exporter that no-ops until an endpoint exists. Choosing and operating an externally-hosted tracing backend (SigNoz, Honeycomb, Datadog, Grafana Tempo, or a self-hosted OTel Collector) and adding OpenTelemetry *metrics* (counters/histograms, as distinct from traces) are deferred to Phase 2. Per the roadmap, no new phase opens until Phase 1 is in daily use, and a backend needs infrastructure and credentials that are out of scope here. The env-gating means the substrate is ready the day a backend is chosen: set one env var, no code change.

## Alternatives considered

**Stand up a second, independent OpenTelemetry `NodeSDK`** (the `@opentelemetry/sdk-node` quickstart that most OTel tutorials lead with: `new NodeSDK({ traceExporter, instrumentations }).start()`). Rejected because Sentry v9 already owns the global TracerProvider. A second `NodeSDK.start()` either no-ops (the global provider is already registered and OpenTelemetry refuses to replace it) or, if it wins a startup race, replaces Sentry's provider and silently detaches Sentry's span processor — breaking the Sentry trace/error correlation that ADR-0018 is paying for. It would also register its own HTTP and Prisma instrumentations on top of Sentry's, double-instrumenting every request and every query (duplicate spans, and the instrumentation libraries warn or throw on double-registration). The supported, lower-surface path in Sentry v9 is to feed extra processors and instrumentations into the setup Sentry already runs, via the documented `openTelemetrySpanProcessors` / `openTelemetryInstrumentations` init options.

**Manually register `@opentelemetry/instrumentation-http` and `@prisma/instrumentation`.** Rejected because Sentry's default integration set already instruments HTTP (`httpIntegration`) and Prisma (`prismaIntegration`) when tracing is enabled. Re-registering them double-instruments the same calls and the instrumentation libraries guard against double-registration. The rule the implementing ticket follows is: add nothing for HTTP or Prisma; they are covered.

**Use Prisma's `previewFeatures = ["tracing"]` plus `@prisma/instrumentation` (the Prisma-5-era recipe).** Rejected because Prisma 6 GA'd tracing; the preview flag no longer gates anything and adding it churns the schema and forces a migration for no behavioral gain. Sentry's `prismaIntegration` reads Prisma 6's GA tracing without a schema change. This is the single most likely wrong turn for a reader who remembers the Prisma-5 instructions, which is why commitment 4 states it negatively and explicitly.

**Keep nestjs-pino's `genReqId` request IDs as the only correlation identifier; skip OpenTelemetry trace IDs entirely.** Rejected because request IDs are per-process: they identify a request inside one API process but do not propagate across the process boundary Phase 2 introduces (driver app → API, and any future API → worker flow). The roadmap explicitly wants "the foundation for distributed tracing in place before Phase 2 needs it," and W3C trace context is that foundation. Request IDs and trace IDs are not mutually exclusive — the mixin adds trace IDs while `genReqId` keeps producing request IDs — so this is a strict addition, not a swap.

**Adopt a tracing backend vendor now (SigNoz / Honeycomb / Datadog / self-hosted Collector).** Rejected for Phase 1. Choosing and operating a backend is Phase-2 work: the roadmap gates new-phase work on Phase 1 being in daily use, a backend needs infrastructure and credentials this program explicitly excludes, and the SaaS-vs-self-hosted trade-off deserves its own decision with its own ADR. The env-gated exporter is the hedge — the substrate is wired and dormant, and flipping `OTEL_EXPORTER_OTLP_ENDPOINT` turns it on with no code change. Deferred, not forgotten.

**Switch from `@sentry/node` to `@sentry/nestjs` as part of this work.** Rejected as out of scope. ADR-0018 chose `@sentry/node` and named the specific signal that would justify `@sentry/nestjs` (manual `@sentry/node` context wiring becoming verbose). OpenTelemetry instrumentation does not depend on the Sentry-NestJS surface, and bundling an SDK swap into a tracing ADR would conflate two independent decisions. If `@sentry/nestjs` is later adopted, it is a separate ADR; this ADR's "extend Sentry's OTel setup" decision is expressed in terms of init options that both SDKs share.

## Consequences

### What this makes easier

The API gains a real trace substrate. Every inbound HTTP request becomes a root span, Prisma queries become child spans under it, and — via the mixin — every log line emitted during the request carries `trace_id` and `span_id`. Root-cause analysis stops being "grep the logs by request ID within a single process" and becomes "follow the trace." The day a tracing backend is chosen in Phase 2, turning it on is a one-env-var change (`OTEL_EXPORTER_OTLP_ENDPOINT`) with no code edit and no redeploy-with-new-dependencies. The SLI signals from the SLI-instrumentation tickets (ADR-0011) become trace-correlatable: a latency-budget breach or a trip-creation failure can be tied to the exact span tree that produced it. The performance budget (committed later in this program) inherits a measurement substrate — request latency is visible both at the log layer (the SLI signal's `response_time_ms`) and at the trace layer (the HTTP span's duration) for whenever a backend exists to receive it.

### What this makes harder

There is now one more thing in the `main.ts` bootstrap that must respect the ADR-0021 ordering. ADR-0021's body-parser dance is already the most fragile part of `main.ts`; adding span-processor construction above it is more code that a careless refactor could reorder. The mitigation is the same as ADR-0021's: this ADR documents the ordering and the reason, and the implementing ticket keeps the OTel init inside the existing top-of-`bootstrap()` block rather than introducing a separate preload step. (For the record: because the build is CommonJS, if a preload file were ever chosen instead of in-bootstrap init, it would be `node --require ./dist/instrumentation.js`, not `--import`; the in-bootstrap path is preferred precisely because Sentry already inits there.)

Sentry's OpenTelemetry coupling means a future `@sentry/node` major could change the `openTelemetrySpanProcessors` / `openTelemetryInstrumentations` option surface. We pin `@sentry/node` to `^9` and treat any change to its OTel-extension API as a coordinated upgrade rather than an incidental one (see "Revisit when").

### Costs we accept

- **A small set of new top-level dependencies in `apps/api`.** They are all first-party OpenTelemetry packages plus Sentry's own OpenTelemetry bridge; `@opentelemetry/api` and `@sentry/opentelemetry` are already transitive dependencies of `@sentry/node`, so the net new supply-chain surface is modest. Each is named in commitment 2 with its reason, and the conditional ioredis package is the only open question for the implementing ticket.
- **The OTLP export path is not exercised against a real backend in Phase 1**, because no backend exists yet (commitment 7). The implementing ticket tests the FleetCo-authored seams — the env→span-processor builder and the pino mixin — and we accept that "spans actually arrive at a collector" is verified at the first production deploy with a backend (Phase 2). This is named so a future reader does not mistake the export path for battle-tested.
- **Trace data can carry Tier 2/Tier 3 data in span attributes** (an HTTP span's `http.url` can embed a query-string value such as an email in a filter), exactly as ADR-0018 noted Sentry breadcrumbs and tagged context can. In Phase 1 there is no OTLP egress (no endpoint set by default), so there is nothing leaving the process to protect yet. The span-attribute equivalent of pino's `redact` list — a `beforeSendSpan` hook or an OpenTelemetry `SpanProcessor` that scrubs sensitive attributes before export — is a Phase-2 concern wired when the backend lands. It is named here per ADR-0013 so it is not forgotten the moment an endpoint is configured.

## Revisit when

- **A tracing backend is chosen (Phase 2).** That decision gets its own ADR (or supersedes the deferred half of this one) naming the vendor or self-hosted topology, the sampling policy, and — load-bearing — the span-attribute scrubbing that the Tier 2/Tier 3 egress concern above requires before any span leaves the process.
- **`@sentry/node` majors past v9 and changes or removes `openTelemetrySpanProcessors` / `openTelemetryInstrumentations`.** The signal is a type error on upgrade. The response is to follow Sentry's then-current OTel-extension recipe as part of the upgrade, coordinated rather than standalone — the "extend, don't duplicate" principle survives even if the exact option names move.
- **OpenTelemetry metrics become worth adding.** For example, when the SLI reporting (ADR-0011) wants a latency histogram rather than a per-request good/bad boolean, or when queue-depth and job-duration metrics matter. This ADR scopes to traces; metrics are a separate decision.
- **Prisma majors past v6 and changes its tracing surface.** Prisma 6 GA'd tracing; a future major could change the integration shape. The signal is Prisma query spans disappearing after a Prisma upgrade; the response is to re-check whether Sentry's `prismaIntegration` still covers it or whether the GA surface moved.
- **The double-provider hazard recurs.** If a future dependency or a well-meaning refactor calls `NodeSDK.start()` (or otherwise registers a second global TracerProvider) and Sentry's traces go quiet, this ADR's "extend, don't duplicate" rule (commitment 1) is the diagnosis and the fix.
