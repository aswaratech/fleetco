# ADR-0028: Office-staff RBAC model — two live roles (ADMIN, OFFICE_STAFF) with DRIVER reserved, a library-aware role on the session, a hardcoded permission map, and a RolesGuard composing AuthGuard

- **Status:** Proposed
- **Date:** 2026-06-03
- **Decider:** Product owner (CEO) — pending acceptance

## Context

Phase 2 ("Driver app and telematics") is open per ADR-0025. One of its named deliverables is "the introduction of office staff as a user role with basic role-based access control" (`docs/product/roadmap.md` §"Phase 2 — Driver app and telematics"). Until now FleetCo has had exactly one human — the CEO — and exactly one authorization rule: "are you signed in?" (the `AuthGuard`). Phase 2 adds a second class of human (office staff who do operational data entry) and, simultaneously, the most sensitive data category the system has ever held (raw GPS traces, ADR-0027). The moment those two facts are true at once, "are you signed in?" is no longer a sufficient authorization model: some authenticated humans must be able to do things other authenticated humans must not. This ADR decides the model that draws that line. It is the third of three Phase-2 foundation ADRs proposed together — ADR-0026 (observability upgrade, merged) and ADR-0027 (GPS data-classification amendment, proposed) are its siblings — and it is the one the **other two explicitly defer their access decisions to**.

This decision has been anticipated by the repo's memory for a long time, which is why it can be made cleanly rather than invented from nothing:

- **ADR-0015 (auth library)** chose better-auth partly because "the library ships RBAC primitives that the Phase 2 RBAC introduction (office staff role) can build on without rewriting the auth module," and its "Revisit when" (c) names "the Phase 2 RBAC requirements (office staff as a distinct role with row-level scoping)" as a trigger to re-examine the library. This ADR is that Phase-2 moment; a finding of this ADR is whether better-auth's primitives are in fact sufficient (they are — see Decision commitment 2).
- **ADR-0021 (auth integration shape)** built the `AuthGuard` + `GET /me` pattern every module copies, and its "Revisit when" (a) predicted exactly this: "RBAC arrives in Phase 2 with office staff as a distinct user role. Better-auth's roles/permissions primitives become load-bearing; the `AuthGuard` may grow into `RolesGuard` or compose with one. The integration shape itself stays — only the guard surface grows." This ADR realizes that prediction: the integration shape is untouched; a `RolesGuard` composes the existing `AuthGuard`.
- **The glossary** already names the attach point. Its "Session-based authentication" entry: "the server-side session record is the natural place to attach role and permission state as Phase 2's RBAC arrives." Its "RBAC" entry: "deferred until Phase 2 in FleetCo, when office staff are introduced as a user role distinct from the CEO." This ADR closes that deferral.
- **ADR-0013 (data classification)** wrote a promissory note this ADR pays: its Tier-3 rule is that "access follows the role-based access control rules **to be defined in Phase 2**." Those rules are defined here.

The current substrate is greenfield for roles: there is **no `role` column** on the `User` model and **no role/permission/RolesGuard code anywhere in `apps/api/src`** today (verified). The `User` and `Session` models are better-auth-managed (`apps/api/prisma/schema.prisma`); `AuthGuard` resolves a session via `auth.api.getSession({ headers })` and attaches it to the request (`apps/api/src/modules/auth/auth.guard.ts`); `SessionPayload` is `NonNullable<Awaited<ReturnType<AuthInstance["api"]["getSession"]>>>` (`auth.types.ts`); `GET /me` returns `{ id, email }` (`auth.controller.ts`); the admin is seeded by an explicit idempotent `db:seed` script (`apps/api/scripts/seed-admin.ts`, ADR-0021 commitment 5).

Two binding requirements arrive from the siblings, and satisfying them is a hard constraint on the role set this ADR defines:

- **ADR-0027 commitment 7** requires that "raw-trace access is the most-privileged data class" and that "ADR-0028 must include a permission for raw-trace access **distinct from** operational/derived views; this ADR fixes the *requirement* that the two are different privilege levels, not the mechanism." Concretely: a role must be able to express "may query/export a named driver's full-resolution GPS history" as strictly more privileged than "may see the live-location map, geofence status, and trip route summary." The default posture ADR-0027 set: raw-trace query/export is CEO/ADMIN-only; office staff get derived views, not raw-trail export.
- **ADR-0026 commitment 7** defers who-may-see observability data here: "for Phase 2 observability access stays with the operator (CEO); if office staff ever need observability access, that is governed by ADR-0028's role model, not granted ad hoc here."

The constraints every FleetCo decision lives inside also apply. ADR-0014 fixes a single 2–4 GB VPS under Docker Compose operated by **one person** (the CEO, technically literate, not a hands-on coder) for a single-company internal tool in Nepal where cost matters — so the right RBAC is the *simplest model that is correct*, not an enterprise policy engine. ADR-0001's modular-monolith rule keeps the auth module's public interface narrow, so the guard surface other modules see should grow by composition, not by a rewrite. ADR-0005 is strict TypeScript, no `any`. And per CLAUDE.md's "How to work on a task" step 4, a change that touches a **cross-cutting concern (auth)** requires PO confirmation *before code is written* — which is precisely why this ADR exists ahead of any RBAC code: it is the PO-confirmation vehicle for the cross-cutting auth change. ADR-0025 also records that Phase 2 is being built ahead of the production deploy, so no second human actually exists in production yet; this model is set before the first office-staff account is created, which is the cheap moment to set it.

This ADR **decides policy and shape only**. It writes no schema, no migration, no `RolesGuard`, no decorator, and no seed change. Those land with the RBAC code slice that consumes the decisions fixed here.

## Decision

**Adopt a two-live-role model — `ADMIN` (held by the CEO/owner; the most-privileged human) and `OFFICE_STAFF` (operational data entry plus derived views) — with `DRIVER` reserved-but-not-yet-defined; store the role as a *library-aware* field better-auth manages (recommended: a `user.additionalFields` role with `input: false` and a least-privilege default; the better-auth `admin` plugin is the richer alternative; a library-*unaware* raw column is rejected — final pick is the PO's), so the role rides on `session.user` and is read at enforcement time with no extra query; map roles to coarse capability *permissions* through a single hardcoded in-tree table (not a DB-driven policy store); enforce with a FleetCo-authored `RolesGuard` + `@RequirePermission(...)`/`@RequireRole(...)` decorators that compose the existing `AuthGuard` (403 for an authenticated-but-unauthorized caller, distinct from `AuthGuard`'s 401); make raw-GPS-trace access and observability access ADMIN-only while derived GPS views are available to OFFICE_STAFF (discharging ADR-0027 c7 and ADR-0026 c7); keep role assignment an admin-only operation that is never client input; have `GET /me` return the role so the web can gate UI for UX; and defer row-level/field-level scoping to the DRIVER role.** Nine commitments define the shape. As with its siblings, this ADR fixes decisions and **writes no code**; the implementing slice builds them.

1. **Role set: two live roles, one reserved.** The Phase-2 role set is **`ADMIN`** and **`OFFICE_STAFF`**, with **`DRIVER`** named and reserved but deliberately left undefined.
   - **`ADMIN`** is held by the CEO/owner — the single most-privileged human. In Phase 2 there is exactly one ADMIN; the role name (rather than literally "CEO") is chosen so a second trusted owner-level account is a data change, not a schema change, and to align with better-auth's conventional `admin` role token. ADMIN can do everything OFFICE_STAFF can, plus every privileged/sensitive operation (raw GPS, observability, user/role administration).
   - **`OFFICE_STAFF`** is a trusted employee who does operational data entry. The threat model is *not* "office staff is hostile" but "limit the blast radius of a compromised office-staff session, and enforce the GPS/observability privacy splits the siblings require." OFFICE_STAFF therefore gets the Phase-1 operational CRUD (vehicles, drivers, customers, jobs, trips, fuel logs, expense logs) and **derived** GPS/operational views, but **not** raw-GPS-trace export, **not** observability access, and **not** user/role administration.
   - **`DRIVER` is reserved, not defined here.** A driver's permission shape is almost entirely *row-level* ("their own trips, their own traces"), which depends on the driver-app design (ADR-0026/0027 both flag driver-app mechanics as that slice's to settle) and on the row-level-scoping work this ADR defers (commitment 9). Defining DRIVER now would be speculative. The role *name* is reserved so the role type/enum and the permission map are designed to accept a third role without rework; its permissions land with the driver-app slice.

2. **Storage: a library-aware role field — recommended `user.additionalFields`; the `admin` plugin as the richer alternative; a library-unaware column rejected.** The role must travel on the session payload so enforcement is cheap (commitment 3); that requirement drives the storage choice, because better-auth's `getSession` returns only the user fields better-auth knows about. The three candidates:
   - **(Recommended) A better-auth-declared field via `user.additionalFields`** — a `role` field declared in the `betterAuth({...})` config with a string/enum type, a **least-privilege default (`OFFICE_STAFF`)**, and **`input: false`** so it can never be set through the public sign-up/update API (the single most important privilege-escalation defense — see commitment 8 and the security note). Because better-auth manages the field, it is returned on `session.user` automatically, it lands in the better-auth-owned `user` table via a normal Prisma migration, and the enforcement guard reads it with no extra query. This is the lightest primitive that satisfies ADR-0015's "build on better-auth's RBAC primitives" intent.
   - **(Alternative, available to the PO) The better-auth `admin` plugin** (and its `better-auth/plugins/access` access-control module: `createAccessControl`, statements, typed role/permission definitions). This is the most batteries-included path and gives client-side permission helpers, but it also brings user-management, **impersonation**, and ban machinery FleetCo does not need in Phase 2 — and impersonation in particular *widens* the threat surface (an admin acting as another user) in a way a two-role internal tool should opt into deliberately, not inherit by default. It remains a clean upgrade path if user-management or impersonation is later wanted (see "Revisit when").
   - **(Rejected) A raw Prisma `role` column better-auth is unaware of.** It would **not** appear on `session.user`, forcing either a per-request extra DB lookup inside the guard (a latency regression on every authenticated request) or a hand-maintained denormalization — fighting the library for no benefit and contradicting ADR-0015's intent. Rejected.

   The **final pick between the recommended `additionalFields` path and the richer `admin` plugin is the PO's**, because it trades minimal surface against built-in admin/impersonation features and is properly an owner call; the implementing ticket pins the exact better-auth config and verifies the field-on-session behavior, exactly as ADR-0026's turn-on ticket verifies its exporter header path. **No new top-level dependency is introduced either way** — the role primitives live inside the already-adopted better-auth, and the guard is NestJS-native (`@nestjs/common`), so CLAUDE.md's "propose a new dependency first" rule is not triggered.

3. **Attach point: the role lives on the User and rides the session; read at the guard with no extra query.** The role is stored **on the User** (commitment 2) and is read at enforcement time off **`request.session.user.role`** — the session being where it is *read*, the user being where it is *stored*. Because better-auth returns the managed user on the session, the role is present on the existing `SessionPayload` once the field is declared, and `SessionPayload`'s derivation (`auth.types.ts`) picks it up automatically with no `any`. Two rejected sub-alternatives: **denormalizing the role onto the `Session` row** is redundant — `getSession` already returns the user, so a session-row copy buys nothing and adds a sync obligation; **a per-request `prisma.user.findUnique` in the guard** reintroduces the query the session payload exists to avoid. One honest nuance the implementing ticket must handle: a **role change must propagate to live sessions** — a downgraded user's existing session should reflect the new role promptly. better-auth's revocable sessions (a property ADR-0015 valued over JWTs precisely for this) are the lever; the implementing ticket decides whether a role change force-revokes the user's sessions or relies on better-auth's session-refresh/cache window. This is a genuine decision the code slice settles, flagged here rather than silently assumed.

4. **Permission model: a single hardcoded in-tree role→permission map of coarse capability tokens — not a DB-driven policy store.** Controllers do not check roles directly; they declare the **capability** they require, and a single hardcoded table maps each role to the set of capabilities it holds. This indirection is the standard RBAC decoupling: adding or renaming a role touches one table, not every controller. The map is **hardcoded TypeScript in the auth module**, not a database-backed editable policy, because the permission model *is project memory*, and per ADR-0009 memory belongs in version-controlled files that diff cleanly and are code-reviewed — not in a mutable DB row a future agent cannot see in a diff and an attacker (or a fat-fingered admin) could rewrite at runtime. For two roles and a small capability set, a DB-driven permission editor is pure overhead and a new runtime-mutable security surface; it earns its place only when the role count grows or the PO needs to re-scope permissions without a deploy (see "Revisit when"). Capabilities are **coarse** (operation-class granularity, e.g. `customers:write`, `gps:read-raw`), not per-record — per-record scoping is commitment 9's deferred concern. The starter token set the implementing ticket finalizes, with the load-bearing entries fixed here:

   | Capability (illustrative) | ADMIN | OFFICE_STAFF |
   | --- | --- | --- |
   | Operational read+write — `vehicles:*`, `drivers:*`, `customers:*`, `jobs:*`, `trips:*`, `fuel-logs:*`, `expense-logs:*`, `reports:read` | ✅ | ✅ |
   | `gps:read-derived` — live-location map, geofence status, trip route summary | ✅ | ✅ |
   | `gps:read-raw` / `gps:export-raw` — query/export a named driver's full-resolution trace | ✅ | ❌ |
   | `observability:read` — traces/metrics surface (see commitment 6 for the SaaS nuance) | ✅ | ❌ |
   | `users:manage` / `roles:assign` — create accounts, assign roles | ✅ | ❌ |

   The exact token strings and the full operational list are the implementing ticket's to finalize; the **non-negotiable shape** is the GPS raw-vs-derived split and the ADMIN-only observability/user-admin tokens (commitment 6).

5. **Enforcement: a FleetCo-authored `RolesGuard` + decorators composing `AuthGuard`; 403 ≠ 401; opt-in restriction by default.** Enforcement is a NestJS-native `RolesGuard` at `apps/api/src/modules/auth/` that **composes** the existing `AuthGuard` (which runs first and attaches `session`), reads `session.user.role`, resolves the role's capability set from the commitment-4 map, and allows or denies. This is ADR-0021 Revisit-when (a) realized: the integration shape is untouched; only the guard surface grows. Specifics:
   - Controllers declare requirements with **`@RequirePermission('gps:read-raw')`** (preferred — capability-based, so a new role does not require editing every controller) or the coarser **`@RequireRole('ADMIN')`** shortcut for genuinely role-scoped surfaces (observability, user admin). Decorator metadata is read via NestJS's `Reflector`.
   - An authenticated caller lacking the capability receives **HTTP 403 Forbidden**, deliberately distinct from `AuthGuard`'s **401 Unauthorized** (401 = "I don't know who you are"; 403 = "I know who you are and you may not"). This distinction is part of the contract the implementing tests pin.
   - **Default posture is opt-in restriction, not global deny-by-default.** A route behind `AuthGuard` with no permission decorator stays accessible to any authenticated user — which, for ADMIN + OFFICE_STAFF (both of whom *should* do operational CRUD), is the correct default and makes applying this model to the existing Phase-1 controllers a near no-op. The **sensitive** surfaces (raw GPS, observability, user/role admin) — most of which are Phase-2 features that do not exist yet — are born with an explicit `@RequirePermission`. The stricter **global deny-by-default** (every route must name a permission or be denied) is the orthodox least-privilege posture and is recorded under "Revisit when"; it is dispreferred for Phase 2 because, against a *trusted* office-staff threat model, its cost (every controller must be annotated, and a forgotten annotation silently locks staff out) outweighs its benefit, while its core protection — keeping the sensitive surfaces ADMIN-only — is already achieved by annotating those few surfaces. The low blast radius on existing code (Phase-1 controllers keep working unchanged; new sensitive surfaces are gated from birth) is a deliberate property of this choice.

6. **The GPS and observability bindings — this ADR discharges ADR-0027 c7 and ADR-0026 c7.** The role set and permission map above are specified so that the two siblings' deferred access decisions are now concrete:
   - **Raw vs derived GPS (ADR-0027 c7).** `gps:read-raw`/`gps:export-raw` is **ADMIN-only**; `gps:read-derived` is **ADMIN + OFFICE_STAFF**. This makes raw-trace access "the most-privileged operational data access in the system," strictly above the derived live-map/geofence/route-summary views — exactly the requirement ADR-0027 fixed and left the mechanism of to this ADR. A driver's right to *their own* data (ADR-0013's data-subject-request commitment) is a separate path and is unchanged.
   - **Observability (ADR-0026 c7).** `observability:read` is **ADMIN-only**. One honest nuance, faithful to ADR-0026's own framing: in Phase 2 the trace/metric data lives in the external SaaS and is *actually* gated by that account's own credential (held by the CEO), and there is no in-app observability surface yet. The `observability:read` token is therefore the *specification of the gate* for the day any observability surface is exposed in-app or any SaaS-access delegation is considered — so that decision is governed by this role model rather than granted ad hoc, which is precisely what ADR-0026 c7 asked for.

7. **`GET /me` returns the role; the web gates UI for UX, not for security.** `GET /me` extends from `{ id, email }` to **`{ id, email, role }`** so the Next.js admin web can gate UI — hide the raw-GPS-export control, the observability link, and admin-only navigation from an OFFICE_STAFF session. This UI gating is **UX, not a security boundary**: the authority is always the server-side `RolesGuard` returning 403; a hidden button that is un-hidden by a tampered client still hits a 403. Stating this prevents a future agent from mistaking the client gate for the enforcement. The `/me` shape change is small and additive and lands with the RBAC slice.

8. **Role assignment is an admin-only operation and never client input; seeding extends the existing explicit pattern.** New users default to the **least-privileged** live role (`OFFICE_STAFF`); `ADMIN` is granted only by an explicit privileged action. Role is **never** accepted from a public sign-up/profile-update payload (`input: false` on the field, commitment 2) — a self-service request that could set its own role to `ADMIN` is a privilege-escalation hole and is closed by construction. Office-staff accounts are created by the CEO through the **explicit, idempotent `db:seed`-style pattern** ADR-0021 commitment 5 established (the existing `seed-admin.ts` generalizes to take an email + role, or a sibling `create-user` script is added) — credential and role operations stay explicit, never an implicit side effect of process boot, exactly as ADR-0021 reasoned. The data migration that adds the `role` column **backfills the one existing user to `ADMIN`** so the CEO's current session keeps full access across the change. The `role` column's data-classification mark (the `/// Tier N per ADR-0013` convention) is set when the column lands — it is **authorization metadata, not a credential and not PII**, so it is classified low (Tier 3 operational) and is **not** added to the pino `redact` list (knowing a session's role is operational signal, not a leak); its sensitivity is in *who may write it*, not in hiding its value. The implementing ticket sets the exact mark.

9. **Scope boundary: role-level capability gating now; row-level/field-level scoping deferred.** Phase-2 RBAC is **role-level capability gating** — "may this role perform this class of operation?" It is explicitly **not** row-level scoping — "may this user see *this particular* record?" Office staff in Phase 2 see the operational aggregates wholesale (that is the point of adding them: to offload data entry), not a filtered subset. Row-level scoping (a driver sees only their own trips and traces; a future multi-office structure partitions data by office) is named and **deferred**, because its first real consumer is the `DRIVER` role and the driver-app design, and building a record-scoping engine before that consumer exists would be speculative. The capability-token indirection (commitment 4) is chosen partly so row-level predicates can later attach to capabilities without re-architecting the guard. This boundary is the honest scope of "basic role-based access control" the roadmap names.

### Relationship to prior ADRs (what this realizes, discharges, and fulfills)

- **Realizes ADR-0015's** forward-reference ("better-auth ships RBAC primitives the Phase 2 RBAC introduction can build on") and resolves its Revisit-when (c): the primitives are sufficient (commitment 2), so no library swap is warranted.
- **Realizes ADR-0021's** Revisit-when (a): the integration shape is untouched; `AuthGuard` is composed by a new `RolesGuard`, not rewritten.
- **Discharges ADR-0026 commitment 7** (observability access → ADMIN-only token, with the SaaS-credential nuance) and **ADR-0027 commitment 7** (raw-vs-derived GPS access → `gps:read-raw` ADMIN-only, distinct from `gps:read-derived`).
- **Fulfills ADR-0013's** promissory note that Tier-3 "access follows the role-based access control rules **to be defined in Phase 2**." This ADR defines them. ADR-0013 is **not edited** (it is append-only); this ADR supplies the rules it pointed forward to.
- **Is the PO-confirmation vehicle** for the cross-cutting auth change CLAUDE.md step 4 requires before code. No RBAC schema, guard, or migration is written until this ADR is accepted.
- **On acceptance**, the glossary's "RBAC" and "Session-based authentication" entries gain a pointer to ADR-0028 and a one-line description of the concrete model, and the CURRENT_PHASE/retrospective record notes the model is set — these memory updates land with the implementing slice (not in this proposal, which leaves semantic memory pointing at the still-accurate "RBAC arrives in Phase 2" statement until the decision is ratified).

## Alternatives considered

**Adopt the full better-auth `admin` plugin as the primary (rather than the lighter `additionalFields` field).** The closest alternative and a legitimate PO choice (commitment 2 keeps it on the table). Dispreferred as the *default* because it ships user-management, ban, and **impersonation** surfaces FleetCo does not need in Phase 2; impersonation especially widens the threat surface in a direction a two-role internal tool should adopt deliberately, not inherit. The recommended `additionalFields` path gives the one thing Phase 2 needs (a role on the session) with the least surface; the plugin remains a clean upgrade if user-management or impersonation is later wanted.

**A library-unaware custom `role` column.** Rejected (commitment 2): it does not ride the session, forcing a per-request query or a denormalization sync, and it fights the library against ADR-0015's intent for zero benefit.

**A DB-driven / runtime-editable permission store.** A `Role` and `Permission` table with an admin UI to edit mappings. Rejected for Phase 2: for two roles it is pure overhead and introduces a mutable runtime security surface, and it puts the permission model — which is project *memory* — in a DB row instead of a version-controlled, code-reviewed, diffable file (ADR-0009). It earns its place only at a role/permission count a single-company internal tool is nowhere near. Recorded as a "Revisit when" trigger, not adopted.

**Role-based decorators only, with no permission indirection (`@RequireRole` everywhere, no capability tokens).** Simpler at first glance. Rejected because it couples every controller directly to role names: adding the `DRIVER` role (or splitting OFFICE_STAFF) would mean editing every annotated controller instead of one map. The capability-token indirection (commitment 4) localizes role changes to a single table and is the standard RBAC decoupling; `@RequireRole` survives only as a coarse shortcut for genuinely role-scoped surfaces.

**Global deny-by-default enforcement (every route must declare a permission or be denied).** The orthodox least-privilege posture. Rejected as the Phase-2 default (commitment 5) because, against a trusted-office-staff threat model, its cost — annotating every existing controller, and silently locking staff out of anything a future agent forgets to annotate — outweighs its marginal benefit, while its core protection (sensitive surfaces ADMIN-only) is already obtained by annotating those few surfaces. Kept as an explicit "Revisit when" for if office staff become less trusted or the role count grows.

**Define the `DRIVER` role fully now.** Rejected (commitment 1): a driver's permissions are mostly row-level ("their own data"), which depends on the deferred row-level-scoping work and the driver-app design that ADR-0026/0027 leave to that slice. Defining it now would be speculative; the name is reserved so the model accepts it without rework.

**Denormalize the role onto the `Session` row.** Rejected (commitment 3): `getSession` already returns the user, so a session-row copy is redundant and adds a sync obligation for no read-path benefit.

**Build the RBAC code now without a separate ADR.** Rejected on CLAUDE.md step 4: auth is a cross-cutting concern, and a second human role is a security-surface change that warrants explicit PO confirmation before code. This ADR is that confirmation gate; the siblings (ADR-0026/0027) both deferred concrete access decisions to it precisely so the PO ratifies one coherent access model rather than discovering it piecemeal in feature PRs.

## Consequences

### What this makes easier

The day the PO accepts this model, the RBAC code slice has a settled spec: a known role set, a known storage mechanism, a known enforcement guard that composes the existing one, and a known starter permission map with the load-bearing GPS/observability splits already drawn. The two sibling ADRs are unblocked — their deferred access questions now have concrete answers, so the GPS schema slice and the trace turn-on can proceed knowing exactly which role sees what. Because enforcement is opt-in restriction over a capability map, the existing Phase-1 controllers keep working unchanged and the new sensitive Phase-2 surfaces are gated from birth — a low-blast-radius migration. The capability-token indirection means the eventual `DRIVER` role is a new map row plus its row-level predicates, not a sweep through every controller. And the whole model is legible memory: a future agent, a future PO, or a security reviewer reads one ADR and one hardcoded map and knows who can do what and why.

### What this makes harder

A second class of human credential now exists — a second account to phish, leak, or misconfigure — and the system grows a privilege-escalation surface that must be defended (the `input: false` role field and admin-only assignment are that defense, and they are now permanent invariants every future auth change must preserve). The GPS/observability privilege splits become correctness obligations that a future feature can violate by forgetting a decorator on a new sensitive endpoint — the same standing "every new surface must be checked against the policy" cost ADR-0026's scrub seam and ADR-0027's denylist accepted. Role-change propagation to live sessions (commitment 3) is a new operational behavior to get right. And `GET /me`, the `User` schema, and the seed path all gain a small permanent surface area.

### Costs we accept

- **This model is set without a second human in production (ADR-0025).** Like the siblings' provisional numbers, the role boundary is drawn against the *anticipated* office-staff workflow, not an observed one; the first real office-staff use may show a capability mis-scoped (too tight or too loose), which the hardcoded map makes a one-line, code-reviewed change to correct.
- **Opt-in restriction is a deliberate not-maximally-strict default.** We accept that a forgotten decorator on a *future* sensitive endpoint would leave it open to OFFICE_STAFF, and we mitigate with the convention that sensitive surfaces are annotated at birth plus the "Revisit when" escalation to deny-by-default; we judge this proportionate to a trusted-employee threat model and not to a hostile-insider one.
- **The richer better-auth `admin` features are foregone in the recommended path.** Choosing the minimal `additionalFields` field means no built-in impersonation, user-management, or client-side permission helpers; if those are later wanted, adopting the plugin is the named upgrade. We accept the smaller surface now in exchange for less to secure and reason about.

## Revisit when

- **The `DRIVER` role / driver app arrives.** Define DRIVER's permissions and, with it, the deferred **row-level/field-level scoping** (commitment 9) — a driver seeing only their own trips and traces is the first real consumer of record-level predicates. Coordinate with ADR-0027's data-subject-request path for a driver's own data.
- **The role or permission count grows, or the PO needs to re-scope permissions without a deploy.** Revisit the hardcoded map (commitment 4) against a **DB-driven** policy store; the capability-token indirection is designed so this swap changes how the map is *loaded*, not how controllers declare requirements.
- **Office staff need observability or raw-GPS access** (ADR-0026 c7 / ADR-0027 c7 both point back here). Re-examine the ADMIN-only splits against the concrete need rather than granting access ad hoc; any change is governed here and reflected back in the sibling ADRs.
- **The trusted-office-staff threat model weakens** — more roles, less-trusted staff, an audit/compliance requirement — revisit the opt-in-restriction default in favor of **global deny-by-default** (commitment 5).
- **User-management, impersonation, or client-side permission checks become wanted.** Adopt the better-auth **`admin` plugin** (the named upgrade path from the recommended `additionalFields` field), and supersede commitment 2's storage choice with a follow-up note.
- **A security review at driver-app integration.** Introducing the third human role and mobile sessions is the natural point for a focused review of the now-multi-role auth surface (ADR-0012's baseline runs continuously; this is the human-review checkpoint), since "a second/third human role widens the threat surface" is the standing security note this ADR records.
- **ADR-0015 triggers (better-auth is swapped).** The `RolesGuard` + capability-map interface is the contract that should remain stable across a library swap; a successor library re-implements storage (commitment 2) behind the same guard surface, the same way ADR-0021 kept `AuthGuard` stable as the swap contract.
