# ADR-0018: NestJS, nestjs-pino, and Sentry as the API stack

- **Status:** Accepted
- **Date:** 2026-05-20
- **Decider:** Product owner (CEO)

## Context

The architecture overview and `CLAUDE.md` describe "NestJS for the API," "pino for structured logging," and "Sentry for error tracking" as load-bearing choices for FleetCo's backend. Until now, none of the three had a dedicated ADR. ADR-0001 (modular monolith) mentions NestJS only in passing as the framework that supplies the module pattern; ADR-0005 (TypeScript everywhere) references it as a TypeScript-native backend choice; ADR-0006 (vertical slices) does not name a framework at all. Neither pino nor Sentry is named in any ADR. The session-1 kickoff plan's gap analysis flagged this explicitly: NestJS, pino, and Sentry are named as load-bearing in `docs/architecture/overview.md` without dedicated ADRs that explain why those three were chosen over alternatives.

The threshold for writing an ADR per BOOTSTRAP step 7 is "would a future session need to know this?" A future session reading `apps/api` would see NestJS-flavored decorators, nestjs-pino-flavored structured logging, and a Sentry init in `main.ts` with no obvious explanation of why those three. The reasoning would live only in the heads of the people who chose them, until those heads forgot. The reasoning must be in the repo before that happens.

The three choices are also not independent. NestJS provides the application framework and module pattern. nestjs-pino provides structured logging inside that framework, with request-context middleware that propagates a request ID into every log line for the duration of a request. Sentry consumes structured-error context to produce useful triage signal. Choosing one without the others is incoherent: NestJS without a logger is silent; structured logs without error tracking miss errors that never reach the logger; Sentry without structured context produces low-signal stack traces. The three are best understood as one decision about the API stack's framework + observability spine, which is why they land in one ADR rather than three.

The decision is being made at Phase 0 Ticket 7 — the ticket that first creates `apps/api` as a running process. The ADR lands together with that code so that future readers see the decision and its first implementation in one diff.

## Decision

The FleetCo API at `apps/api` is built on **NestJS 11** with the **Express adapter** (`@nestjs/platform-express`), with **nestjs-pino** as the structured-logging integration (wrapping `pino` and `pino-http`), and with **`@sentry/node`** as the error-tracking SDK.

The four specific commitments are:

1. **Backend framework: NestJS 11** (current major as of 2026-05-20). The module pattern (`@Module`, dependency injection, public service interfaces) is the framework-level enforcement mechanism for the module-boundary discipline of ADR-0001.

2. **HTTP adapter: Express via `@nestjs/platform-express`.** NestJS's default. Better-auth's Phase-0 adapter (used in Ticket 10) is Express-ecosystem-first; nestjs-pino's request-context middleware assumes Express request/response shapes.

3. **Structured logging: nestjs-pino.** JSON output one line per event in production; pino-pretty single-line readable output in development (switched on `env.NODE_ENV === "production"`). Per-request UUIDs generated by nestjs-pino's `genReqId`, accepted from the client when an `x-request-id` header is present, surfaced in every log line for the request, and echoed back in the `x-request-id` response header. Redaction list per ADR-0013: Tier 1 fields (`authorization`, `cookie`, `set-cookie`, `password`, `token`, `secret`) plus a starter Tier 2 PII list (`driverName`, `licenseNumber`, `phoneNumber`) seeded so the redaction is in place when Phase 1 schemas introduce those fields. Censored value `[Redacted]`.

4. **Error tracking: `@sentry/node`**, initialized in `main.ts` behind `if (env.SENTRY_DSN)` so that local development with no DSN produces no Sentry telemetry. The DSN is Tier 1 per ADR-0013 and never appears in any committed file.

## Alternatives considered

### For the backend framework

**Express or Fastify directly, no framework.** The lightest option. Rejected because the module-boundary discipline of ADR-0001 has no framework-level support in this configuration — every boundary becomes a code-review concern with no machinery behind it. NestJS's `@Module` + dependency-injection container makes module boundaries first-class. For a one-person project relying on AI agents to maintain boundary discipline, framework support is not optional.

**Fastify via `@nestjs/platform-fastify`.** Faster than Express (~2-3× throughput on synthetic benchmarks); slimmer dependencies. Rejected because (a) throughput is not the bottleneck at FleetCo's scale (~100 vehicles, ~500 trips/day per overview.md), (b) better-auth's Phase-0 adapter (ADR-0015, Ticket 10) is Express-first and the Fastify path requires non-trivial glue, (c) much of the NestJS ecosystem documentation assumes Express request/response shapes. The throughput advantage is real but unused; the integration cost is real and paid.

**Encore.ts.** TypeScript-native framework with built-in observability and infrastructure provisioning. Rejected because (a) it is less well-known to AI agents than NestJS (per ADR-0005's training-mass argument), and (b) the boundary patterns it enforces (services + endpoints + subscriptions) overlap with NestJS's module pattern in ways that produce no net gain. Its provisioning story is a strength but one we do not need at our scale.

**AdonisJS.** TypeScript-native, opinionated, all-batteries-included. Rejected because its opinions (Lucid ORM, its own auth, its own validation) conflict directly with FleetCo's already-locked choices (Prisma per Phase 1, better-auth per ADR-0015). Adopting AdonisJS would mean either fighting its opinions or replacing already-decided architecture.

### For structured logging

**NestJS's built-in `Logger`.** Rejected because it is unstructured by default; producing JSON requires implementing a custom logger transport, at which point we have re-implemented nestjs-pino badly.

**Winston via `nest-winston`.** Mature; ecosystem-rich; well-documented. Rejected because pino is faster (synchronous-by-default; less allocator pressure), and because pino's JSON-by-default model fits the production-log-shipping pipeline we'll eventually build with less configuration. Winston is more flexible than pino but the flexibility we'd use is small.

**Hand-rolled pino without nestjs-pino.** Saves one dependency. Rejected because nestjs-pino's request-context middleware (which propagates the request ID and other context into every log line emitted during the request's lifetime via Node's `AsyncLocalStorage`) is precisely the feature we want, and re-implementing it would be more code than the dependency removes.

### For error tracking

**`@sentry/nestjs`.** NestJS-specific Sentry integration (filters, interceptors, request-context wiring). Rejected for Phase 0 because `@sentry/node` is sufficient for the immediate use case (catch unhandled errors, capture context), and `@sentry/nestjs`'s NestJS-specific surface adds value only when we wire request-context interceptors and request-aware spans, which is post-Phase-0 work. The threshold to switch is when the manual `@sentry/node` context wiring becomes verbose; see "Revisit when."

**Rollbar.** Comparable feature set. Rejected because Sentry has higher AI-agent training mass (per ADR-0005's argument) and its free tier comfortably covers FleetCo's projected volume.

**Bugsnag.** Comparable. Rejected for the same reasons as Rollbar.

**Self-hosted error tracking (e.g., GlitchTip, an open-source Sentry-compatible).** Rejected because operating yet another service contradicts the one-person-can-operate-it discipline that runs through every other architectural choice. SaaS is appropriate for error tracking; the data captured is operational debug context, not core business state.

## Consequences

### What this makes easier

Every module shipped from Phase 1 onward has a known shape: a NestJS `@Module` declaration with controllers and providers, services that inject the nestjs-pino `Logger`, and uncaught throws that reach Sentry with request-context tags. Future agents see this consistency in `apps/api/src/modules/health/` and produce code that fits without needing to invent patterns. The `apps/api/src/modules/<module>/` convention from CLAUDE.md gets framework-level reinforcement (each module is a `@Module`, with its public surface in service interfaces and its data in repositories).

The `/health` endpoint is the canonical probe target for the API-availability SLI defined in ADR-0011. Future readiness checks (DB, Redis, queue depth) attach to the same controller when Ticket 9 brings those services online.

### What this makes harder

NestJS adds compile-time complexity (decorators, metadata) that bare Express does not have. The `reflect-metadata` shim must be imported before any decorated class is loaded, which is the standard NestJS pattern but bears mentioning because mishandling it produces confusing decorator errors at startup. The CommonJS + decorator metadata interaction also means `apps/api`'s `tsconfig.json` cannot be `"module": "NodeNext"` without ESM friction; we pin to `"module": "CommonJS"` deliberately.

pino's JSON output is harder to read in raw form than Winston's default pretty output. We mitigate this with pino-pretty in development (`env.NODE_ENV !== "production"`); production gets the JSON we want for log-shipping.

### Costs we accept

- **NestJS surface area is larger than hand-rolled Express.** Onboarding cost for the founder and any future contributor; learning cost for AI agents that are not already strong on NestJS-specific patterns. The discipline payoff (framework-enforced boundaries) justifies it.
- **Sentry can capture Tier 2/3 data in stack traces, breadcrumbs, and tagged context.** The pino `redact` list protects logs but not Sentry directly; Sentry's `beforeSend` hook is the equivalent escape valve and is not wired in Phase 0 because there are no error-producing paths yet. This gap is named here and will be closed in a later ticket when the first user-attributable error paths land (likely Ticket 10 with auth, or the first Phase 1 slice).
- **Sentry's SaaS dependency.** We accept being on Sentry's reliability. The error-tracking surface is not on the critical path for serving requests; a Sentry outage degrades observability, not the product.
- **No tracing yet.** OpenTelemetry, distributed tracing, and `traceId` propagation are out of scope for Phase 0. nestjs-pino's `genReqId` produces request IDs but they are not OpenTelemetry trace IDs; a future ADR will introduce the trace-id discipline when measurement justifies it.

## Revisit when

- **`@sentry/nestjs`:** When the manual `@sentry/node` context wiring becomes verbose — for example, when we find ourselves duplicating request-context augmentation across multiple modules. The Phase 1 Vehicles slice is the first natural test of this; if the slice's Sentry wiring stays compact, `@sentry/node` is fine. If it grows tendrils, switching to `@sentry/nestjs` is the right move.
- **Fastify:** When measurement shows Express request-handling overhead is a real bottleneck. This is unlikely at our scale; the revisit-when is the alert, not a scheduled review.
- **Hand-rolled pino:** When a reliability incident traces to nestjs-pino's middleware behavior under load (e.g., a request-id collision, a context-leak across requests). nestjs-pino is well-maintained; this is a hedge, not an expectation.
- **NestJS:** When a separate TypeScript backend framework gains decisive AI-agent training mass over NestJS, or when a measured friction with NestJS's decorator patterns under strict TypeScript becomes a recurring source of bugs. The cost of framework migration is large; the revisit threshold is correspondingly high.
- **Tracing / OpenTelemetry:** When request volumes are high enough that point-in-time logs are insufficient for root-cause analysis, or when a Phase 2+ feature (driver app talking to API) introduces cross-process flows that need trace-level visibility.
