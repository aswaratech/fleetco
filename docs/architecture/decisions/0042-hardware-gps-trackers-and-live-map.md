# ADR-0042: Hardware GPS trackers via a self-hosted Traccar gateway + the live fleet map

- **Status:** Accepted (decisions taken interactively by the PO in the 2026-07-02 planning session; the PO merging this PR is the durable ratification ‚Äî see ¬ßPhase mismatch)
- **Date:** 2026-07-02
- **Decider:** Product owner (CEO)

## Context

The PO wants a **live map view of the yard and the fleet's current locations** in the admin web app. The read side is unusually ready: the ADR-0029 telematics backend (batch ingest ‚Üí BullMQ ‚Üí `GpsPing` with PostGIS Point, per-vehicle latest-fix endpoint built "for the live-location map"), the ADR-0030 geofence aggregate (the yard is representable today as a `GeofenceType.DEPOT` polygon with full CRUD and a Leaflet drawing editor), and the Leaflet/react-leaflet stack already shipped in `apps/web`. What does not exist is a **producer**: the ingest seam has only ever been fed synthetic test batches. ADR-0035 planned the driver's phone as the producer (slices D4‚ÄìD6, unbuilt), which tracks only while a driver is on-trip with the app running.

On 2026-07-02 the PO decided, with the trade-offs surfaced explicitly: **dedicated hardware GPS trackers, now** ‚Äî always-on tracking of the vehicles themselves, at a per-vehicle device + SIM cost, rather than waiting on (or being limited by) driver-phone GPS.

Two structural facts shape everything below. First, **ADR-0041** (ratified the same day) froze new feature scope until the first production deploy reaches daily use; its commitment 4 requires any new program to argue its case in a fresh PO-ratified ADR ‚Äî this is that ADR for the tracker + map program (see ¬ßPhase mismatch). Second, hardware trackers push positions to a server **24/7 over the public internet**, so the never-executed first production deploy (ADR-0014: single VPS + Docker Compose, artifacts CI-validated, `deploy.yml` zero runs) stops being a deferrable obligation and becomes a **technical prerequisite**: it is this program's opening ticket, M1.

## Decision

Put Teltonika-class hardware trackers on the fleet, ingest their positions through a self-hosted **Traccar** protocol gateway into the existing telematics pipeline, and ship a read-only **/map** page. Eleven commitments:

1. **Hardware: Teltonika FMC-class 4G units, hardwired.** The recommended device class is Teltonika FMC-series (FMC920/FMC130-class): 4G Cat-1 with 2G fallback (2G-only devices risk network sunsets), +10..+30 V DC input so one SKU wires into 12 V tippers and 24 V trucks/excavators alike, ignition sensing on DIN1, internal backup battery, and on-device flash buffering so coverage dead zones (mountain routes) lose nothing ‚Äî the device flushes when coverage returns. Budget alternative if the pilot warrants it: Sinotrack ST-901L 4G (H02 protocol ‚Äî the same gateway decodes it). Units are bought **NTA type-approved from Nepali suppliers** (type approval is the local regulatory requirement), with an NTC or Ncell prepaid data SIM per vehicle. Indicative costs for the PO: **NPR ~8,000‚Äì12,000 landed per Teltonika-class device (+install), NPR ~4,000‚Äì6,000 for the budget alternative, NPR ~150‚Äì350/month per SIM** (a tracker uses well under 50 MB/month, fitting the smallest monthly packs). A **2-device pilot** (one truck, one excavator; ~2 weeks) precedes fleet rollout.

2. **Gateway: a self-hosted Traccar container in the existing compose file.** Devices speak their native binary TCP protocols (Teltonika Codec 8; H02 for Sinotrack). Rather than implementing binary protocol decoding inside the monolith, a **version-pinned official Traccar image** (Apache-2.0) joins `docker-compose.prod.yml` as an infrastructure container ‚Äî the same category as Caddy/Postgres/Redis, **not** a split of the FleetCo application (module boundaries, one deployable app, and the no-message-broker rule are untouched; Traccar is proposed here as CLAUDE.md's dependency rule requires). Devices reach it on a published TCP port (5027 for Teltonika; 5013 added only if H02 devices are bought) ‚Äî device TCP goes straight to the container, since Caddy proxies HTTP only. Traccar's own web UI binds to `127.0.0.1` (operator access via SSH tunnel only). It decodes 200+ device protocols, so the device choice in commitment 1 is not a lock-in.

3. **Traccar's storage: a separate `traccar` database in the existing Postgres container.** Not Traccar's default embedded H2 file (corruption-prone, a second storage engine to back up), and not a second Postgres instance. One stateful engine, one backup story: `deploy/backup.sh` is extended to dump the `traccar` database too. The database must be **created explicitly** (the Postgres image's entrypoint creates only `POSTGRES_DB`, and only on first init of the data volume ‚Äî which happens at M1, before Traccar exists); the runbook carries the `CREATE DATABASE` + grant step. **No secrets enter the committed `traccar.xml`:** Traccar's `CONFIG_USE_ENVIRONMENT_VARIABLES=true` mechanism injects the database password and the forward auth header from the on-box environment.

4. **Forwarding: Traccar ‚Üí the monolith over the compose network.** Traccar's built-in position forwarding (`forward.type=json`, retries enabled with raised limits) POSTs each decoded position to a new adapter endpoint `POST /api/v1/telematics/ingest/traccar`. Known shape consequence, accepted and recorded: JSON forwarding sends **one position per request**, so the adapter enqueues batches of one ‚Äî fine at this fleet's scale (~a few positions/second worst case); the fleet-size threshold at which adapter-side batching becomes worthwhile is a tech-debt entry, not built now. Delivery is at-least-once end-to-end (device flash ‚Üí Traccar; bounded Traccar retries ‚Üí API), consistent with ADR-0035 c4's no-dedup posture; Traccar's own DB retains every decoded position, so an API outage longer than the retry window loses nothing permanently (a reconciliation/backfill job against Traccar's REST API is named tech-debt).

5. **Machine auth for the gateway‚ÜíAPI hop: a static ingest key.** A new Tier-1 secret `INGEST_API_KEY` (‚â•32 chars, validated in `apps/api/src/config/env.ts` like `BETTER_AUTH_SECRET`), checked by a dedicated `IngestKeyGuard` (`crypto.timingSafeEqual`; 401 on mismatch; **fails closed** ‚Äî absent key means the route refuses, never open) on its **own controller**, so the existing AuthGuard/RolesGuard chains are untouched. Rejected alternative: a machine user + bearer session ‚Äî better-auth 1.6.11 ships no api-key plugin, bearer tokens are session-derived and expire, so a machine would need a stored password and a re-login state machine: strictly more secret surface. The hop never traverses the public internet (compose-network-internal), so the static key is defense-in-depth, not the only wall. Ingested pings are stamped with a **seeded, credential-less gateway system user** (no better-auth account row, so it can never log in; the seed supplies an explicit id since `User.id` has no default; its role is inert ‚Äî the guard bypasses RolesGuard ‚Äî and the seed documents that rationale).

6. **Device‚Üívehicle mapping: a new `TrackerDevice` aggregate.** Trackers are assets with their own lifecycle (swapped between vehicles, kept as spares, retired, carrying SIM metadata), so a model ‚Äî not a `Vehicle.trackerImei` column: `imei String @unique` (Tier 3, like `registrationNumber`), nullable `vehicleId String? @unique` (FK Restrict; nullable = spare/uninstalled; unique = one active tracker per vehicle), `label`, `simMsisdn` (Tier 3), a `TrackerStatus` enum (ACTIVE/SPARE/RETIRED), `installedAt`, and the standard audit fields. CRUD API + a minimal `/trackers` admin surface, gated by new `trackers:read` (ADMIN + OFFICE_STAFF) / `trackers:write` (ADMIN) capabilities. The adapter maps Traccar's payload to the existing ping shape (knots ‚Üí m/s √ó0.514444, `course` ‚Üí heading, `fixTime` ‚Üí timestamp, `attributes.ignition` ‚Üí the new column) via a **tolerant, non-`.strict()` Zod schema** ‚Äî a deliberate, documented deviation from the house rule, because FleetCo does not own Traccar's contract and a Traccar upgrade must not become an ingest outage ‚Äî then **re-validates the mapped output through the existing ping bounds before enqueueing** on the existing `gps-ingest` queue. Unknown-IMEI and `valid=false` positions are dropped with a 202 (retrying cannot fix an unmapped IMEI), a Tier-3-only warn log (IMEI, never coordinates), and a dropped-count in the ack.

7. **Schema riders: `GpsPing.ignition Boolean?` + the composite latest-fix index.** Ignition (idle-vs-parked) is a core fleet signal trackers report; adding a column to the system's highest-row-count table later is painful, so the additive nullable column ships with this program (phone/synthetic pings simply leave it null). The latest-position query gets a hand-authored composite index `(vehicleId, timestamp DESC)`, and the now-redundant single-column `vehicleId` index is dropped in the same migration.

8. **Privacy: continuous asset tracking is an explicit decision, and every Tier-5 control extends to the gateway.** Hardware trackers report whenever powered/ignition-on ‚Äî a real change from ADR-0027 c2's on-trip-only minimization, which was person-centric (a driver's phone). This ADR decides: **continuous tracking of company-owned assets via dedicated vehicle hardware is in scope**; the vehicle is tracked, not a person's device; drivers are informed in writing (the PO owns the notice). All other ADR-0027 controls carry over unchanged **and extend to Traccar's own store**: the ‚â§90-day raw retention applies to Traccar's `positions` table via a deterministic daily SQL prune (the built-in `database.positionsHistoryDays` has reliability complaints; the prune mirrors the existing `traces-prune` pattern), and ‚Äî a gap the design review caught ‚Äî **Traccar's own application log records decoded positions**, so its logging is configured down/rotated, the log directory is classified Tier 5, and its retention is bounded. `tripId` stays null on hardware pings (trip-correlation is a later derived feature, named tech-debt).

9. **Device-spoofing residual risk, named.** The published tracker port accepts unauthenticated TCP from the whole internet, and tracker protocols identify devices by IMEI alone ‚Äî someone who knows a registered IMEI could inject fake positions upstream of the ingest key. Mitigations: Traccar registers **known IMEIs only** (no auto-register), the adapter applies plausibility bounds, and the fix-age UI makes anomalies visible. The residual risk is accepted for a single-tenant fleet tool and recorded here rather than hidden.

10. **Read path + the /map page.** One new fleet-wide endpoint `GET /api/v1/telematics/positions/latest`, gated `gps:read-derived` (ADMIN + OFFICE_STAFF, same tier as the existing per-vehicle `/location`): `SELECT DISTINCT ON ("vehicleId") ‚Ä¶` ordered by the new composite index, LEFT-JOINed from non-retired vehicles so untracked vehicles appear with `fix: null`, returning **one latest fix per vehicle** (honoring ADR-0027 c6's anti-circumvention rule ‚Äî never trails) plus a server-computed `fixAgeSeconds`. Never the geometry column; never logged; never in spans. The **/map page** (`apps/web/src/app/(app)/map/`) reuses the sanctioned Leaflet stack, OSM raster tiles, and the existing WKT‚ÜîLatLng serializers ‚Äî **zero new frontend dependencies**: vehicle markers colored by fix age (proposed: fresh <2 min / stale ‚â•15 min / dead ‚â•24 h / no-tracker ‚Äî ratified via the DESIGN.md PR), popups (registration, kind, speed km/h, ignition, fix age), the DEPOT geofence polygons drawn read-only as the yard, and **~20 s client polling paused while the tab is hidden**. No SSE/WebSocket ‚Äî a realtime channel would be a new cross-cutting pattern needing its own ADR; polling matches the existing refresh idiom and upgrading later is additive. Per the house UI-surface gate, a **DESIGN.md ¬ßSurfaces "Live map" section** precedes the page.

11. **Sequencing: the deploy is M1.** Tickets: **M1** first production deploy (operator-led: VPS ‚â•4 GB amd64, DNS, on-box secrets, first `deploy.yml` run, smoke, restore-from-backup drill, runbooks DRAFT‚ÜíACTIVE, first DORA entry) ¬∑ **M2** this ADR ¬∑ **M3** schema (TrackerDevice, ignition, composite index) ¬∑ **M4** trackers CRUD + UI + capabilities ¬∑ **M5** ingest adapter + `IngestKeyGuard` + system-user seed ¬∑ **M6** Traccar compose service + config + DB creation + retention + backup + `docs/runbook/traccar.md` ¬∑ **M7** latest-positions endpoint ¬∑ **M8** DESIGN.md map spec ¬∑ **M9** /map page + nav entry ¬∑ **M10** 2-device pilot ‚Üí fleet rollout (depends on M1, M4‚ÄìM6, **and M9**, since the pilot's acceptance is live fixes on the map). M3‚ÄìM9 are fully testable pre-deploy with synthetic pings, so code work parallelizes with the operator-led M1.

## Alternatives considered

**Driver-phone GPS first (ADR-0035 D4‚ÄìD6, the one ADR-0041-permitted stream).** Rejected by the PO: positions only exist while a driver is on-trip with the app running; the fleet-visibility goal (including parked/idle assets in the yard) wants always-on hardware. D4‚ÄìD6 are deprioritized, not cancelled.

**Implement tracker TCP protocols natively in NestJS.** Rejected: months of binary-protocol/ACK-framing/device-management work that re-implements what Traccar already does for 200+ protocols, couples FleetCo to one vendor's codec, and puts a raw TCP listener inside the API process.

**A vendor cloud platform (Trackon/GeoMate-style subscription) + REST polling.** Rejected: recurring per-vehicle platform fees, lock-in, uncertain API access ‚Äî and the fleet's raw Tier-5 location trail resident in a third party's cloud, contradicting ADR-0027's egress posture. (Their bundled devices remain a viable hardware source.)

**MapLibre GL / a keyed tile provider for the map.** Rejected for v1: Leaflet + OSM raster is already sanctioned, installed, and shipping in the geofence editor; MapLibre would be a new top-level dependency and vector tiles need a keyed provider. Revisit only if the always-open map's tile load conflicts with OSM's usage policy.

**SSE/WebSocket realtime channel.** Rejected for v1: a new cross-cutting architectural pattern (own-ADR territory) to replace a 20-second polling loop serving one or two concurrent operators.

## Consequences

**Easier.** The inert ADR-0029/0030 backend finally gets a real producer; the yard + fleet become visible at a glance; the first deploy ‚Äî the project's highest-leverage owed item ‚Äî lands as M1 and flips the DORA/SLO/runbook disciplines from stubs to live; the map arrives with always-on data rather than on-trip-only fragments.

**Harder / costs accepted.** A second server-side application (Traccar) to operate, upgrade deliberately, and back up (~1 GB RAM headroom; the VPS is sized ‚â•4 GB at M1). Recurring per-vehicle SIM costs and a monthly top-up chore (runbook line; owner = operator). A widened privacy posture (continuous asset tracking) taken as an explicit, recorded decision with driver notice. The ADR-0041 window is re-opened for this program ‚Äî mitigated by the deploy riding first inside it.

## Revisit when

- **The pilot (M10) concludes** ‚Üí confirm the device pick, SIM carrier per route coverage, and fleet rollout; record actuals against the cost estimates here.
- **20 s polling proves too slow** (operator feedback) ‚Üí the SSE/WebSocket channel gets its own ADR.
- **Fleet size approaches ~50+ devices** ‚Üí revisit adapter-side batching (commitment 4's recorded threshold) and Traccar resource sizing.
- **OSM tile usage policy or reliability bites** the always-open map ‚Üí evaluate a keyed tile provider (MapTiler-class) with Nepal coverage.
- **A Traccar upgrade is needed** ‚Üí deliberate PR: re-verify the forward payload against the adapter's contract test and the retention/logging config.
- **Trip-correlation of hardware pings** is wanted (auto-linking pings to Trips) ‚Üí its own design; `tripId` stays null until then.

## Phase mismatch (ADR-0041 commitment 4, exercised)

This is new feature scope opened before the first deploy reaches daily use, which ADR-0041 forbids without a fresh PO-ratified ADR arguing the case ‚Äî this section is that argument. The PO chose to open it with the audit's risk framing surfaced explicitly ("build both now, deploy later", 2026-07-02). The case: (a) the program's hardware makes **the deploy itself ticket M1**, so ADR-0041's central concern ‚Äî the deploy receding behind ever more built-but-undeployed scope ‚Äî is inverted rather than repeated; (b) the map is the read-side consumer that makes the already-built telematics investment observable; (c) all code tickets are synthetic-data-testable, so nothing here blocks or displaces the deploy work. ADR-0025 is not cited as precedent.

- **ADR-0027 (GPS data classification)** ‚Äî amended in one respect: commitment 2's on-trip-only collection scope does not govern dedicated hardware on company-owned vehicles (commitment 8 above); every other Tier-5 control carries over and extends to Traccar's store and logs. ADR-0027 carries a dated annotation.
- **ADR-0029 / ADR-0030** ‚Äî consumed unchanged: the ingest queue, `GpsPing` hybrid storage, the raw/derived read split, and the geofence aggregate are exactly what this program feeds and reads.
