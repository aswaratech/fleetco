# ADR-0038: Email/notification infrastructure — a transactional-email provider behind a thin mailer interface, a dedicated `NotificationModule` owning a BullMQ daily-repeatable scan modelled on the ADR-0029 T4 retention prune, a sent-reminder notification-log for send-once idempotency, the single CEO/operator as the v1 recipient, and a "reminder delivery" SLI

- **Status:** Accepted (ratified by the PO 2026-06-18 — see ## Acceptance)
- **Date:** 2026-06-18
- **Decider:** Product owner (CEO)

## Acceptance

Ratified by the Product Owner on 2026-06-18 (this session): the PO accepted all four decision records (ADR-0036–0039) together, with the explicit directive "I accept all ADRs, go ahead", and chose the **build-now** path (option (a)) for each. The implementing slice — **Program C's notification ticket**, compliance-source-first with the maintenance source wired after ADR-0037's producer ships — is authorized to proceed. The transactional-email provider's API key remains an operator-supplied production secret (never committed). The standing production-deploy caveat in §"Phase mismatch and the dependency posture (stated honestly)" is acknowledged and accepted by the PO.

## Context

FleetCo can compute that a compliance document is lapsing or a service is overdue, but it cannot **tell the operator** unless the operator happens to open the right page. Reminders are entirely **passive** today:

- **Compliance expiry (ADR-0031).** The Vehicle detail page paints an amber "Expiring soon" / red "Expired" `<Badge>` beside each of the three compliance documents (`bluebookExpiresAt` / `insuranceExpiresAt` / `routePermitExpiresAt`), and the Home dashboard rolls those into a worst-of-fleet count. The classifier — `complianceBadgeState(expiryIso, now, 30)` (`apps/web/src/lib/compliance.ts`) — is shipped and correct. But it only fires when the page renders. ADR-0031 §F drew the boundary explicitly: "this slice ships a passive visual indicator only; it does not notify, schedule, or instrument delivery… compliance reminder *delivery* — email/SMS/cron/queue and the 'reminder delivery' SLI — is **Phase 3** and stays there."
- **Service due/overdue (ADR-0037, DEC2).** The preventive-maintenance aggregate proposes a `serviceScheduleState` helper — the rotation of `complianceBadgeState` into "Service due soon / Service overdue" — surfaced as the same badge on the Vehicle detail page. ADR-0037 §F(a) deferred delivery in the same words: "**reminder *delivery*** — email/SMS/push notification when a service crosses into due-soon/overdue is **DEC3/ADR-0038**… this aggregate is the passive *producer* of 'service due/overdue' that the reminder channel will read, the same relationship the compliance badge has to its Phase-3 delivery."

Both siblings, by design, built the **passive producer** and named **this ADR** as the **active delivery layer** they always pointed at. This ADR is that layer.

What is missing is concrete and small in number:

1. **A channel.** FleetCo has no way to send an email (or any outbound message). There is no mailer, no transactional-email dependency, no SMTP wiring anywhere in `apps/api` (verified).
2. **A daily scan-and-send job.** A background job that, once a day, reads the due/overdue sources, decides what is worth telling the operator, and sends it.
3. **Send-idempotency.** A way to send a reminder **once per lapse**, not re-email the same overdue bluebook every single day until it is renewed — the failure mode that trains an operator to ignore the channel entirely.
4. **Recipient configuration.** Who the email goes to.
5. **The "reminder delivery" SLI.** ADR-0011's "Revisit when" anticipated it by name: "Phase 3's compliance reminders will need a 'reminder delivery' SLI"; the roadmap §"Phase 3" lists it in the SLI catalog.

Three pieces of machinery already in the repo make this cheap, and the recommendation is to **compose them, not invent**:

- **The BullMQ daily-repeatable (ADR-0029 T4, `apps/api/src/modules/retention/`).** The retention prune is the exact template a daily reminder scan follows: a dedicated `RetentionModule` owns the `traces-prune` queue via `BullModule.registerQueue` (per-feature ownership, ADR-0029 commitment 2); `RetentionService implements OnApplicationBootstrap` registers a **single keyed** `upsertJobScheduler` (bullmq 5.78's Job Schedulers API, not the deprecated `repeat`) — id `traces-prune-daily`, cron `0 3 * * *` — idempotently at boot, so a restart upserts the one entry rather than stacking duplicates; a `@Processor(TRACES_PRUNE_QUEUE, { concurrency: 1 })` worker does the work, wrapped in one span carrying only safe attributes. The root BullMQ config (the dedicated `maxRetriesPerRequest: null` connection, the `attempts: 3` / exponential-backoff / bounded-`removeOnComplete` / `removeOnFail: false` defaults) lives in the `@Global()` `QueueModule` (ADR-0029 commitments 2/3/5) — whose own comment already forward-references "any future **notifications** / report-generation queue" as the next adopter. A reminder scan is the same shape with a different body.
- **The SLI emitter (ADR-0011, `apps/api/src/common/sli.ts`).** The pure, dependency-free SLI vocabulary already defines two indicators (`api_availability`, `trip_creation_success`) plus the driver-app `trip_start_success`, each a constant `SLI_*` tag + a typed `*Signal` interface + an `isXGood` predicate + a `buildXSignal` builder, with the worker/controller logging one tagged line per event and a future 28-day report computing the good-share. The "reminder delivery" SLI joins this catalog as one more entry of the identical shape.
- **The two reminder sources.** `complianceBadgeState` / `worstComplianceState` (ADR-0031) and the proposed `serviceScheduleState` (ADR-0037) are the due/overdue **producers**; this channel reads their output and delivers it. Crucially, those classifiers live in the **web** app (`apps/web/src/lib/`), and the scan runs in the **API** — so the classification logic must become reachable API-side without the badge and the reminder ever disagreeing (the drift hazard addressed in commitment 6).

The data-classification and secrets posture is already settled and constrains the design: a recipient **email is Tier-2 PII** (ADR-0013) — and is **already** covered by the pino `redact` path `*.email` (`apps/api/src/app.module.ts`), so unlike the GPS keys this channel adds **no** redact-list entry; the provider **API key is a Tier-1 secret** (ADR-0013) that lives **only** in the production secret store (the deployment-ADR-0014 secret management), never committed (ADR-0012: "never including a real secret in any committed file"; GitHub native secrets scanning runs every push; the PR template's no-secrets checkbox), never logged (`*.secret` / `*.token` are already redacted).

This ADR **decides shape and policy only.** It writes **no code, no `schema.prisma` edit, no migration, and adds no dependency.** Those land with the implementing slice (Program C's notification ticket) once this ADR is accepted. Per the project's settled practice — ADR-0029 fixed the ingestion architecture before the GPS code, ADR-0030 fixed the geofence aggregate before the geofence code, ADR-0031 fixed the BS/dependency choice before the date code — the confirmation vehicle for a new top-level dependency plus a cross-cutting background subsystem is an ADR.

### Phase mismatch and the dependency posture (stated honestly)

A fresh reader must see three facts plainly before the PO weighs the pull-forward:

1. **This proposes Phase-3 / Program-C work ahead of its phase.** Email/notification infrastructure is **Phase-3 headline material** — the roadmap's §"Phase 3 — Calendar-driven concerns" names it directly: "email notifications for both maintenance and compliance, with SMS to be added later. The SLI catalog adds a 'reminder delivery' indicator." It is proposed here under the **ADR-0025 ahead-of-gate pattern** — the documented-exception mechanism that opened Phase 2 before Phase 1 reached daily use. But ADR-0025 was an explicit *one-time* exception scoped to the Phase-1 → Phase-2 transition; pulling Phase-3 scope forward is a **new** exception that needs its own deliberate decision, exactly as ADR-0037 framed its own pull-forward. This ADR records the proposal; accepting it is what authorizes the pull-forward. An agent must never self-accept this ADR, add an `## Acceptance` block, or flip the status — the PO ratifies separately.

2. **The first production deploy is still owed — and a reminder channel is doubly deploy-dependent.** Phase 1 is content-complete but **not in daily use**: the deploy, the restore-from-backup drill, live SLI/DORA reporting, and the retrospective sign-off all remain open (`docs/CURRENT_PHASE.md`; ADR-0025 commitment 1). Every feature added before the deploy widens the "built-but-unused" surface ADR-0025 flagged as its central risk. A notification channel widens it **more than most**: it can only ever *deliver* from a **running production system** with a configured provider, a verified sending domain, and a real inbox to receive into — none of which exist while the app has never run in production. A reminder system on an undeployed app is the purest form of the ADR-0025 concern: code shipped against assumption that cannot fire until the deploy lands.

3. **The richest reminder source depends on a not-yet-accepted ADR.** The **compliance** source (ADR-0031) is shipped and real today. The **service-due/overdue** source is ADR-0037, which is itself only `Proposed` — its `serviceScheduleState` helper and `ServiceSchedule` aggregate do not exist until ADR-0037 is accepted **and** its slice ships. So the honest ordering is: this channel can deliver **compliance** reminders the moment it is built, and **maintenance** reminders only after ADR-0037 lands. The implementing slice should be sequenced accordingly — compliance-first, with the service-due source wired in once ADR-0037's producer exists — or scoped to compliance-only until then. This is called out in the Decision (commitment 5), the Consequences, and "Revisit when."

The honest framing, as with ADR-0037: **this ADR proposes the pull-forward; it does not assume it.** The PO decides at acceptance whether to (a) accept and let Program C build the notification slice now (compliance-first, maintenance after ADR-0037), (b) accept but hold the implementing slice until the production deploy lands (so the channel is built against a system that can actually deliver), or (c) reject the pull-forward and keep this as a `Proposed` record for when Phase 3 arrives.

## Decision

**Adopt a transactional-email API provider (recommended: `resend`; runner-up: `postmark`) reached over HTTPS and wrapped behind a thin FleetCo-owned `Mailer` interface, pinned-and-verified at install and isolated to its first code ticket; build a dedicated `NotificationModule` (`apps/api/src/modules/notifications/`) that owns a `notifications` BullMQ queue and registers a single keyed `upsertJobScheduler` daily reminder-scan at boot — the exact ADR-0029 T4 retention-prune pattern; on each scan, read the due/overdue sources (ADR-0031 compliance today, ADR-0037 service-schedules once it lands), diff them against a new `NotificationLog` model that records which (subject, reminder-kind, state, occurrence) lapses have already been sent so each lapse is emailed once and not re-sent daily, and dispatch the newly-crossed items as one digest email per recipient rendered in the DESIGN.md §Voice register with dates in Bikram Sambat (ADR-0031); send to the v1 recipient — the ADMIN operator's `User.email` (ADR-0021/0028), with a `NOTIFICATION_RECIPIENTS` env override; instrument a "reminder delivery" SLI (ADR-0011) in `apps/api/src/common/sli.ts` as reminder-send success rate; keep the provider API key a Tier-1 secret (production secret store only, never committed, never logged) and treat recipient emails as Tier-2 PII (already redacted via `*.email`); and propose the dependency for PO acceptance per CLAUDE.md.** Ten commitments define the shape, grouped. As with its decision-record siblings (ADR-0029/0030/0031/0037), this ADR **recommends** and leaves the genuinely owner-level picks — the provider, the recipient configuration, the digest-vs-per-event cadence, the dedup-key granularity, and the phase pull-forward — to the PO at acceptance.

### A. The channel, the provider, and the new dependency

1. **A transactional-email API provider, reached over HTTPS, behind a thin `Mailer` interface (recommended: `resend`; runner-up: `postmark`).** *(The library-*approach* — a hosted transactional-email API over self-run SMTP — is the recommendation; the specific package is the PO's owner-level pick at acceptance.)* The recommendation is **`resend`** (the official Node SDK): it sends over a single outbound **HTTPS** call — no new SMTP egress port (commonly blocked on a VPS for spam reasons), the same HTTPS-out the box already does for Cloudflare R2 and Sentry; it ships first-class TypeScript types; its free tier (thousands of emails/month) dwarfs a once-daily digest to one operator; and a hosted provider owns deliverability (SPF/DKIM/DMARC alignment, IP reputation, bounce handling) that a self-run SMTP relay would make FleetCo's standing operational burden. **`postmark`** is the runner-up — best-in-class transactional deliverability and an equally clean API, dispreferred only on a tighter free tier; it is the drop-in if the PO prefers it. Per CLAUDE.md's "propose a new top-level dependency first," this ADR proposes the dependency; the code slice **pins an exact version and verifies at install** that it (a) imports and sends under plain Node with no browser/DOM coupling, (b) ships or has `@types`, and (c) completes a real smoke send to a verified test address. The provider is **wrapped behind a FleetCo-owned `Mailer` (or `NotificationChannel`) interface** — `send({ to, subject, text, html })` — so the SDK is the only place that knows the vendor, exactly as `nepali-date.ts` wraps `nepali-date-converter` (ADR-0031 commitment 1) and `apps/api/src/common/wkt.ts` centralizes the WKT builder: if verification fails or the PO flips to Postmark or to SMTP, only the interface's one implementation changes, never the scan or the digest renderer. The dependency is **isolated to the notification slice's first ticket** for a clean blast radius, as `bullmq` was (GPS T1), Leaflet was (geofence G4), and `nepali-date-converter` was (BS N1).

### B. The module and the daily-repeatable scan — the ADR-0029 T4 pattern, rotated

2. **A dedicated `NotificationModule` owning a `notifications` queue, modelled on `RetentionModule`.** A new `apps/api/src/modules/notifications/` module owns the **`notifications`** queue (the exact name the `QueueModule` comment already reserves) via `BullModule.registerQueue` — per-feature queue ownership (ADR-0029 commitment 2), so the scheduler, the worker, and the concern live together. It is a **dedicated** module (not folded into vehicles/telematics) for the same reasons `RetentionModule` is dedicated: it is a cross-aggregate background concern that reads several aggregates' due/overdue state through the shared `PrismaService` (the in-repo `ReportsModule` / `RetentionModule` precedent — the shared `PrismaService` is infrastructure, not another module's repository, so ADR-0001's no-cross-module-internals rule is not breached), and it exposes **no HTTP surface** — no controller, no `AuthModule` import — it is a scheduled job, not a request-handling feature. The root BullMQ config (connection + defaults) stays in the `@Global()` `QueueModule`; this module adds only its queue, its scheduler service, and its worker.

3. **A single keyed `upsertJobScheduler` daily scan, registered idempotently at boot.** A `NotificationService implements OnApplicationBootstrap` registers **one** repeatable via bullmq's Job Schedulers API — `upsertJobScheduler(NOTIFICATION_SCAN_SCHEDULER_ID, { pattern: NOTIFICATION_SCAN_CRON }, { name: NOTIFICATION_SCAN_JOB_NAME })` — keyed on a stable id (e.g. `reminder-scan-daily`) so each restart **upserts** the single entry rather than stacking a duplicate per boot, exactly as `RetentionService.onApplicationBootstrap` does. The cadence is a **once-daily morning** cron, tunable, with the same timezone note retention carries: production runs UTC, so e.g. `0 1 * * *` (≈ 06:45 Nepal, UTC+05:45) lands the digest at the start of the operator's workday; the exact time is the PO's to tune and is a restart-safe edit precisely because the scheduler id stays fixed. The queue-name / job-name / scheduler-id / cron / concurrency constants are **exported from one module file** (the retention `TRACES_PRUNE_*` discipline) so producer, worker, and module name one source of truth and a typo cannot silently wire a producer to one queue and a worker to another.

4. **An in-process, concurrency-1 worker that scans, then enqueues a send per recipient.** A `@Processor(NOTIFICATION_QUEUE, { concurrency: 1 })` worker runs the scan in-process (ADR-0029 commitment 5; no split deployment, ADR-0014 single-VPS), drained on Nest shutdown, wrapped in **one span carrying only safe attributes** (counts, not recipient addresses or document contents — the retention-processor posture). Concurrency 1 guarantees two scans never overlap. **Recommended decomposition for clean retry semantics:** the scan job computes *what to send* (idempotent — see commitment 5) and **enqueues one send job per recipient** onto the same `notifications` queue; each send job calls `Mailer.send` and is **independently retried** by the queue's default `attempts: 3` + exponential backoff, so a transient provider blip retries the one failed send rather than re-running the whole scan. A scan that produces an empty digest **enqueues nothing and sends nothing** — no "all clear" email (DESIGN.md §Voice: state the fact, no noise). The single-job degenerate (scan-and-send inline) is acceptable for the one-recipient v1; the scan→send split is the recommended shape because it isolates delivery failure from scan correctness.

### C. Idempotency — the notification-log and the send-once dedup key

5. **A `NotificationLog` model is the dedup ledger and the audit trail: send once per lapse, not daily.** The load-bearing correctness requirement is that an overdue bluebook is emailed **once when it lapses**, never re-emailed every morning until it is renewed — the behavior that would train the operator to filter the channel to trash. The mechanism is a new **`NotificationLog`** model (Tier-3 operational data; proposed here, added by the slice — no schema written in this ADR) that records every reminder dispatched, keyed by the tuple **`(subjectType, subjectId, reminderKind, state, occurrenceKey)`**:
   - `subjectType` + `subjectId` — e.g. `VEHICLE` + the vehicle id (later `SERVICE_SCHEDULE` + the schedule id).
   - `reminderKind` — e.g. `BLUEBOOK` / `INSURANCE` / `ROUTE_PERMIT` (later a service-schedule kind).
   - `state` — the threshold crossed: `expiring-soon` vs `expired` are **distinct** keys, so an item reminds at most once on entering each state (one "expiring soon" email, then one "expired" email), never daily within a state.
   - `occurrenceKey` — the **due date itself** (the expiry ISO date, or a service schedule's `nextDue` anchor). This is what makes **renewal** reset the reminder: when the operator renews the bluebook to a new expiry, the new date is a new `occurrenceKey`, so a future lapse of the renewed document is a genuinely new reminder rather than a suppressed duplicate.

   Each scan classifies the current sources, computes this key for every item now in a remind-worthy state, and **emails only the keys absent from the log** (the newly-crossed items). The log row is the idempotency ledger **across the scan→send boundary** (recommended: written at scan time as the intent, completed by the send with the provider's result + timestamp), so a crash between enqueue and delivery cannot double-send or silently drop. The log doubles as the **audit trail** ADR-0013 favors ("we notified about that lapse on date X" is a fact the system can produce). A periodic re-nudge for *still-overdue* items (a weekly escalation, distinct from the daily de-duplicated scan) is a **named, tunable follow-up**, not v1 — v1 is "once per state per occurrence."

### D. The reminder sources and the shared classifier — the drift hazard, made a commitment

6. **The sources are the ADR-0031 compliance state and the ADR-0037 service-due state, computed by the SAME classifier the badges use.** The scan reads two producers: vehicle compliance expiries via `complianceBadgeState` (shipped, ADR-0031) and — once ADR-0037 lands — service schedules via `serviceScheduleState`. The **non-negotiable correctness rule**: the reminder and the badge must classify an item **identically**, or an operator sees a red "Expired" badge on the page but never gets the email (or vice-versa) — the precise "drift is the failure mode" discipline the project enforces between the redact and span-scrub lists and between Tailwind and DESIGN.md. Today those classifiers live in **`apps/web/src/lib/`** and the scan runs in **`apps/api`**, so the implementing slice must make the **pure** classifier reachable from both without duplication that can drift. The recommended resolution is to host the pure helper where both consume one copy — extract `complianceBadgeState` (and the UTC-calendar-day rule it carries, ADR-0031 commitment 5) into a **shared location both apps import** (a `packages/` pure module, the monorepo's sanctioned shared-library home per CLAUDE.md) — rather than copy-pasting it into the API. The same applies to the BS date formatter (`formatNepaliDate`) the digest needs (commitment 7). The exact extraction is an implementing-slice detail; the **commitment** is that one pure classifier serves both surfaces.

### E. Recipient configuration and the email content

7. **Recipient v1 = the ADMIN operator's `User.email`, with an env override; content is a §Voice digest with BS dates.** The v1 recipient is the single CEO/operator, who is already a `User` (better-auth, ADR-0021) with an `email`. The recommendation is to send to the **ADMIN role's** `User.email` (ADR-0028) — reusing the existing auth aggregate, no new config surface, and a natural path to per-role reminders later — with a **`NOTIFICATION_RECIPIENTS`** env override (comma-separated) as the escape hatch for a non-user address or a shared inbox. A dedicated per-recipient **subscription/preferences** model (office-staff choosing which reminder kinds they receive) is a named follow-up, not v1. The **email body** is a **digest**: a short, factual list of the newly due/overdue/expiring items grouped by kind, every date rendered in **Bikram Sambat** via `formatNepaliDate` (ADR-0031) so the operator reads the calendar they think in, written in the **DESIGN.md §Voice register** — state the fact, no exclamation, no apology (subject e.g. "FleetCo — 3 items need attention", not "⚠️ URGENT!!!"). The renderer is a **pure function** (digest data → `{ subject, text, html }`), unit-testable in isolation like every FleetCo formatter; a minimal plain-text body is mandatory (HTML optional) so deliverability and accessibility hold.

### F. The "reminder delivery" SLI

8. **A `reminder_delivery` SLI in `sli.ts`: reminder-send success rate over the rolling 28-day window.** ADR-0011's "Revisit when" named this indicator; the roadmap §"Phase 3" lists it in the SLI catalog. The slice adds, in `apps/api/src/common/sli.ts`, a `SLI_REMINDER_DELIVERY = "reminder_delivery"` constant + a `ReminderDeliverySignal` interface + `isReminderDeliveryGood` / `buildReminderDeliverySignal`, mirroring the `trip_creation_success` shape exactly. The **valid event** is a reminder **send attempt** (a `Mailer.send` call against a real recipient) — a scan that finds nothing to send is **not** a valid event, so an idle day does not inflate the denominator (the same "count attempts, not non-attempts" discipline as the trip SLIs). The send worker logs one line per attempt tagged `sli: "reminder_delivery"` with `sli_good` (true on the provider's accept/ack, false on a thrown-and-rethrown send error) and, on failure, `error_kind` = the **exception class name only**, never `err.message` (which can embed the recipient address — Tier-2 PII — exactly the rule the trip SLIs follow for the vehicle/driver id). The **target is 99.0% over 28 days** to match ADR-0011's catalog; this constant only *instruments* it. Like the other Phase-2 SLIs, it emits structured signals now; the 28-day report is a **post-deploy operator concern** (and doubly so — there are no sends to measure until the channel runs in production).

### G. Secrets and data classification

9. **The provider API key is a Tier-1 secret; recipient emails are Tier-2 PII already redacted.** The provider API key (e.g. `RESEND_API_KEY`) is **Tier-1** (ADR-0013): it lives **only** in the production secret store (the ADR-0014 deployment secret management that ADR-0013 deferred to the deployment ADR), is **never committed** to any file (ADR-0012's "never including a real secret in any committed file"; GitHub native secrets scanning runs every push and over full history; the PR-template no-secrets checkbox), and is **never logged** (`*.secret` / `*.token` are already in the pino `redact` paths; the key is loaded through the typed `env` config like `REDIS_URL`, never string-concatenated into a log). **Recipient emails are Tier-2** (ADR-0013) and are **already** covered by the existing `*.email` redact path — so, unlike the GPS keys (ADR-0029 T2), this channel adds **no** redact-list entry; the implementing PR verifies a send attempt cannot leak the address into a log line or a span (the worker span carries counts only, not addresses).

### H. Scope boundaries — what this ADR does not decide

10. **Out of scope, stated so the boundaries are clean.** This ADR does **not** decide: (a) **SMS or push delivery** — email is the v1 channel (the roadmap's "email notifications… with SMS to be added later"); SMS needs a **Nepal SMS gateway** integration with per-message cost and a sender-ID registration with the Nepal Telecommunications Authority (a Nepal-specific compliance surface flagged for operator verification, not assumed here), and push needs the driver app (ADR-0033) + device tokens and only reaches drivers, not the office — both are added **channels behind the `Mailer`/`NotificationChannel` seam** later, not a rewrite. (b) **A generic notification/rules engine** (arbitrary "remind me when condition X" rules over any field) — that is the EAV/rules-engine anti-pattern ADR-0037 also rejected; v1 reads two well-known, typed sources. (c) **Per-recipient subscription preferences** — v1 is the single operator; a preferences model is a follow-up. (d) **In-app / notification-center UI** — this is the outbound channel, not an inbox surface. (e) **The exact provider SDK API and version semantics** — the code slice pins and verifies at install, as every prior new-dependency slice did. (f) **The final cadence, digest-vs-per-event choice, and dedup-key granularity** — recommended here, the PO's to confirm at acceptance.

## Alternatives considered

**A transactional-email API provider vs. `nodemailer`-over-SMTP.** `nodemailer` (+ `@types/nodemailer`) over an SMTP relay is the more portable, no-vendor-lock alternative, and it is a real option. Dispreferred as the recommendation because it pushes **deliverability operations** onto FleetCo — SPF/DKIM/DMARC alignment, IP/domain reputation, bounce and complaint handling — which a hosted provider does for you; because SMTP egress (ports 587/465) is **frequently blocked by default on a VPS** (the ADR-0014 topology) for spam reasons, whereas the API providers send over the HTTPS the box already egresses for R2/Sentry; and because `nodemailer` still needs an SMTP server *somewhere* (often a provider's SMTP endpoint anyway, re-introducing the vendor without its clean SDK). Kept available behind the `Mailer` interface: if the PO prefers SMTP/no-lock-in, the one interface implementation swaps with no change to the scan, the dedup, or the digest.

**Email-only vs. SMS vs. push for v1.** SMS reaches the operator even without an inbox open and is high-signal in Nepal, but it requires a **Nepal SMS gateway** (a local aggregator), carries per-message cost, and needs **sender-ID registration** with the telecom authority — a separate, Nepal-specific compliance integration the operator must drive, not something to assume in a v1 channel. Push requires the **driver app** (ADR-0033) and device-token plumbing and only reaches *drivers*, while the compliance/maintenance reminders are for the **office/operator**. Email reaches the CEO **today**, with the least integration, and the roadmap explicitly sequences "email notifications… with SMS to be added later." So v1 is email; SMS/push are added channels behind the same interface (see commitment 10 / "Revisit when").

**A daily digest vs. per-event emails.** Per-event (one email per item the moment it crosses a threshold) is simpler — no batching — but noisy: a fleet with several documents lapsing in the same window floods the inbox, and a flooded channel gets ignored. A **digest** (one email batching the day's newly-crossed items, grouped by kind) is the recommendation — fewer, denser, higher-signal messages that suit the §Voice register. The dedup key (commitment 5) makes "once per lapse" hold under *either* shape; the digest is preferred for signal-to-noise. The PO confirms at acceptance.

**Storing sent-reminder state vs. recomputing each run.** A stateless scan that recomputes due/overdue every day and emails whatever it finds **cannot know whether it already told the operator** — so it either re-emails the same lapse daily (spam) or, if it tries to avoid that by only emailing "new today," loses the signal across restarts and clock boundaries. The **`NotificationLog`** (commitment 5) is the small piece of durable state that makes send-once correct and auditable; the cost is one new table, which the audit-trail value (ADR-0013) repays. Rejected: the stateless approach.

**A generic notification abstraction vs. an email-specific channel for v1.** A fully polymorphic notification system (channels × templates × subscriptions × an event bus) is the over-engineered path — the same EAV/rules-engine anti-pattern ADR-0037 rejected for maintenance intervals. v1 is **email-specific**, but with the provider isolated behind a **thin `Mailer` interface** so SMS/push later is an *added implementation*, not a rewrite — the seam without the speculative abstraction, mirroring how ADR-0029 wired BullMQ once for the next feature to copy. The generic system, if ever warranted, is a much later and separately-justified thing.

**A system cron (or a `setInterval`) instead of the BullMQ repeatable.** Scheduling the scan with an OS cron calling a script, or an in-process `setInterval`, avoids the queue. Rejected for the same reason ADR-0029 commitment 12 put the retention prune on BullMQ: the project already has the in-process, DI-resolved, gracefully-drained, idempotently-registered repeatable pattern (T4); a parallel scheduling mechanism outside the app is new operational surface with worse restart/observability semantics. The reminder scan is the second adopter of the pattern the `QueueModule` comment anticipated, not a new mechanism.

**Build the notification infrastructure now without a separate ADR.** Rejected on CLAUDE.md step 4: a new top-level dependency, a new cross-cutting background subsystem, a new SLI, a new outbound-egress surface, and a Phase-3 pull-forward are exactly the changes that warrant PO confirmation before code. The ADR is that gate, mirroring ADR-0029 (architecture before GPS code), ADR-0031 (dependency before BS code), and ADR-0037 (shape before maintenance code).

## Consequences

### What this makes easier

Reminders stop being something the operator has to **go look for**. The amber/red badges ADR-0031 and ADR-0037 paint on pages the operator may never open become an email that arrives in the inbox the morning a document lapses or a service comes due — the whole point of compliance and maintenance tracking, finally closed. Because the design is "the T4 retention prune, rotated," the implementing slice is mostly composition of proven parts: the module/queue/scheduler shape is copied from `RetentionModule`, the SLI is one more entry in the established `sli.ts` catalog, the classifiers are the shipped `complianceBadgeState` (and the proposed `serviceScheduleState`), the digest dates reuse `formatNepaliDate`, and the provider hides behind a one-method interface. The `NotificationLog` gives a real audit trail — "we notified about that lapse on date X" becomes a fact the system can produce (ADR-0013). And it is legible memory: one ADR explains how a lapse becomes an email, why it is sent once and not daily, why email before SMS, and where the secret lives.

### What this makes harder

A new outbound surface enters the system — the API now **sends email to the outside world**, which is new operational surface (a misconfigured sender domain, a bounced address, a leaked API key, a provider outage) that did not exist when the app only served HTTP and ran internal jobs. The classifier-drift obligation becomes load-bearing across the app/web boundary: the badge (web) and the reminder (API) must compute identical states from one shared pure helper, or the operator gets an email for a lapse the page doesn't show, or worse, sees a red badge and never gets the email. The `NotificationLog` is new standing state whose dedup key must be exactly right — too coarse and renewals never re-remind, too fine and the operator is spammed. And a new vendor dependency joins the web app's `nepali-date-converter` and Leaflet as a thing whose major versions and free-tier limits must be watched.

### Costs we accept

- **A new dependency and a vendor before the deploy (ADR-0025), the most deploy-dependent feature yet.** The provider, the sending domain, and the recipient inbox are all things that only exist for real in production; a reminder channel built on an undeployed app cannot fire. We accept this as the explicit, PO-decided cost of the pull-forward — and it is the sharpest instance of the ADR-0025 "built-but-unused" risk, which is itself an argument the PO may weigh toward **building this only once the deploy lands**.
- **A hard dependency on a not-yet-accepted ADR for half the value.** The service-due/overdue source is ADR-0037, still `Proposed`. The channel delivers **compliance** reminders the day it is built and **maintenance** reminders only after ADR-0037 ships; we accept the ordering constraint (compliance-first, service-due wired in later) and the risk that ADR-0037's `serviceScheduleState` shape changes at its own acceptance.
- **Email content and cadence sized against assumption, not observed use.** The digest format, the morning send time, and the dedup granularity are guesses about how one Nepali operator wants to be reminded, not measurements; they are explicit and tunable so calibration is a small change once the channel runs.
- **A standing deliverability and secrets obligation.** A sending domain with DNS records (SPF/DKIM/DMARC) is real setup; the API key is a Tier-1 secret that must be rotated and never leaked. We accept these as the price of an active channel, mitigated by the hosted provider owning deliverability and the secret store owning the key.

## Revisit when

- **PO acceptance fixes the open picks.** Acceptance ratifies the provider (recommended `resend`, runner-up `postmark`, or the SMTP alternative), the recipient configuration (ADMIN `User.email` vs. env list vs. a future subscription model), the digest-vs-per-event cadence and the send time, and the dedup-key granularity. If the PO chooses differently, the implementing slice follows the PO's pick and this ADR is annotated.
- **The PO rules on the phase pull-forward and the deploy ordering.** Acceptance is where the PO decides whether the notification slice proceeds now (compliance-first), waits for the production deploy so the channel is built against a system that can deliver, or is held. An undeployed app carrying a reminder channel that cannot fire is the ADR-0025 "built-but-unused" risk at its purest; if it accumulates, that is the signal the deploy must precede further pull-forwards.
- **ADR-0037 is accepted or changes shape.** Confirm the `serviceScheduleState` the maintenance aggregate produces is the shape the scan reads; wire the service-due source in once ADR-0037's producer exists.
- **SMS or push is wanted.** SMS pulls in a Nepal SMS gateway + sender-ID registration (a Nepal-specific compliance step for the operator to drive); push pulls in the driver app (ADR-0033) + device tokens. Both are added channels behind the `Mailer`/`NotificationChannel` seam — the trigger to revisit the email-only v1.
- **A still-overdue re-nudge or per-recipient preferences are wanted.** The trigger to add a weekly escalation cadence (distinct from the daily de-duplicated scan) and/or a subscription model letting office-staff pick their reminder kinds.
- **Deliverability degrades, the provider majors, or the free tier is exceeded.** Re-verify the SDK and the sending-domain DNS at the bump; the `Mailer` interface keeps the blast radius at its one implementation.
- **The shared classifier or the BS formatter is extracted to `packages/`.** When the compliance classifier and `formatNepaliDate` move to a shared pure module so the API scan and the web badge consume one copy, confirm the badge and the reminder still classify identically (the drift guard, commitment 6).
