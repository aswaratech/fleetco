# ADR-0021: Authentication integration shape — API-owned handler, email + password, env-seeded admin

- **Status:** Accepted
- **Date:** 2026-05-21
- **Decider:** Product owner (CEO)

## Context

ADR-0015 chose **better-auth** as FleetCo's authentication library and committed to four substantive things: session-based auth (not JWTs); sessions stored server-side in Postgres in Phase 0–1 via better-auth's database adapter; opaque cookies as the session identifier; and a single admin user seeded from environment variables for Phase 0. ADR-0015 deliberately left the *integration shape* — the choices that arise when better-auth is wired into a NestJS API plus a Next.js admin web — to the implementing ticket (Ticket 10 of the kickoff plan). Those choices are: where the better-auth handler lives across the two-process boundary; which credential method is enabled; how cookies are configured for cross-port development; how CORS is opened to the web app; how the single admin is seeded; and what pattern protected routes follow.

This ADR is being written together with the code that implements Ticket 10, so that future readers see the integration choices and their first implementation in one diff. The constraints in play: TypeScript end-to-end with `strict: true` (ADR-0005); modular monolith with the auth module living at `apps/api/src/modules/auth/` (ADR-0001 and the architecture overview); Prisma + Postgres + Redis as the data plane with Prisma scoped to `apps/api/prisma/` (ADR-0020); shadcn-ui as the component primitive source on apps/web (ADR-0016, ADR-0019); the session secret is Tier 1 data per ADR-0013 and never appears in any committed file; the ADMIN_EMAIL is Tier 2 PII and the ADMIN_PASSWORD is Tier 1, both gitignored in `apps/api/.env`. The deployment ADR (ADR-0014, reserved) will later replace these dev cookie/CORS defaults with same-origin or proxy-based production settings.

The threshold for an ADR per BOOTSTRAP step 7 is "would a future session need to know this?" A future session reading `apps/api/src/modules/auth/` would see a NestJS factory provider holding a `betterAuth({...})` instance, a non-trivial bootstrap dance in `main.ts` (bodyParser disabled, handler mounted at `/auth/{*splat}`, body parsers re-attached, CORS enabled with credentials), a separate `db:seed` script for admin creation, an `AuthGuard` that future modules copy, and a `GET /me` controller — with no obvious explanation of why any of those shapes. The reasoning must be in the repo before that session arrives.

## Decision

The auth integration takes the following six-part shape. All six commitments land together in Ticket 10's PR.

1. **Handler location: API-owned.** better-auth's handler is mounted at `/auth/{*splat}` (Express 5 wildcard syntax) in `apps/api/src/main.ts` via better-auth's `toNodeHandler`. The Next.js admin web does not host any auth route handlers; it consumes the API's handler through better-auth's React client (`createAuthClient({ baseURL: ${NEXT_PUBLIC_API_URL}/auth })`) for client-side calls, and through a thin `fetch` wrapper at `apps/web/src/lib/session.ts` that forwards the inbound Cookie header to `/auth/get-session` for server-side gating. This honors ADR-0015's Context paragraph ("the API issuing and validating sessions and the web reading them"), ADR-0001's "auth module lives in the API" rule, ADR-0018's Express adapter pin (chosen specifically because better-auth's Phase-0 adapter is Express-first), and ADR-0020's "Prisma stays at `apps/api/prisma/`" decision (since the handler holds the PrismaClient, it must run where Prisma runs).

2. **Credential method (Phase 0): email + password.** better-auth's `emailAndPassword` provider is enabled with `autoSignIn: true`. Passwords are hashed by better-auth's default (`scrypt`) and stored in the `account.password` column (nullable, present only for email/password accounts; future OAuth accounts leave it null). No magic link, no OAuth, no passkey/WebAuthn in Phase 0.

3. **Cookie semantics: better-auth defaults, environment-aware.** `httpOnly: true` always. `sameSite: "Lax"` and `secure: false` in development (better-auth selects from `NODE_ENV !== "production"`); `secure: true` in production. The cross-port dev setup (apps/web on `:3000` ↔ apps/api on `:3001` or overridden) is same-site under the modern browser definition because both share the registrable domain `localhost`, so `sameSite=lax` works without `none + secure`. Production cookie domain and `secure` settings are deferred to ADR-0014.

4. **CORS posture: explicit allowlist + credentials.** NestJS `app.enableCors({ origin: env.CORS_ORIGIN.split(",").map(s => s.trim()), credentials: true })` after the better-auth mount in `main.ts`. The same allowlist is mirrored as better-auth's `trustedOrigins` config (read from the same `CORS_ORIGIN` env var) so the CORS layer and the better-auth origin-check layer cannot drift. `credentials: true` is required because cookies cross-origin; wildcard origin is forbidden by the browser when credentials are sent, so origins are enumerated. Default in `.env.example` is `http://localhost:3000`.

5. **Admin seeding: explicit `db:seed` script.** `apps/api/scripts/seed-admin.ts`, invoked via `pnpm --filter @fleetco/api db:seed`. Idempotent: queries `prisma.user.findUnique({ where: { email: ADMIN_EMAIL }})`; if absent, calls `auth.api.signUpEmail({...})`; if present, logs the existence and exits 0. `ADMIN_EMAIL` (Tier 2) and `ADMIN_PASSWORD` (Tier 1) are read from `apps/api/.env` (gitignored). The script is invoked manually from `docs/runbook/dev-setup.md`; it is not run automatically on every API boot. Explicit > implicit for credential operations.

6. **Protected-route pattern: `AuthGuard` + `GET /me`.** A NestJS `CanActivate` guard at `apps/api/src/modules/auth/auth.guard.ts` calls `auth.api.getSession({ headers })` (converting Node headers to Web headers via better-auth's `fromNodeHeaders`); attaches the resolved session to the request or throws `UnauthorizedException`. Other modules opt into auth via `@UseGuards(AuthGuard)`. The first user of the guard is `GET /me` on `AuthController`, which returns `{ id, email }` of the current admin and is the canonical "who is signed in" endpoint for both the acceptance test (401 without a session) and the web's server-side session check.

## Alternatives considered

**Web-owned handler.** Mount better-auth in a Next.js Route Handler at `apps/web/src/app/api/auth/[...all]/route.ts` (the pattern better-auth's docs lead with for Next-only stacks). Rejected because it inverts ADR-0001's module ownership ("the auth module lives in `apps/api/src/modules/auth/`"), forces `apps/web` to gain a `@prisma/client` runtime dependency that ADR-0020 deliberately scoped to apps/api, and contradicts ADR-0015's Context paragraph. The NestJS API would still need to validate sessions, so we would have to add a thin "ask the web app" hop that adds latency and a runtime coupling without removing the original coupling.

**Both-process handler.** Run the better-auth handler in both apps and have each validate sessions locally. Rejected because it duplicates the source of truth for sessions (two adapters, two connection pools to the same DB), multiplies the surface area for misconfiguration, and offers no Phase-0 benefit — there is no scenario in Phase 0 where the web alone needs to issue sessions.

**Magic link as Phase 0 credential method.** Email-only sign-in that ships a one-time link. Rejected because it requires SMTP in development (Mailhog or a real provider) and the placeholder login form already has an email-and-password shape from session 9. Magic link can land later under "Revisit when" without re-architecting.

**OAuth (Google or GitHub) as Phase 0 credential method.** Rejected because a single internal admin in Nepal does not justify a third-party identity dependency. Re-evaluate if the system ever gains non-internal users.

**Hard-coded credentials in env with no login form.** Conjure a session by knowing a shared secret. Rejected because the login form already exists from session 9 and because the Phase-1 RBAC introduction (office staff role) will need a real auth flow; building it now is cheaper than retrofitting.

**OnApplicationBootstrap admin seeding.** A NestJS lifecycle hook that creates the admin every time the API boots. Rejected because credential operations should not be implicit side effects of process start. Idempotency does not save it: a misconfigured `ADMIN_PASSWORD` would silently re-create the admin's password on the next boot, which is a confusing operational pattern. The explicit `db:seed` script is also the natural place to wire production seeding when ADR-0014 lands.

**Single endpoint for AC (reuse `/auth/get-session`).** Use better-auth's built-in `/auth/get-session` as the demonstration that unauthenticated requests return 401. Rejected because `/auth/get-session` returns `null` (and HTTP 200) when no session is present, not 401 — which would not satisfy the AC, and which would conflate library-internal routes with the FleetCo API surface. `GET /me` is a FleetCo route that returns 401 cleanly and is also useful in its own right (the server-side session check has a clear endpoint to call).

**Hand-rolled shadcn-ui components.** The fallback if the shadcn CLI emitted Tailwind-3-shaped code under our Tailwind-4 setup. Not exercised; the CLI at version 3.4.0+ supports Tailwind 4 and emitted clean output. Noted here so a future session that hits a CLI incompatibility knows the recovery path.

**No CSRF / origin-check enabled in better-auth.** Rejected by inaction — better-auth's origin check is on by default and we kept it on. The check fires on state-changing routes (sign-out, sign-up, password-change) and requires the request's `Origin` header to be in `trustedOrigins`. Browser-based flows always send `Origin`; only manual curl-style requests need to set it explicitly (documented in `docs/runbook/dev-setup.md`).

## Consequences

**What this makes easier.** Every Phase-1 module that wants authentication adds `@UseGuards(AuthGuard)` to its controller and reads `req.session.user` from the resolved request — no per-module session wiring. Future agents see the pattern in `apps/api/src/modules/auth/auth.controller.ts` and reproduce it. The web's gated routes follow a single pattern (Server Component calls `getServerSession()` and redirects to `/login` if absent), so layout-level vs page-level gating is a tactical choice, not a re-architecture. The session secret, admin credentials, and CORS origin live in one zod-validated env schema, so a missing or malformed value fails loud at API boot instead of silently at runtime.

**What this makes harder.** The bootstrap order in `main.ts` is non-obvious: `bodyParser: false` at NestJS construction, then `app.use("/auth/{*splat}", toNodeHandler(auth))`, then `app.useBodyParser("json")` and `app.useBodyParser("urlencoded", { extended: true })`, then `app.enableCors(...)`, then `app.listen(...)`. Each step matters: better-auth reads the raw request body itself and a body parser running first would consume the stream silently. A future agent who refactors `main.ts` without understanding this order will produce a build that boots but where sign-in returns inscrutable empty errors. This ADR's existence is the primary defense against that refactor.

Cross-port dev requires both CORS and cookie alignment to work. If `CORS_ORIGIN` does not include the web's origin, the browser blocks the response and the user sees a generic error in DevTools. If `BETTER_AUTH_URL` does not match the API's actual listening port, better-auth's internal URL construction can produce wrong redirect URLs. Both failure modes are surfaced as "What can go wrong" entries in `docs/runbook/dev-setup.md`.

**Costs we accept.** better-auth's smaller community (ADR-0015's mitigation) is now combined with NestJS-specific integration patterns that have less Stack Overflow corpus than Express-only or Next.js-only patterns; the official better-auth docs do not cover NestJS directly. The integration code we wrote is small (one factory in `auth.ts`, one module/controller/guard, ~30 lines of bootstrap in `main.ts`) and entirely owned in-tree, which means a future library swap or version upgrade rewrites only this code and not every consumer (the AuthGuard surface is what other modules see). The cost of the bespoke integration is paid once and at a small scale.

The web side's session fetch in `apps/web/src/lib/session.ts` is a hand-rolled `fetch` rather than better-auth's React client because the React client expects browser globals (`document.cookie`) that are absent server-side. The cost is duplicate URL knowledge (`${NEXT_PUBLIC_API_URL}/auth/get-session` is hard-coded in two places, one in `auth-client.ts` and one in `session.ts`). Both derive from the same env var so the drift surface is small.

The `account.password` column accepts NULL because OAuth accounts (future) will not have a password. For the Phase-0 single-admin row, the column is always populated. This is better-auth's schema choice, not ours; we keep the comment in `schema.prisma` noting the Tier 1 classification.

## Revisit when

- (a) **RBAC arrives in Phase 2** with office staff as a distinct user role. Better-auth's roles/permissions primitives become load-bearing; the `AuthGuard` may grow into `RolesGuard` or compose with one. The integration shape itself stays — only the guard surface grows.
- (b) **SMTP becomes available** (Phase 1+). Magic link as an alternative or additional credential method becomes a small change to the `betterAuth({...})` config plus an SMTP env var. The handler location, cookie semantics, and seed script do not change.
- (c) **ADR-0014 (deployment) lands** and names the production cookie domain, the production `secure` posture, the production CORS origin (likely a same-origin proxy), and the production session-secret store. This ADR's "dev defaults" become "dev settings; production settings in ADR-0014."
- (d) **The body-parser dance in `main.ts` becomes a recurring source of bugs.** If we find that auth middleware ordering breaks across NestJS upgrades or new global middleware additions, the right move is to refactor toward a NestJS interceptor or a dedicated express subapp for `/auth/*`. The revisit signal is the second incident, not the first.
- (e) **ADR-0015 itself triggers.** If better-auth is swapped for Lucia, Auth.js, or a successor, this ADR is superseded by the new ADR that describes the replacement library's integration shape. The `AuthGuard` interface is the contract that should remain stable across the swap.
