# ADR-0029: Telematics ingestion architecture — BullMQ integration shape and GPS-ping storage representation

- **Status:** Proposed
- **Date:** 2026-06-04
- **Decider:** Product owner (CEO) — pending acceptance

## Context

Phase 2 ("Driver app and telematics") is open per ADR-0025. Its three foundation ADRs are accepted — ADR-0026 (observability upgrade), ADR-0027 (GPS/location data-classification), and ADR-0028 (office-staff RBAC). Those three fixed the **policy** around telematics: how sensitive GPS is (Tier 5), how long raw pings live (a short window), how the egress is scrubbed, who may read raw vs. derived traces, and what gets traced and measured. None of them fixed the **architecture of the ingestion path itself** — the seam between "a GPS ping arrives at the API" and "a GPS ping is stored in Postgres, retained, and read back." That seam has two load-bearing, cross-cutting choices, and this ADR makes them so the follow-up GPS code program builds against a settled architecture rather than inventing one inline under delivery pressure.

The two choices are:

1. **How BullMQ is wired into the NestJS app.** ADR-0020 chose "Redis 7.4 + BullMQ" as the cache-and-queue substrate, but it chose the *substrate*, not the *integration shape* — exactly the gap ADR-0021 filled for authentication (ADR-0015 chose better-auth as the library; ADR-0021 decided how the library is mounted, where the handler lives, how the guard composes). BullMQ has the identical gap today: the dependency is sanctioned but unwired. There is **no BullMQ code anywhere in `apps/api/src`** (verified) and **no `bullmq` dependency in `apps/api/package.json`** yet. This is greenfield infrastructure, and the integration shape — module structure, queue topology, the Redis connection it uses, worker lifecycle, retry/backoff policy, scheduled jobs — is a cross-cutting concern that every future background-work feature (notifications, report generation, the retention prune) will copy. Per CLAUDE.md step 4, a cross-cutting concern plus a new top-level dependency requires PO confirmation before code; per the project's established pattern (ADR-0021 for auth, ADR-0028 for RBAC), the confirmation vehicle is an ADR.

2. **How GPS pings are stored.** This is the **first PostGIS geometry usage in FleetCo.** The baseline migration enabled the `postgis` extension (ADR-0020 commitment 7) and the glossary has named PostGIS/Geofence/Geospatial since Phase 0, but no table has yet held a geometry. Geofencing for the depot, customer sites, and route corridors is a **core Phase-2 deliverable** (`docs/product/roadmap.md` §"Phase 2"), and ADR-0027 commitment 4 requires **native GPS precision** in the operational window because geofencing needs meter-level accuracy. The storage representation must serve both the high-volume, high-frequency insert path and the spatial-query path, and it interacts awkwardly with Prisma (Prisma cannot natively select or insert a PostGIS `geometry` column). Picking the wrong representation taxes the highest-row-count table in the system on every read and every write, forever — so it is a decision to make deliberately, before the first ping is stored, which is the cheap moment (the same reasoning ADR-0027 used to set the classification before the schema).

The substrate this ADR builds on already exists and constrains the choices:

- **`RedisModule` / `RedisService`** (`apps/api/src/modules/redis/`) is a thin `@Global()` module wrapping a single `ioredis` connection, constructed with **`maxRetriesPerRequest: 1`** — chosen deliberately so the `/health/ready` probe (`apps/api/src/modules/health/health.service.ts`, which calls `redis.ping()`) **fails fast** when Redis is down instead of retrying for tens of seconds. This is load-bearing below: **BullMQ Workers require an ioredis connection with `maxRetriesPerRequest: null`** (BullMQ uses blocking commands — `BRPOPLPUSH`, `BZPOPMIN` — and throws at worker startup if `maxRetriesPerRequest` is any finite number). The health probe's `:1` and BullMQ's `null` are mutually exclusive requirements on the same setting, which forces a decision about whether BullMQ shares `RedisService`'s connection or gets its own.
- **`enableShutdownHooks()` is not currently called** in `apps/api/src/main.ts` (verified). `RedisService.onModuleDestroy` therefore only runs on an explicit `app.close()`, not on a SIGTERM/SIGINT from a deploy or container stop. Graceful draining of in-flight BullMQ jobs depends on Nest's shutdown lifecycle actually firing, so this gap is in scope for the worker-lifecycle decision below.
- **The Prisma schema** (`apps/api/prisma/schema.prisma`) declares `previewFeatures = ["postgresqlExtensions"]` + `extensions = [postgis]`, and the **`FuelLog` model** is the canonical money-and-FK pattern this ADR mirrors for `GpsPing`: a required `vehicleId` FK + a nullable `tripId` FK + a required `createdById` FK, all `onDelete: Restrict`, with single-column indexes on each FK and a `(date desc)` index for the default sort.
- **The Trip delete forward hook.** `TripsService.delete` (`apps/api/src/modules/trips/trips.service.ts`) maps only `P2025` (not-found) today; its comment and `TripsController.remove`'s comment both anticipate that "fuel logs and GPS pings, which will reference Trip, arrive in Phase 2," at which point the endpoint "will either switch to soft delete or grow a `P2003 → 409` catch arm." Adding `GpsPing.tripId` with `onDelete: Restrict` is one of the referencers that makes resolving that deferred decision due.
- **The pino `redact` list** in `apps/api/src/app.module.ts` (`req.headers.authorization`, `cookie`, `set-cookie`, `*.password`, `*.token`, `*.secret`, `*.email`, `*.driverName`, `*.licenseNumber`, `*.phoneNumber`, `*.contactPerson`) and the ADR-0026 commitment-5 span-scrub denylist are the two egress layers ADR-0027 commitment 5 requires the GPS keys to be added to — *by the code slice*, kept in sync.

This ADR **decides shape and policy only**, exactly as its three siblings did. It writes **no code, adds no dependency, writes no schema or migration, and edits no redact/scrub list.** Those land with the GPS code program that consumes the decisions fixed here. Authoring this ADR is the sanctioned in-bounds task for this program and the PO-confirmation vehicle for the cross-cutting BullMQ + first-PostGIS-geometry code that follows; the code waits for acceptance.

## Decision

**Wire BullMQ into NestJS through `@nestjs/bullmq` (recommended) over a thin global root `QueueModule` that owns the shared, dedicated `ioredis` connection (`maxRetriesPerRequest: null`, separate from the health-probe connection) and default job options, with named purpose-specific queues (`gps-ingest`, `traces-prune`) owned per-feature, in-process workers drained on Nest shutdown; store GPS pings in a `GpsPing` model whose FKs mirror `FuelLog`, recommending a hybrid coordinate representation — Prisma-native `Float` latitude/longitude (plus altitude/speed/heading/timestamp) as the canonical columns, alongside a Postgres `GENERATED ALWAYS … STORED geometry(Point, 4326)` column with a GIST index for spatial queries — every coordinate/movement column carrying the `/// Tier 5 per ADR-0027` mark; ingest pings through an authenticated batch endpoint that enqueues onto `gps-ingest` for a batching worker to bulk-insert; enforce retention with a repeatable `traces-prune` BullMQ job (default: hard-delete raw pings past a provisional 90-day window, aggregation deferred); and propose the `bullmq` (+ `@nestjs/bullmq`) dependency for PO acceptance — sanctioned by ADR-0020, not a forbidden broker.** Fourteen commitments define the shape, grouped into the five areas the code program needs. As with its siblings, this ADR **writes no code**; it recommends and leaves the genuinely owner-level picks — above all the coordinate representation and the dependency — to the PO at acceptance.

### A. BullMQ integration shape (modelled on ADR-0021)

1. **Library: `@nestjs/bullmq` layered on `bullmq` (recommended); raw `bullmq` is the alternative.** Use the official NestJS wrapper `@nestjs/bullmq` rather than driving the raw `bullmq` engine by hand. The wrapper gives NestJS-native dependency injection (`@InjectQueue('gps-ingest')`), decorators (`@Processor('gps-ingest')` + a `WorkerHost` subclass, `@OnWorkerEvent(...)`), and module-lifecycle integration (queues and workers are providers tied to Nest's startup/shutdown), so producers and consumers read like the rest of the codebase and resolve through the DI container instead of being hand-instantiated singletons. **The honest counter-precedent:** `RedisModule` wraps raw `ioredis` *directly* — there is an in-repo house pattern for "thin Nest module around the raw client," and one could follow it for BullMQ too. It is dispreferred here because BullMQ's surface (queues, workers, processors, schedulers, lifecycle, events) is an order of magnitude larger than `ioredis`'s connection surface, so the wrapper earns its keep where it did not for Redis; hand-rolling DI providers and `OnApplicationShutdown` plumbing for every queue and worker would be bespoke code we maintain for no benefit. **The final pick is the PO's** — both are sanctioned by ADR-0020 and neither is a new infrastructure dependency (see commitment 6).

2. **Module structure: a thin `@Global()` root `QueueModule` for shared connection + defaults, with per-feature queue ownership.** Register the BullMQ root configuration **once** in a global module (mirroring `RedisModule`'s `@Global()` reach) via `BullModule.forRootAsync(...)` — this holds the shared connection (commitment 3) and the default job options (commitment 5). Each **queue** is then registered and owned by the **feature module that uses it** (`BullModule.registerQueue({ name: 'gps-ingest' })` inside the telematics module; the retention concern owns `traces-prune`), so a queue's producer, its worker/processor, and the feature it serves live together. This honors ADR-0001's modular-monolith rule — a module owns its surface — better than the alternative of one global god-module that registers and owns *every* queue, which would force the telematics module to reach into a shared registry it does not own (a mild ADR-0001 smell) and would centralize unrelated concerns. The root-config-global + per-feature-queue split is the idiomatic `@nestjs/bullmq` shape and the recommended one; the single-global-queue-owner module is the dispreferred alternative.

3. **Redis connection: a dedicated BullMQ connection with `maxRetriesPerRequest: null` — NOT the health-probe connection.** BullMQ gets its own `ioredis` connection configuration (the same `REDIS_URL` host, a **distinct connection instance**), set to **`maxRetriesPerRequest: null`** as BullMQ requires. It does **not** reuse `RedisService`, whose `maxRetriesPerRequest: 1` is a deliberate fast-fail choice for the `/health/ready` probe and would make BullMQ workers throw at startup. The two connections are intentionally different and **must stay separate** — a future agent who "DRYs them up" into one shared connection breaks either the probe (if it takes BullMQ's `null`) or the workers (if they take the probe's `1`). This is the BullMQ analogue of ADR-0021's non-obvious `main.ts` body-parser ordering: a deliberate, fragile-looking arrangement whose rationale must be in the repo before the refactor that would undo it. (BullMQ also internally duplicates connections — a blocking connection per worker — from the config it is handed; letting BullMQ own its connection lifecycle from the root config, rather than passing it a shared `ioredis` instance, keeps that duplication contained and away from the probe connection.)

4. **Topology: named, purpose-specific queues — not one polymorphic queue.** Use distinct named queues per workload — at minimum **`gps-ingest`** (high-frequency ping ingestion) and **`traces-prune`** (low-frequency scheduled retention) — rather than a single polymorphic queue keyed by a `type` field. Named queues let each workload carry its own concurrency, retry, backoff, and rate-limit policy; be paused/drained independently; surface clean per-queue metrics (the telematics-ping-freshness signal of ADR-0026 commitment 6 is naturally a per-queue/per-job metric); and isolate failures. A polymorphic queue would couple the hot ingestion path to the slow scheduled prune — precisely the coupling ADR-0026 commitment 4 separated when it made "the unit of tracing the BullMQ batch-processing job, not the individual ping." Additional queues (notifications, report generation — both named in the glossary's BullMQ entry) are added the same way when those features land.

5. **Worker lifecycle and default policy: in-process workers, graceful drain on shutdown, bounded retry with exponential backoff, failed-set as dead-letter.** Workers run **in-process** — inside the same NestJS API process, not a separate deployment — honoring CLAUDE.md's "no split deployments" rule and ADR-0014's single-VPS topology; the named exit ramp is a dedicated worker process if the box ever saturates (see "Revisit when"). Graceful shutdown drains in-flight jobs: the worker's `close()` runs from Nest's shutdown lifecycle (`OnApplicationShutdown` / `WorkerHost` teardown), which **requires `app.enableShutdownHooks()` to be enabled in `main.ts`** — not currently the case (see Context), so the code slice turns it on (a change that also makes `RedisService.onModuleDestroy` fire on SIGTERM as a welcome side effect). Default job options on the root config: **`attempts: 3`** with **`backoff: { type: 'exponential', delay: ~2000ms }`**, **`removeOnComplete`** bounded (keep a small recent window or age-out, so completed jobs do not grow Redis unbounded), and **`removeOnFail: false`** (or a bounded keep) so jobs that exhaust their retries remain in BullMQ's **`failed` set as the de-facto dead-letter** for inspection — BullMQ has no separate DLQ primitive; the failed set is the idiom, optionally drained to a dedicated failed queue later. Concurrency is per-worker and per-queue (the ingest worker tuned to its batch size; the prune worker concurrency 1). These are defaults the code slice may tune per queue; the commitment is that they exist and are explicit.

6. **Dependency proposal: `bullmq` (+ `@nestjs/bullmq`) — sanctioned by ADR-0020, not a forbidden broker.** Per CLAUDE.md's "propose a new top-level dependency first," this ADR proposes adding **`bullmq`** (the engine) and, if the PO takes the recommended library path (commitment 1), **`@nestjs/bullmq`** (the Nest integration; it lists `bullmq` as a peer). This is explicitly **sanctioned by ADR-0020**, which already chose BullMQ as the queue substrate — it is the wiring of an already-decided choice, not a new infrastructure decision. It is emphatically **not** one of CLAUDE.md's forbidden message brokers (Kafka, RabbitMQ): BullMQ is a Redis-backed in-process job library, and the Redis it runs on already exists in the stack (ADR-0020). No new server, broker, or split deployment is introduced. The dependency is **proposed here and installed in the code program's first ticket**, not in this docs-only ADR.

*Forward-reference to ADR-0026:* the BullMQ telematics workers are **traced** (the batch-processing job is the span unit, not the ping — ADR-0026 commitment 4); **GPS coordinates are scrubbed** from span attributes before egress (ADR-0026 commitment 5, finalized against Tier 5 by ADR-0027 commitment 5); and the **telematics-ping-freshness SLI** is a job/queue metric (ADR-0026 commitment 6) that lands with the metrics work alongside this slice.

### B. GPS-ping storage representation

7. **`GpsPing` model and FK shape: mirror `FuelLog`.** A `GpsPing` row carries the raw fix and movement fields ADR-0027 commitment 1 enumerates — **`latitude`, `longitude`, `altitude`, `speed`, `heading`, `timestamp`** — plus FKs mirroring `FuelLog`: **`vehicleId` required, `onDelete: Restrict`** (a ping always belongs to a vehicle); **`tripId` nullable, `onDelete: Restrict`** (collection is scoped to on-trip windows per ADR-0027 commitment 2, so most pings carry a trip, but nullable is the honest shape for a fix captured at a shift boundary, and Restrict protects per-trip route reconstruction exactly as it protects per-trip fuel cost); and **`createdById` required, `onDelete: Restrict`** to `User`, filled from the authenticated principal per ADR-0021 (`request.session.user.id`), the audit anchor for who/what submitted the ping. Indexes mirror `FuelLog`: single-column on `vehicleId`, `tripId`, `createdById`, and a `(timestamp desc)` index for recency queries (plus the spatial GIST index of commitment 8). **Consequence for the Trip delete hook:** `GpsPing.tripId` (Restrict) joins `FuelLog`/`ExpenseLog` as a Trip referencer, which makes the deferred `TripsService.delete` `P2003 → 409` decision due — the code slice that lands `GpsPing` resolves it (a 409 catch arm mirroring `VehiclesController.remove`/`DriversController.remove`, or soft delete), rather than leaving a `P2003` to surface as a 500.

8. **Coordinate representation: a hybrid — Prisma-native `Float` lat/lon plus a generated `geometry(Point, 4326)` column (recommended).** Three options were weighed (the two rejected ones are detailed under Alternatives):
   - **(a) Two `Float` columns only** — Prisma-native and simple, but geofencing falls back to app-side or raw-SQL math with no spatial index.
   - **(b) A PostGIS `Unsupported("geometry(Point, 4326)")` column only** — real spatial indexing and `ST_Contains` geofencing, but Prisma cannot select or insert `Unsupported` types, so **every** ping insert and **every** coordinate read becomes `$queryRaw`/`$executeRaw` on the highest-volume table in the system.
   - **(c, recommended) The hybrid** — canonical **`Float` `latitude`/`longitude`** columns that Prisma reads and inserts natively, **plus** a Postgres **`GENERATED ALWAYS AS (ST_SetSRID(ST_MakePoint(longitude, latitude), 4326)) STORED`** `geometry(Point, 4326)` column with a **GIST index**, used only by spatial queries.

   The hybrid is recommended because it serves both masters: geofencing — a core Phase-2 deliverable that needs `ST_Contains`/`ST_DWithin` over a GIST index — gets a real PostGIS geometry, while the **high-frequency insert path and ordinary coordinate reads stay on the type-safe Prisma client** (raw SQL is confined to the inherently `ST_*`-shaped spatial queries, where it would live anyway). It satisfies ADR-0027 commitment 4's native-precision requirement (Postgres `double precision` holds native GPS lat/lon without truncation, and the geometry derives from it losslessly). And the **generated column cannot drift** from the floats — the database derives it — which is the same "drift is the failure mode" discipline the project enforces between the redact and scrub lists and between Tailwind and `DESIGN.md`. Prisma mechanics the code slice must handle, noted so they are not a surprise: the generated column and its GIST index are **hand-authored SQL appended to the Prisma migration** (Prisma's schema language expresses neither generated columns nor `geometry` types); the column is represented in `schema.prisma` as `Unsupported("geometry(Point, 4326)")?` so Prisma tracks its existence for migrate-diffing but never tries to select it; because it is `GENERATED … STORED`, inserts must **not** supply it — which Prisma satisfies for free by inserting only `latitude`/`longitude`; and the **`ST_MakePoint(longitude, latitude)` argument order is X,Y = lon,lat** (a classic PostGIS foot-gun) — the code slice gets the order right. **This is the genuinely owner-level pick** — it trades Prisma-native ergonomics against spatial-query power and a hand-written migration — so the recommendation is the hybrid, but the final choice among (a)/(b)/(c) is the PO's at acceptance.

9. **Tier-5 marking applies to every coordinate and movement column, and to the geometry.** Per ADR-0027 commitment 8's convention, **each** of `latitude`, `longitude`, `altitude`, `speed`, `heading` (and the derived `geometry` column) carries a `/// Tier 5 per ADR-0027 — …` triple-slash mark — ADR-0027 commitment 1 classes the whole `lat/lon/altitude/speed/heading/timestamp` fix as Tier 5, not just lat/lon. The mark states, e.g., "raw GPS latitude (WGS84); never logged (pino redact) and never exported in a span (ADR-0026 scrub); raw pings pruned/aggregated after the retention window." The FK and audit columns (`vehicleId`, `tripId`, `createdById`, `timestamp` as an operational time) are not themselves Tier 5 location data; `timestamp` is the fix time and travels with the Tier-5 fix but is not independently sensitive. Derived geofence-crossing events and trip route summaries, when those land, are marked **`/// Tier 3 per ADR-0013`** per ADR-0027 commitment 6 — not Tier 5 — subject to its anti-circumvention clause (a "derived" product dense enough to reconstruct the raw trail stays Tier 5).

### C. Ingestion shape

10. **Pings arrive at an authenticated batch endpoint that enqueues; a worker batches and stores.** An authenticated endpoint (e.g. `POST /api/v1/telematics/pings`) accepts a **batch payload** (an array of pings; a single ping is the degenerate batch-of-one), does minimal validation, **enqueues onto `gps-ingest`, and returns fast** (202/201) so the API thread is not blocked on the database write — the exact "background work that should not block the API: GPS ingestion" the glossary's BullMQ entry names. The **`gps-ingest` worker** dequeues and **bulk-inserts** (`createMany`) the batch. Batch-first is recommended because the driver app streams background location and will buffer/flush batches (offline queueing, whose mechanics ADR-0027 commitment 2 leaves to the driver-app slice), and because batching matches ADR-0026 commitment 4's "trace the batch job, not the ping" unit. The **producer is the future React Native driver app, which does not exist yet** — so the endpoint and worker are **built and tested without it**, with integration tests posting synthetic batches as an authenticated principal, exactly as auth and RBAC were built and tested before any office-staff account existed.

11. **RBAC posture: authenticated ingestion now (a `gps:ingest` capability, ADMIN-held), the read split per ADR-0028.** Ingestion is gated by the existing `AuthGuard`; the recommended shape is a dedicated **`gps:ingest`** capability (ADR-0028 commitment 4's capability-token indirection) held by **ADMIN** today so tests can exercise it, and **later granted to the `DRIVER` role** when that role is defined with the driver-app slice — a new map row, not a controller edit, which is the point of the capability indirection. **Reads** use the split ADR-0028 commitment 6 already fixed and the already-built `RolesGuard` enforces: **`gps:read-raw` / `gps:export-raw` (ADMIN-only)** for querying/exporting a named driver's full-resolution trace — the most-privileged operational data access in the system — versus **`gps:read-derived` (ADMIN + OFFICE_STAFF)** for the live-location map, geofence status, and trip route summary. The read endpoints land in a later code ticket (commitment 13, T5); this ADR fixes that they exist and which capability each requires.

### D. Retention job

12. **A repeatable `traces-prune` BullMQ job enforces ADR-0027's window; default is hard-delete, aggregation deferred.** Retention (ADR-0027 commitment 3) is enforced by a **BullMQ repeatable/scheduled job** on the `traces-prune` queue — the "likely a BullMQ scheduled job" ADR-0027's Consequences anticipated — added idempotently at boot (a single scheduler entry keyed so restarts do not duplicate it; the code slice pins BullMQ's repeatable-job vs. newer Job-Schedulers API against the installed version). It prunes raw pings older than the window — **provisionally 90 days, the PO sets the final number at acceptance** of ADR-0027. ADR-0027 commitment 3 allows either hard-delete *or* aggregation to a Tier-3 derived summary (simplified polyline, distance, stop/geofence events); the **default is hard-delete of raw pings past the window**, because aggregation is a feature (polyline simplification, stop detection) that does not exist yet — ADR-0027's own "Costs we accept" names deletion as the fallback until that feature is built. Aggregation-to-summary is the preferred end-state and lands as its own ticket when the PO wants it. **This ADR specifies, the code slice implements:** when the `GpsPing` columns land, the pino `redact` list (`apps/api/src/app.module.ts`) and the ADR-0026 span-scrub denylist **gain the GPS attribute keys** (ADR-0027 commitment 5's provisional set: `latitude`, `longitude`, `lat`, `lng`, `lon`, `altitude`, `heading`, `speed`, `coordinates`, `geometry`, `location`, `point`, `position`, and geofence polygon geometry), the two lists kept in sync. No redact/scrub code changes in this ADR.

### E. Follow-up code-slice sequence (proposed, not executed)

13. **The GPS code program's recommended ticket order.** This ADR sketches the sequence so the PO sees the path and the code program has a roadmap; it **does not execute any of it**. Each ticket is a vertical, independently shippable slice:
    - **T1 — `QueueModule` + `bullmq` dependency + worker lifecycle.** Install `bullmq` (+ `@nestjs/bullmq`); add the global root `QueueModule` (dedicated `maxRetriesPerRequest: null` connection, default job options); enable `app.enableShutdownHooks()` and wire graceful drain; prove enqueue → process → graceful-shutdown end-to-end with a trivial throwaway queue + worker + test, and assert the health-probe `:1` connection is untouched. Pure infrastructure, no GPS — so the dependency and lifecycle land and are reviewable in isolation.
    - **T2 — `GpsPing` schema + migration + Tier-5 marks + redact/scrub denylist updates.** The `model GpsPing` (FKs per commitment 7; coordinate representation per the PO's pick in commitment 8; the hand-authored generated-column + GIST-index SQL if hybrid); Tier-5 marks on every coordinate/movement column (commitment 9); the pino `redact` + span-scrub denylist edits landing **atomically** with the columns (so a ping is never loggable before its keys are denylisted); and resolution of the deferred Trip-delete `P2003 → 409` (commitment 7). `prisma migrate dev`.
    - **T3 — ingestion endpoint + worker.** The authenticated batch `POST` that enqueues onto `gps-ingest`; the worker that bulk-inserts; the `gps:ingest` capability; tests posting synthetic batches as an authenticated principal (no driver app); the batch job traced (ADR-0026 commitment 4) with GPS scrubbed from spans.
    - **T4 — retention/prune job.** The repeatable `traces-prune` job enforcing the (provisional 90-day) window with the hard-delete default (commitment 12); the job traced/measured; groundwork for the telematics-ping-freshness metric (ADR-0026 commitment 6) if metrics land here.
    - **T5 — RBAC-gated raw/derived read endpoints.** `gps:read-raw` (ADMIN) raw-trace query/export and `gps:read-derived` (ADMIN + OFFICE_STAFF) live-location / geofence-status / route-summary, enforced by the existing `RolesGuard`; geofencing via PostGIS `ST_Contains`/`ST_DWithin` over the geometry column is the derived layer's first real spatial query. (Geofence-polygon storage for the depot / customer site / route corridor — a `geometry(Polygon, 4326)`, Tier 3 per ADR-0027 commitment 6 — is the related configuration this layer reads against, and may be its own sibling slice.)

14. **What this ADR does not decide.** It does not design the driver app, its offline/queueing behaviour, or the `DRIVER` role's permissions (ADR-0026/0027/0028 and the driver-app slice own those). It does not choose the observability backend or wire metrics (ADR-0026). It does not finalize the retention window number or the aggregate-vs-delete end-of-window action (ADR-0027 and its acceptance own those). It does not build the geofencing feature, only the storage that enables it. And it does not pin the exact `@nestjs/bullmq` API surface or BullMQ version semantics — the code slice verifies those at install, exactly as ADR-0026's turn-on ticket verifies its exporter header path.

## Alternatives considered

**Drive the raw `bullmq` engine with hand-rolled DI and lifecycle (no `@nestjs/bullmq`).** The closest alternative to commitment 1, and it has an in-repo precedent: `RedisModule` wraps raw `ioredis` directly. Dispreferred because BullMQ's surface (queues, workers, processors, schedulers, events, lifecycle) is far larger than a single Redis connection, so we would re-implement the injection, decorator, and graceful-shutdown plumbing `@nestjs/bullmq` already provides — bespoke code maintained for no benefit. Kept available as the PO's call since both are ADR-0020-sanctioned and neither adds infrastructure.

**One global `QueueModule` that registers and owns every queue.** Mirrors `RedisModule`'s `@Global()` shape most literally. Rejected as the default (commitment 2) because it makes feature modules reach into a shared queue registry they do not own — an ADR-0001 modular-monolith smell — and centralizes unrelated workloads (hot ingestion next to a slow prune). The global module is kept only for the *shared connection and defaults*; queue ownership stays with the feature.

**Share `RedisService`'s connection for BullMQ.** Rejected hard (commitment 3): BullMQ requires `maxRetriesPerRequest: null`, the health probe requires `maxRetriesPerRequest: 1` for fast-fail, and the two are mutually exclusive on one connection. Sharing would break either the probe or the workers. A dedicated BullMQ connection is mandatory, not a preference.

**One polymorphic queue with a `type` discriminator.** Rejected (commitment 4): it forfeits per-workload concurrency/retry/rate-limit policy, independent pause/drain, clean per-queue metrics, and failure isolation, and it couples the high-frequency ingestion path to the low-frequency scheduled prune — the exact coupling ADR-0026 commitment 4 structurally separated.

**A separate worker process / split deployment for BullMQ.** Rejected for Phase 2 (commitment 5): it contradicts CLAUDE.md's "no split deployments" and ADR-0014's single-VPS topology. In-process workers are correct at this scale; a dedicated worker process is the named exit ramp under "Revisit when" if the box saturates — a measurement-driven move, not a speculative one.

**Two `Float` columns only, no PostGIS geometry (option a).** Rejected as the recommendation (commitment 8) because geofencing — a core Phase-2 deliverable — would have no spatial index and would re-implement point-in-polygon and distance math app-side, badly and slowly, exactly the work PostGIS exists to do. It is the simplest option and remains the PO's to choose if they judge geofencing can be deferred or done outside the database; the recommendation is that it cannot, at fleet scale.

**A PostGIS `Unsupported("geometry(Point, 4326)")` column only, no Float canonical (option b).** Rejected as the recommendation because Prisma cannot select or insert `Unsupported` types — so the highest-volume table's *every* insert and *every* coordinate read becomes raw SQL, a permanent friction tax on the hot path. The hybrid confines raw SQL to the spatial-query paths (where it is idiomatic anyway) while keeping inserts and ordinary reads type-safe. Geometry-only survives as a legitimate choice if the PO prefers a single source of truth for coordinates over Prisma ergonomics and accepts the raw-SQL ingestion path.

**Truncate/coarsen coordinates on ingest to reduce sensitivity.** Rejected here for the same reason ADR-0027 commitment 4 rejected it as the primary minimization lever: geofencing and route reconstruction need native precision in the operational window, and a coarsened trail is still a trail. Minimization is applied through retention and aggregation (commitment 12), not lossy storage.

**Synchronous ingestion (write the ping to Postgres in the request handler, no queue).** Rejected (commitment 10): it blocks the API on the database write for the highest-frequency request in the system, forfeits batching and backpressure, and contradicts ADR-0020's and the glossary's explicit "GPS ingestion is the BullMQ use case." The queue is the point.

**Aggregate-to-summary as the retention default instead of hard-delete.** Rejected as the *default* (commitment 12) only because the aggregation feature (polyline simplification, stop detection) does not exist yet; ADR-0027 names deletion as the fallback until it does. Aggregation is the preferred end-state and lands as its own ticket — this is a sequencing choice, not a rejection of aggregation.

**Build the BullMQ + GPS code now without a separate ADR.** Rejected on CLAUDE.md step 4: a new top-level dependency, a greenfield cross-cutting queue subsystem, and the first PostGIS geometry are precisely the changes that warrant PO confirmation before code. This ADR is that confirmation gate, mirroring ADR-0021 (auth integration shape before auth code) and ADR-0028 (RBAC model before RBAC code), so the PO ratifies one coherent ingestion architecture rather than discovering it piecemeal in feature PRs.

## Consequences

### What this makes easier

The GPS code program arrives with its architecture settled: a known queue library, module shape, connection rule, worker lifecycle, and retry policy; a known storage representation with the Prisma/PostGIS friction already mapped and confined; a known ingestion endpoint-plus-worker shape tested without the not-yet-built driver app; a known retention job with a default action; and a five-ticket roadmap. The three sibling ADRs' deferred hooks all snap into place — ADR-0026's batch-job trace unit and ping-freshness metric attach to named queues, ADR-0027's prune job and Tier-5 marks have a concrete schema to land on, and ADR-0028's read split has endpoints to gate. Because BullMQ is wired once as a global root with per-feature queues, the next background-work feature (notifications, report generation) copies the pattern instead of re-deciding it. And the whole ingestion architecture is legible memory: a future agent or the PO reads one ADR and knows how pings flow from phone to Postgres to report, and why each seam is shaped the way it is.

### What this makes harder

A new subsystem enters the stack — a job queue with workers running in the API process — which is new operational surface (a stuck queue, a poison job, a worker that will not drain) that did not exist when the API only served HTTP. The dedicated BullMQ connection is a permanent invariant a future "let's share one Redis connection" refactor must not violate. The hybrid storage adds a hand-authored SQL fragment to one migration (the generated column + GIST index) and splits coordinate access across two idioms (Prisma for lat/lon, raw SQL for spatial predicates) — a contributor must know which to reach for. Every new telematics column from here on must be checked against the Tier-5 marking convention and added to the redact/scrub denylists in sync — the same standing correctness obligation ADR-0026 and ADR-0027 already accepted. And landing `GpsPing.tripId` forces the previously deferred Trip-delete `P2003 → 409` decision to be made.

### Costs we accept

- **This architecture is set without production telemetry or the driver app (ADR-0025).** The queue defaults (concurrency, retry, backoff), the batch shape, and the in-process-worker choice are sized against assumption, not measured ping volume; the first real fleet load may show the single-process worker or the default concurrency mis-tuned. The defaults are explicit and per-queue-overridable precisely so calibration is a small change.
- **The hybrid storage trades one hand-written migration and a two-idiom coordinate access for spatial power.** We accept the `Unsupported`-column migration friction and the `ST_MakePoint(lon, lat)` foot-gun in exchange for GIST-indexed geofencing on the type-safe insert/read path.
- **In-process workers couple job throughput to API process health.** A flood of GPS jobs competes with HTTP request handling for the same process on a 2–4 GB VPS. We accept this at Phase-2 scale; the named mitigation is a dedicated worker process, deferred until measurement justifies it.
- **The recommendation leans on `@nestjs/bullmq`'s maintenance and BullMQ's API stability.** A version that changes the repeatable-job/scheduler surface or the connection contract is a coordinated upgrade; the env-and-config-driven shape keeps the blast radius small, and the code slice pins exact APIs at install.

## Revisit when

- **PO acceptance fixes the open picks.** This ADR is Proposed; acceptance ratifies the library (commitment 1), the coordinate representation among (a)/(b)/(c) (commitment 8), and the dependency (commitment 6). If the PO chooses differently from the recommendations, the code slice follows the PO's pick and this ADR is annotated accordingly.
- **The single-VPS box saturates under telematics load (ADR-0014).** In-process workers competing with HTTP handling is the trigger to split workers into a dedicated process — the named exit ramp in commitment 5 — and possibly to revisit the connection/topology for a multi-process deployment.
- **GPS-ping volume threatens Redis memory or the queue's throughput.** Revisit `removeOnComplete`/`removeOnFail` bounds, batch sizes, per-queue concurrency, and whether the ingestion path needs rate limiting or partitioning beyond a single named queue.
- **Prisma gains native PostGIS `geometry` support (out of preview, with select/insert).** Revisit the hybrid — a future Prisma may let the geometry column be read/written natively, collapsing the two-idiom split and removing the hand-authored SQL. The migration SQL is standard and portable, so the move would be mechanical.
- **The aggregation feature is wanted (ADR-0027 end-of-window action).** When the PO prefers aggregate-to-Tier-3-summary over hard-delete, the retention job (commitment 12) grows the polyline-simplification/stop-detection step as its own ticket, governed by ADR-0027.
- **The `DRIVER` role / driver app arrives (ADR-0028 commitment 1).** Ingestion auth moves from the placeholder ADMIN-held `gps:ingest` to the `DRIVER` role (a map row, not a controller edit), and the driver-app slice settles the offline/queueing mechanics ADR-0027 commitment 2 left open — including whether a batch flushed after shift-end is in or out of the collection window.
- **`@nestjs/bullmq` or `bullmq` majors past the version pinned at install and changes the scheduler, connection, or worker-lifecycle surface.** The decisions here are expressed against that surface; an SDK major re-confirms them, coordinated the way ADR-0026's "Revisit when" handles its `@sentry/node` dependency.
