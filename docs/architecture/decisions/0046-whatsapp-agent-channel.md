# ADR-0046: WhatsApp agent channel — Twilio inbound/outbound over the existing chat agent, zero-dependency, per-user phone links

- **Status:** Proposed (decisions taken interactively by the PO across the 2026-07-05 and 2026-07-08 planning sessions; the PO merging this PR is the durable ratification — see §Phase mismatch. An agent must never self-accept this ADR, add an `## Acceptance` block, or flip the status — the PO ratifies separately, per the ADR-0038 rule.)
- **Date:** 2026-07-08
- **Decider:** Product owner (CEO)

## Context

The PO wants **two-way agent interaction from WhatsApp**: send a message to the company's WhatsApp number — "add this vehicle", "what did we spend on fuel this month", "log 40 litres for the tipper" — and have the existing AI chat agent (ADR-0043) execute or answer it and reply in the same thread. The agent already exists and is validated against the real provider; what is missing is a *channel* that lets the PO reach it from the phone they already carry, without opening the admin web.

Five PO decisions were taken interactively, each with trade-offs surfaced:

1. **Provider = Twilio WhatsApp API** (chosen over the Meta Cloud API direct recommendation — faster onboarding and a free sandbox for development; the per-message cost and the Twilio dependency are recorded in §Consequences).
2. **Scope = text + receipt photos.** The text path is the day-one value. The photo path rides the ADR-0044 image-intake pipeline, which is **currently paused** (`AGENT_OCR_URL` unset — the 2026-07-06 V0 decision), so the WhatsApp photo path is **built but inert**, exactly as V3–V7 sit tested-but-dormant on `main`.
3. **Senders = configurable per-user verified numbers** (chosen over PO-number-only). This is a real mapping table, but it is **plumbing, not an authorization change**: the authorization stays `agent:use` (ADMIN-only, ADR-0043 c1). Day-1 the table holds one row — the PO's number → the PO's ADMIN user.
4. **Zero new npm dependency** (chosen this session): Twilio inbound-signature verification and outbound sends use raw `fetch` + `node:crypto`, exactly as ADR-0043 talks to DeepSeek over raw `fetch`. The official `twilio` SDK was offered and declined (see §Alternatives).
5. **Photo path built now, inert** (chosen this session): the inbound media path is built and tested behind the standing image pause rather than deferred, so it is ready the instant image intake un-pauses.

The mechanical ground is good, and this ADR's design is **composition of proven parts, not invention**. A whole-`main` verification (2026-07-08) confirmed the reuse surfaces:

- **Inbound machine-ingress** has a template: `telematics/ingest-key.guard.ts` (`IngestKeyGuard`) — a DI-token secret, **fail-closed 503** when unconfigured, `crypto.timingSafeEqual`, guarding a controller with **no `AuthGuard`** (a machine caller), 202-fast, loose-boundary-then-canonical validation, mounted at `@Controller("api/v1/telematics/ingest")`.
- **Async + outbound** has two templates: the `telematics` `gps-ingest` queue (enqueue → 202 → `WorkerHost` processor) and — the closer one — `notifications/` (`NotificationModule`: a *dedicated delivery module* with the `Mailer` seam, one queue driven by a job-name switch, and a `NotificationLog` audit row carrying `providerMessageId` + a `@@unique` dedup key).
- **The agent** exposes `AgentService.runTurn(conversationId, content, actor, attachmentId?)` (`actor = { userId, role }`), with an in-memory in-flight lock that 409s a concurrent same-conversation turn, a 40-message replay window, and an honest degradation when the vision extractor is unconfigured. `AgentModule` currently exports only `LlmClient`.

What does **not** exist: any Twilio code, any inbound webhook for messaging, any phone→user mapping (the `User` model has no phone field; `Driver.phone`/`Customer.phone` are loose, un-normalized, un-indexed, non-unique), and any delivery ledger for a messaging provider.

Two facts constrain the design and are load-bearing below. First, **`runTurn` trusts the caller-supplied `actor` and does not itself re-check `agent:use`** — the HTTP path's `RolesGuard` did that. The webhook has no `RolesGuard`, so the WhatsApp resolver/processor **becomes the authorization chokepoint**; the phone→user mapping is security code, not plumbing. Second, **a Twilio request signature carries no timestamp or nonce**, so a captured valid webhook POST replays forever — signature verification is authentication, not a replay defense; inbound deduplication is therefore mandatory, not optional.

ADR-0041 (ratified 2026-07-02) freezes new feature scope until the first production deploy reaches daily use; its commitment 4 requires this fresh ADR (see §Phase mismatch). All production use of this channel gates on the M1 deploy — the inbound webhook needs a publicly reachable, 24/7 HTTPS endpoint that does not exist until the app runs in production.

## Decision

A new `apps/api/src/modules/whatsapp/` module: an inbound Twilio webhook that runs the existing chat agent **as the resolved human behind a verified phone number** and replies over WhatsApp, with its own provider seam, queue, phone-link schema, and delivery/idempotency ledger — reusing ADR-0043's tool registry, audit spine, autonomy posture, budgets, and redaction contract unchanged, and adding **no new agent capability**. Nine commitments.

1. **A dedicated `WhatsAppModule`, on the `NotificationModule` precedent (not the Traccar one).** WhatsApp is a full delivery concern — it brings its own provider seam (`WhatsAppSender` Twilio/Mock/factory), its own `whatsapp-inbound` queue and processor, its own signature guard, its own audit table, and new phone-link schema. That profile matches `NotificationModule` (which the codebase deliberately did *not* fold into the aggregates it serves — `notification.module.ts` documents "a cross-aggregate delivery concern… its own home as it grows"), not the Traccar ingest controller (a *thin* adapter that reuses telematics' own service and queue with no seams of its own). Folding WhatsApp into `AgentModule` would bloat "the AI chat agent concern" with Twilio, phone mapping, and a message log. The module **imports `AgentModule`**, which grows its exports to expose `AgentService` (W4) and `AgentAttachmentsService` (W5), each when first needed. **Cost named honestly:** exporting `AgentService` widens the trust surface of `runTurn` (which trusts its `actor`), so commitment 3's turn-time authorization re-check is not optional — it is the wall the `RolesGuard` used to be.

2. **Inbound = the Traccar machine-ingress posture, Twilio-adapted, zero-dependency.** A `TwilioSignatureGuard` mirrors `IngestKeyGuard`: it verifies `X-Twilio-Signature` = `base64(HMAC-SHA1(TWILIO_AUTH_TOKEN, baseString))`, where `baseString` is the **configured canonical webhook URL** (`TWILIO_WEBHOOK_URL`) followed by the POST params concatenated in ascending key order, compared with `timingSafeEqual`. It **fails closed with 503** when either `TWILIO_AUTH_TOKEN` or `TWILIO_WEBHOOK_URL` is unset (without both, no request can be verified). The URL is a **server-side constant, never reconstructed from `X-Forwarded-*`/`Host`** — behind Caddy the app sees `http://api:3001`, which would never match Twilio's signature, and request-controlled headers must never enter a security decision (the official SDK's `validateRequest` requires the same configured URL, so this is not a zero-dep workaround — it is what any implementation needs). The guard reads the **parsed urlencoded body** (Twilio posts `application/x-www-form-urlencoded`, not JSON). The controller is `@Controller("api/v1/whatsapp")`, `@UseGuards(TwilioSignatureGuard)` **only** (no `AuthGuard`), returns **202 fast**, validates with a loose boundary schema then canonical revalidation, and enqueues onto `whatsapp-inbound`. Mounted under `/api/v1/…`, it needs **zero Caddyfile change** (the `@api path /api/*` matcher already routes it). The zero-dep signature recipe is pinned in a unit test against Twilio's published test vector (auth token `"12345"`, the documented URL + params → `0/KCTR6DLpKmkAf8muzZqo1nDgQ=`).

3. **An async, in-process worker turn.** A `WhatsAppInboundProcessor` (`@Processor("whatsapp-inbound")`, one queue driven by a job-name switch, the `NotificationProcessor` shape) runs **in-process** (ADR-0029 c5; no split deployment) so the agent's in-memory in-flight lock still protects a same-conversation collision. Its ordered steps: **(a) deduplicate by the inbound `MessageSid`** before anything else (commitment 7) so a Twilio retry or a replayed POST cannot re-fire autonomous CRUD; **(b) resolve** the sender's number to a `User` (commitment 4), failing closed if unmapped; **(c) enforce `agent:use`/ADMIN** on the resolved user at turn time (commitment 9); **(d) get-or-create** that user's WhatsApp conversation (commitment 4); **(e) `runTurn`**; **(f)** on the 409 in-flight collision, **re-enqueue with backoff** (BullMQ retry), never drop; **(g) render** (commitment 6) and **enqueue an outbound send**.

4. **Per-user verified phone links; authorization unchanged.** A new `AgentPhoneLink` model maps a canonical E.164 phone (`@unique`) to a `userId`, with a nullable `conversationId` pointer (FK → `AgentConversation`, `onDelete: SetNull`; set on first turn — this is how "get the user's WhatsApp conversation" is made deterministic, since `AgentConversation` has no channel discriminator), a `verifiedAt`, and an active/deactivated state (for opt-out). Provisioning is a **privileged script** (the `create-user.ts`/`seed-gateway-user.ts` precedent — no public endpoint) that **refuses to link a non-ADMIN user**. An unmapped or deactivated number **fails closed** — no turn runs. **Authorization is `agent:use`, ADMIN-only (ADR-0043 c1), unchanged**, and re-checked at turn time (roles can change after a link is made). Day-1 the table holds the PO's number → the PO's ADMIN user; widening `agent:use` to office staff over WhatsApp is ADR-0043's own separate "Revisit when" decision, not this ADR's.

5. **Outbound = the `Mailer` seam, rotated to WhatsApp.** An abstract `WhatsAppSender` DI token — `send({ to, body }): Promise<{ sid?: string }>` — is the only place that knows the vendor, exactly as `Mailer` wraps the email provider. `TwilioWhatsAppSender` POSTs to the Twilio Messages API over a single outbound HTTPS `fetch` with Basic auth (`AccountSid:AuthToken`); `MockWhatsAppSender` records sends in memory and never touches the network. A `useFactory` binds the real sender only when the `TWILIO_*` env is present and the mock everywhere else (dev/test/CI) — the `RESEND_API_KEY` kill-switch idiom, so unsetting the Twilio creds turns the channel off.

6. **The reply is server-derived text action cards (ADR-0043 c4c honesty, over text).** A **pure** `renderTurnForWhatsApp(AgentTurnResult) → string` (no DB reads — the turn result already carries `messages` + `actions`) produces the reply: it selects the **final** assistant message (skipping intermediate empty-content tool-call rounds), renders server-authored system notices **as notices** (including the ungrounded-claim-guard flag — the transcript must not ventriloquize the model), and renders each `AgentAction` as a plain-text action card — tool, changed-field summary, and a **deep-link** to the affected record. On WhatsApp there is no `/agent/activity` panel in the thread, so these text action cards are the ADMIN's *only* in-the-moment way to catch a bad autonomous write (commitment 9(D)). The reply is chunked to the Twilio/WhatsApp length limits (~1600 chars/segment, 4096 total). Deep-links resolve against a configured web base URL (`BETTER_AUTH_URL` is the API host — the wrong target — so a dedicated web-URL env is plumbed in W1).

7. **Idempotency + delivery ledger = a dedicated `WhatsAppMessageLog`.** Neither `AgentMessage` (role/content/tokens) nor `AgentAction` (tool audit) has a home for wire facts, so a thin ledger records both directions: `direction`, `phone` (Tier-2), `providerSid`, `status`, `conversationId`, optional `messageId`, joined to `AgentMessage` for content (no transcript duplication). Its load-bearing field is **`@@unique` on the inbound `MessageSid`**: the processor checks-existing before `runTurn`, which is the single mechanism that makes Twilio webhook **retries** and **signature replays** idempotent (a signature is not a replay defense — see §Context). The `NotificationLog` shape (`providerMessageId` nullable-until-sent, `sentAt` nullable, a `@@unique` dedup key) is the literal precedent. An outbound Twilio `StatusCallback` webhook (which would let `status` progress past sync-accepted) is named as **deferred**.

8. **Data flow / classification — this section amends ADR-0013 and extends ADR-0043 c6.** The DeepSeek egress contract (ADR-0043 c6 — user text reaches the provider verbatim; tool results are redacted; transcripts are Tier-2) **reapplies unchanged** to inbound WhatsApp text. WhatsApp adds a **new, bidirectional egress class**: **both Twilio (the BSP) and Meta/WhatsApp** see the inbound message content **and** the outbound reply — and the reply embeds Tier-2/Tier-3 fleet data and deep-links, so this leaks *outward*, not only inward. Both are named as content sub-processors. Consequences: inbound/outbound message bodies are **Tier-2** (they carry PII the user types), of the same class as chat transcripts; `AgentPhoneLink.phoneE164` and `WhatsAppMessageLog.phone` are **Tier-2** PII (add pino redact paths only if the processor logs them — the `NotificationLog.recipient` "no code logs it ⇒ no redact entry" posture); `TWILIO_AUTH_TOKEN` and `TWILIO_ACCOUNT_SID` are **Tier-1** secrets (secret store only, never committed/logged). The **honest limit**, stated rather than hidden: anything a user types into WhatsApp reaches Twilio, Meta, **and** (via the agent) DeepSeek verbatim — a phone number or name in a message egresses by definition. **WhatsApp 24-hour session window:** the core flow is always a reply within 24 h of the user's inbound message, so no message templates are needed; two edges are documented — a queue backlog older than 24 h drops (a free-form reply would be undeliverable), and any *future* proactive/outbound-initiated WhatsApp (e.g. wiring reminders to this channel) would need Meta-approved templates and is out of scope. **Cost:** ADR-0043 c8's envelope extends with Twilio's WhatsApp conversation-based pricing (per-24h-window + message fees), which motivates the per-user cap in commitment 9.

9. **Security invariants, first-class.** **(A) Inbound deduplication** (commitment 7) so retries/replays cannot double-fire the no-confirmation-gate writes (ADR-0043 c3). **(B) Turn-time authorization** — the resolver + processor load the user and enforce `agent:use`/ADMIN at turn time (not just at link time — roles change), and provisioning refuses non-ADMIN links; without this, exporting `AgentService` is a privilege-escalation path. **(C) Open-relay invariant** — the channel replies **only** to the signature-verified inbound `From`, **never** to a number embedded in message content, and an unmapped number fails closed (no reply, no agent run) to avoid confirming a live line and to bound cost. **(D) Prompt injection** inherits ADR-0043 c3's explicitly-accepted risk (a forwarded or pasted injected message — or, once un-paused, an OCR'd receipt's text — can drive an autonomous write as ADMIN); no new acceptance is needed, but it raises the stakes on the c4c text-action-card rendering being the operator's in-thread catch. The **structural exclusions of ADR-0043 c3 hold unchanged**: every delete, every invoice issue/cancel/credit-note, raw GPS trace access, and user/role management are absent from the registry — the worst autonomous WhatsApp action is a reversible create/update by design.

**Recommended-but-PO-confirmed picks** (as ADR-0038 left provider/cadence to the PO): unmapped-number behavior = **silent drop + audit** (no courtesy reply); per-user inbound **daily cap** = a modest number the PO sets (WhatsApp makes each inbound a *billed* outbound, so ADR-0043 c8's "no cap" is riskier here); worker **concurrency** = a modest `>1` with 409 re-enqueue (concurrency-1 would head-of-line-block all users); outbound **delivery-status callback** = deferred.

**Tickets.** **W1** this ADR + the ADR-0041 (4th-exercise) and ADR-0013 (new-egress) annotations + env plumbing + glossary + DESIGN.md §Voice + CURRENT_PHASE + a `docs/runbook/whatsapp-operations.md` DRAFT — governance, PO ratifies by merge, no code. **W2** the `AgentPhoneLink` + `WhatsAppMessageLog` migration + the normalize/resolve service (fail-closed, enforces `agent:use`/ADMIN) + the privileged provisioning script. **W3** the `WhatsAppSender` seam (Twilio/Mock/factory) + the pure `renderTurnForWhatsApp`. **W4** the `WhatsAppModule` + `TwilioSignatureGuard` + inbound controller + processor — **the text path end-to-end** (adds `AgentService` to `AgentModule` exports). **W5** the inbound photo path (inert behind the image pause; adds `AgentAttachmentsService` to exports) + the runbook promotion. Production use of everything gates on M1.

## Alternatives considered

**Meta Cloud API direct (instead of Twilio).** Sending through Meta's own WhatsApp Cloud API avoids the Twilio intermediary and its per-message markup. Rejected **by the PO** in favor of Twilio for faster onboarding and a ready sandbox; recorded here with the cost and the added Twilio sub-processor stated (§Consequences). The `WhatsAppSender` seam keeps this a one-implementation swap if Twilio's cost or reliability disappoints.

**Fold the channel into `AgentModule` (the Traccar-inside-Telematics precedent).** Traccar lives inside `TelematicsModule` because it is a thin adapter reusing that module's own service and queue. Rejected here: WhatsApp is a full delivery concern with its own provider seam, queue, guard, audit table, and phone schema — the `NotificationModule` profile — and folding it in would bloat the agent concern. The cost of the dedicated module (exporting `AgentService`) is named and mitigated (commitments 1/9B).

**The official `twilio` SDK.** The SDK offers `validateRequest()` and typed sends. Rejected **by the PO** in favor of zero-dependency raw `fetch` + `node:crypto`, matching ADR-0043's decision to talk to DeepSeek over raw `fetch` and keeping the dependency count at zero. The SDK does **not** avoid the behind-a-proxy signature-URL problem (it takes the URL as an argument you must configure), so it buys little here; the well-specified HMAC-SHA1 recipe is pinned to Twilio's published test vector. The SDK remains the documented fallback if the hand-rolled signing proves fiddly in practice.

**Confirm-before-write for the WhatsApp channel specifically.** A per-channel confirmation gate (the agent proposes, the user replies "yes", then it executes) would blunt the autonomous-write risk on a channel reachable by forwarded/pasted text. Rejected for v1 to keep one agent behavior across surfaces: the autonomous-write posture is ADR-0043 c3's, inherited, and its compensations — the c4c server-derived action cards (now rendered as text), the pre-image undo, the `/agent/activity` ledger, and the ungrounded-claim guard — apply unchanged. This remains the documented fallback (ADR-0043's own §Revisit-when) if WhatsApp-driven writes misfire.

**A single fixed system actor (the Traccar `GATEWAY_USER_ID` model).** Traccar ingests as one hardcoded account-less user. Rejected: WhatsApp must attribute each action to the **real human** behind the number — `createdById` is that human, the capability ceiling is their role (ADR-0021/0043 c1) — and a single shared actor would both hide *which* human acted and grant a shared ceiling. Hence the per-user `AgentPhoneLink`.

**Normalizing `Driver.phone`/`Customer.phone` into the identity (instead of a dedicated link table).** Reusing the existing phone columns would avoid new schema. Rejected: those columns are loose, un-normalized, un-indexed, and non-unique by recorded decision, and they belong to the fleet aggregates, not to auth — matching an inbound MSISDN against them is neither reliable nor a clean identity seam. A dedicated `AgentPhoneLink` keeps messaging decoupled from the domain aggregates and gives the mapping a `@unique` canonical column.

## Consequences

**What this makes easier.** The agent becomes reachable from the phone the PO already carries — no app to open, no desk — while every action stays as audited (pre-images, action cards, the activity ledger, the ungrounded-claim guard) as the web agent's, because it *is* the web agent, reached over a new pipe. The implementing slices are mostly composition: the inbound guard copies `IngestKeyGuard`, the sender copies the `Mailer` seam, the queue/worker copies `NotificationProcessor`, the ledger copies `NotificationLog`, and the turn is the existing `runTurn`. It is legible memory: one ADR explains how a WhatsApp message becomes an audited fleet write, why the reply is server-rendered and not the model's prose, why unmapped numbers get nothing, and where the secrets live.

**What this makes harder / costs accepted.** A new inbound webhook and a new vendor (Twilio) enter the system, plus a new **bidirectional** egress in which Twilio and Meta both see message content — a wider data-flow than the DeepSeek-only egress, accepted explicitly within the commitment-8 contract. The autonomous-write risk (ADR-0043 c3) is now reachable from a forwarded or pasted message, bounded by the same structural exclusions and compensations and raised in stakes on the text action cards. Twilio's conversation-based pricing is a new recurring cost, bounded by the per-user cap. The channel is doubly deploy-dependent: a public 24/7 HTTPS webhook, a real WhatsApp Business number, and the PO's provisioned phone-link all exist only in production — this is built against a system that cannot receive a real message until M1. And a new phone→user identity table must be exactly right: too loose and a stranger reaches the agent, too strict and the PO's own message is dropped.

## Revisit when

- **WhatsApp-driven writes misfire** (wrong-entity updates, injection-shaped incidents visible on `/agent/activity`) → reinstate the confirm-before-write fallback (ADR-0043's, designed) for this channel.
- **`agent:use` is widened to office staff** → that is ADR-0043's separate decision; once granted, an OFFICE_STAFF phone-link would reach the agent over WhatsApp with that role's ceiling automatically.
- **Twilio cost or deliverability disappoints** → swap the `WhatsAppSender` to Meta Cloud API direct (one implementation).
- **Proactive/outbound-initiated WhatsApp is wanted** (reminders over WhatsApp, not just replies) → needs Meta-approved message templates and a template-management surface — out of scope here.
- **Inbound volume or abuse grows** → tighten the per-user daily cap; add the Twilio `StatusCallback` delivery-status webhook so the ledger's `status` reflects real delivery.
- **The deploy lands and daily use begins** → register the WhatsApp number + webhook URL, provision the PO's phone-link, and re-estimate Twilio cost against real usage.

## Phase mismatch (ADR-0041 commitment 4, exercised — the fourth time)

This is unambiguously new pull-forward feature scope opened before the first deploy reaches daily use — the fourth exercise of ADR-0041 commitment 4 (after ADR-0042, ADR-0043, and ADR-0044/0045). The PO chose to open it with the deploy-first recommendation surfaced and declined, as with the prior three; this ADR is the fresh argument commitment 4 requires. The discipline stays honest: **all production use gates on M1** (the inbound webhook needs a publicly reachable 24/7 HTTPS endpoint that only exists post-deploy), so the owed deploy still precedes the channel's first real message. ADR-0025 is not cited as precedent.
