# Roadmap

FleetCo ships in phases. Each phase ends with a usable product, meaning that at the end of any phase the project could be paused indefinitely and the system would still deliver real value. We do not start a new phase until the previous one is in daily use, because half-built phases produce technical debt and operational confusion that compound. (The Phase-1 → Phase-2 transition took a one-time documented exception to this rule — see ADR-0025 — opening Phase 2 ahead of daily use while the Phase-1 deploy items remained owed in parallel; the rule stands for all other transitions.) The roadmap below has been updated to incorporate the operational, reliability, security, and data-posture commitments captured in ADRs 0010 through 0013, which together establish that FleetCo is being built not just as an architectural skeleton but as a delivery operating model.

## Phase 0 — Kickoff

The goal of Phase 0 is to lay the foundation so an AI agent can produce reliable code from day one. The deliverables are the GitHub repository created and pushed, the monorepo scaffold in place with pnpm workspaces and the apps and packages directories, the NestJS API skeleton at apps/api, the Next.js admin skeleton at apps/web, Postgres and Redis running locally via Docker Compose, Prisma initialized with an empty baseline migration, CI on GitHub Actions running lint and typecheck and build and test, all foundational ADRs (numbered 0001 through 0013) committed and reviewed, the CLAUDE.md operating manual and all docs/ baseline documentation in place, and basic auth scaffolding for a single admin user. There is no role-based access control yet and no business features in this phase.

Phase 0 also includes the initial implementation of the operational and security commitments. The CI pipeline gains the security baseline jobs from ADR-0012: Dependabot is enabled, GitHub native secrets scanning is enabled, Semgrep with the OWASP and security-audit rulesets runs on every PR, GitHub Actions workflows pin third-party actions to commit SHAs, and CycloneDX SBOM generation runs on every production build. The DORA metrics file is created in `docs/operations/` even before any deploys happen, so that the structure exists from day one. The runbook stubs are filled in for `dev-setup.md` (because it is needed in Phase 0 to establish the development environment), and the active procedures (incident response, security incident response, business continuity) are reviewed for first-pass completeness. Phase 0 lasts roughly one week.

## Phase 1 — The Spine

The goal of Phase 1 is to replace spreadsheets and paper for daily operations. The system in Phase 1 is web admin only, with the CEO as the only user. The phase is built as a sequence of vertical slices, each delivered end-to-end (schema, migration, service, API, UI, test) before moving to the next. The slices in build order are Vehicles (registering vehicles with Bluebook, insurance, and registration metadata), Drivers (registering drivers with license, contact, and employment status), Customers (the companies and individuals who hire vehicles), Jobs (a contract with a customer that consists of one or more Trips), Trips (the central aggregate; one vehicle plus one driver plus one job plus start and end odometer plus dates), Fuel logs (fuel attached to a Trip or a Vehicle), Expense logs (toll, repair, miscellaneous attached to a Trip or a Vehicle), and Reports v1 (per-vehicle profit and loss, per-trip profit and loss, fuel efficiency). _(Progress note, 2026-06: Reports v1 shipped the per-vehicle **cost** report in iter-23; its named **fuel efficiency** deliverable then shipped as **Reports v2** — the per-vehicle km/L + NPR/km report with theft/engine outlier flags, computed from data already captured — landing in the Phase-2 window as deferred Phase-1 daily-use polish, the same category as the Home dashboard and the Bikram Sambat date work, not new Phase-2 scope. Per-vehicle and per-trip P&L remain the open Reports items. See `docs/CURRENT_PHASE.md`.)_

Phase 1 also lands a set of operational milestones that turn the architectural skeleton into a live delivery operating model. The deployment infrastructure is set up early in Phase 1, which fills in `docs/runbook/deploy.md` and `docs/runbook/rollback.md`. The first production deploy happens before Trip is built (probably during the Vehicles slice), because deploying early and often is the discipline that makes the DORA targets in ADR-0010 achievable. The first restore-from-backup test happens within two weeks of the first production deploy and fills in `docs/runbook/restore-from-backup.md`. The two SLIs from ADR-0011 are instrumented as soon as there is something to measure. OpenTelemetry is added to the NestJS API so that logs carry trace identifiers and the foundation for distributed tracing is in place before Phase 2 needs it. A performance budget is committed (P95 API latency under 500 milliseconds, P99 under 1500 milliseconds, admin first-contentful-paint under 2 seconds on a Nepal 3G connection) and tracked. The data classification from ADR-0013 is applied to every schema during the Phase 1 slices, with Tier 2 and Tier 3 fields explicitly marked.

The end state of Phase 1 is that the CEO stops using spreadsheets for fleet operations and uses FleetCo daily instead, with the operational substrate measured weekly via the DORA metrics file and the SLOs computed monthly. Phase 1 lasts roughly thirty days for the slices themselves, with the operational milestones distributed across the same window.

## Phase 2 — Driver app and telematics

The goal of Phase 2 is to give drivers a phone-based way to log trips and to give the system real-time GPS data. The deliverables are a React Native (Expo) driver app for Android first, trip start and stop on the phone, odometer and fuel entry by the driver, GPS streaming with background location, geofencing for the depot and customer sites and route corridors, and the introduction of office staff as a user role with basic role-based access control.

Phase 2's prerequisites include several operational upgrades that follow from the move to a distributed system topology and the introduction of more sensitive data. The observability stack is upgraded from logs-only to logs-plus-traces-plus-metrics, with one externally hosted backend chosen (candidates include SigNoz, Honeycomb, or Datadog free tier). Distributed tracing is in place across the driver app, the BullMQ jobs that process telematics events, and the R2 uploads for any captured media. The SLI catalog from ADR-0011 expands to include a "telematics ping freshness" indicator and a "driver-app trip-start success" indicator, both at appropriate targets for the new substrate. The data classification from ADR-0013 is revisited because GPS traces are personally identifying data of a particularly sensitive kind; this likely produces a Phase 2 amendment to ADR-0013 with tighter retention and possibly an additional tier. Infrastructure-as-code is codified (Terraform, Pulumi, or scripts under git) for the production environment because Phase 2's complexity exceeds what is reasonable to manage by hand. SLSA Level 1 build provenance is adopted in the GitHub Actions release pipeline. Phase 2 lasts roughly six to eight weeks.

## Phase 3 — Calendar-driven concerns

The goal of Phase 3 is to stop missing deadlines. The deliverables are maintenance schedules driven by kilometers, hours, and calendar dates; service history per vehicle; compliance reminders for Bluebook renewal, insurance, route permits, and Bishesh Anumati; and email notifications for both maintenance and compliance, with SMS to be added later. The SLI catalog adds a "reminder delivery" indicator. Phase 3 lasts roughly four weeks.

## Phase 4 — Money

The goal of Phase 4 is to close the loop on cash. The deliverables are vendors (fuel stations, repair shops, parts suppliers), vendor bills and payment tracking, lease-taker billing for vehicles given on contract, customer invoicing, a basic chart of accounts, exports in IRD-friendly formats, and month-end closing helpers. The data classification ADR is revisited again because financial data has its own sensitivity profile and may justify additional retention or audit-logging requirements. Phase 4 lasts roughly six weeks.

## Phase 5 — Dispatch and manager mobile

The goal of Phase 5 is to enable operations beyond the CEO as a bottleneck. The deliverables are job assignment and dispatch UI, a manager mobile view (read and approve, not full admin), basic route planning (not full optimization), and a customer portal with read-only trip status. The introduction of additional human users requires the role-based access control from Phase 2 to be hardened; the introduction of a customer portal requires public-facing security review. Phase 5 lasts roughly six weeks.

## What we do not do in any phase

There are several capabilities that might seem natural to add but that are deliberately excluded from the v1 plan. We do not do real-time route optimization, because the operational complexity exceeds the value at our scale. We do not do driver behavior scoring or AI-based scoring, because the data quality required is unrealistic in early phases. We do not do multi-tenant SaaS-ification, because FleetCo is built for one company and turning it into a product is a different project. We do not build a replacement for full IRD-grade accounting, because the regulatory and audit requirements would dominate the rest of the work; FleetCo feeds accounting, it does not replace it. These exclusions can be revisited after Phase 5 if the business genuinely asks for them, but they do not block any phase before then.

## Memory adoption mapping

Throughout the phases, the project's memory architecture grows alongside the code. The episodic memory (ADRs) accrues with every architectural decision; we start Phase 1 with thirteen ADRs and expect to add at least one or two per major decision moment. The semantic memory (glossary) grows as new domain terms enter the codebase. The procedural memory (runbook) grows as deployment, rollback, and operational procedures become routine; the active procedures established in Phase 0 are extended with deploy/rollback/restore in Phase 1 and with telematics-specific procedures in Phase 2. The prospective memory (the roadmap, the current-phase tracker, the tech-debt register) is updated as priorities shift and items are completed. The perceptual memory (design folder) is established at the start of Phase 1 with DESIGN.md and grows slice-by-slice as locked mockups are committed for new surfaces. The operational memory (the DORA metrics file and any subsequent SLO reports) accrues weekly from the moment Phase 1 ships. The expectation is that by the end of Phase 1 all six categories of memory are actively maintained, not just the ones that felt urgent at the time. This is not separate work; it is part of the work that produces the code.
