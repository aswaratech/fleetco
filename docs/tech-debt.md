# Tech Debt Register

This is the project's prospective memory for items that have been deliberately deferred and must not be lost. The tech debt register is not a wishlist of features and not a roadmap of phases; it is the place where small commitments that emerged during work-in-flight are held until they are addressed. The discipline is that nothing accepted as a future commitment leaves the conversation that produced it without being written here.

Each entry has a short title, a brief description of what is owed, the slice or PR that introduced or surfaced the debt, the reason the debt was accepted (rather than fixed in place), and a rough estimate of the work required to discharge it. Entries are removed from the register only when the debt is genuinely paid off, and the PR that pays it off mentions the debt entry in its description.

### Secret scanning and secret scanning push protection unavailable at current GitHub plan tier

- **What is owed:** Enable repository-level secret scanning and secret scanning push protection for the `addressanup/fleetco` repository, satisfying in full the four-feature commitment in ADR-0012. Today, only two of the four (Dependabot alerts and Dependabot security updates) are enabled.
- **Where surfaced:** Phase 0 — Ticket 1 (repository bootstrap), first commit on `main`. The `gh api -X PATCH` calls to set `security_and_analysis.secret_scanning.status=enabled` and `security_and_analysis.secret_scanning_push_protection.status=enabled` both returned HTTP 422 with body `"Secret scanning is not available for this repository."` The repo's `security_and_analysis` field comes back as `null`, GitHub's signal that the feature is not offered for this repo/account combination — not a payload error. Verified by also attempting `secret_scanning` alone, which fails identically.
- **Why accepted now:** GitHub does not include repository-level secret scanning on private repos at the Free plan tier. Enabling it requires the GitHub Secret Protection add-on (≈$19/active-committer/month at announced pricing) or the larger GitHub Advanced Security bundle. Phase 0 contains no real secrets in the repository (only documentation), so the absence of repo-level scanning is low-risk in the immediate term. Defense-in-depth is partially provided by (a) GitHub's user-level push protection, which scans every push from the account across all repos and is free and on by default, and (b) the Semgrep job arriving in Ticket 12 per ADR-0012, whose `p/security-audit` ruleset includes secret-pattern detection that overlaps with what GitHub's scanner would catch.
- **Estimate to discharge:** ~10 minutes once the account is on a tier that includes secret scanning. The original `gh api -X PATCH` payload should then succeed unchanged; verification is one read-only `gh api /repos/addressanup/fleetco --jq '.security_and_analysis'` call.
- **Revisit when:** any of: (a) Phase 1 begins, because that is when real secrets (database password, Cloudflare R2 credentials, session secret, Sentry DSN) enter the project's operational surface, even if scoped to a local `.env`; (b) the project moves to a GitHub organization account or upgrades to a paid plan for any other reason; (c) GitHub changes the bundling or pricing of secret scanning in a way that makes enablement cheap or free.

### Orchestration loop's destructive-Bash blocklist will need tuning as the stack evolves

- **What is owed:** Periodically review `scripts/orchestration/src/destructive-bash.ts` against the project's current stack and operational practices. The blocklist as built is conservative (over-blocks rather than under-blocks) and tuned for FleetCo's TypeScript / NestJS / Next.js / PostgreSQL / Prisma / Docker Compose / pnpm / Slack stack at Phase 0. As the project picks up new tools (a deploy CLI, a database migration tool different from Prisma, a different package manager, a new CI helper), the blocklist will need entries added; as the project finds false positives that cause repeated operator interrupts, entries may need narrowing.
- **Where surfaced:** ADR-0022 and `docs/runbook/orchestration-loop-design.md`. The principle is documented in the design as "tune as the project's stack evolves" but no automation enforces revisit cadence.
- **Why accepted now:** A conservative blocklist on day one is the right tradeoff because the cost of a false positive is one operator interrupt while the cost of a false negative can be data loss. Over time, false-positive friction dominates and the blocklist needs trimming. Tuning requires real operational signal (which operator interrupts the blocklist caused, which destructive commands the agent attempted) that does not exist before the loop runs in earnest.
- **Estimate to discharge:** ~30 minutes per quarterly review, or as-needed when an operator interrupt traces to a blocklist false positive. The reviewer reads the recent `decisions.log` for `destructive_bash_denied` entries, judges which were legitimate denials and which were false positives, and adjusts the patterns.
- **Revisit when:** any of: (a) the loop has been used to drive at least one program of 10+ tickets and operational signal is available; (b) an operator interrupt traces to a blocklist false positive; (c) a destructive command escapes the blocklist and causes damage (treat as a SEV2 minimum and add a postmortem-driven blocklist entry); (d) the project's stack changes meaningfully (new package manager, new database, new deploy tooling).

### Fuel-log odometer-monotonicity check deferred

- **What is owed:** A decision (and, if affirmative, an implementation) on whether `FuelLogsService.create` / `update` should reject a fuel log whose `odometerReadingKm` is less than the most-recent prior reading on the same vehicle. Today there is no such check: a fill can record any odometer value, including one lower than an earlier fill's. A `// FUTURE` comment in `FuelLogsService.create` marks the spot. The owed work is either (a) a service-layer monotonicity guard with a clear error message, or (b) a documented, deliberate decision to leave odometer non-monotonic (with the rationale recorded in the glossary's "Fuel log" / "Odometer" entries) — not silence.
- **Where surfaced:** Fuel logs write path, iter 20 (`feat/fuel-logs-write-path`, PR #48). The iter-20 kickoff explicitly deferred the check, and `docs/CURRENT_PHASE.md`'s iter-20 sentence records the deferral; this register entry is the formal home so it is not lost.
- **Why accepted now:** Real-world fuel data has gaps and out-of-order entries (a receipt logged days late, an odometer misread at the pump, a fill recorded against the wrong vehicle then corrected). A hard monotonicity check on every write would block the bulk-import path a future iter needs and would force operators to fight the system over benign data-entry order. Odometer monotonicity is also genuinely a cross-aggregate invariant — it interacts with the Trip start/end odometer and the iter-11 Trip→Vehicle odometer auto-update — so it is best designed once, alongside trip-end-odometer reconciliation, rather than bolted onto the fuel-log write path in isolation.
- **Estimate to discharge:** ~1–2 hours for option (a): a `findFirst` for the latest prior reading on the vehicle within the service, a comparison, a `BadRequestException` with a clear message, and ~4 service tests (in-order accepted, out-of-order rejected, first-fill accepted, null-odometer fills skip the check). ~15 minutes for option (b): the documented-decision path.
- **Revisit when:** any of: (a) the per-vehicle fuel-efficiency (km/L) report is built (likely alongside or after Reports v1) — bad odometer ordering corrupts km/L, which is the first consumer that makes the invariant matter; (b) a bulk-import path for historical fuel logs is added (which would need a chosen policy for out-of-order rows); (c) the trip-end-odometer reconciliation work lands, at which point the cross-aggregate odometer invariant should be designed as a whole.

### Vehicle/Driver delete must map P2003 to HTTP 409 once Trip write path lands — paid off 2026-05-26

- **Discharged in:** Trips slice iter 9 (`feat/trip-write-path`). The two service-layer `delete` methods now catch `Prisma.PrismaClientKnownRequestError` with `code === "P2003"`, run `prisma.trip.count({ where: { vehicleId: id } })` (or `driverId`), and throw `new ConflictException(\`Cannot delete vehicle: ${count} trip${count === 1 ? "" : "s"} reference this vehicle.\`)` (or driver). The plural arm is exercised by the new Vitest cases in `apps/api/test/vehicles.service.test.ts` (count=2) and `apps/api/test/drivers.service.test.ts` (count=1), both pinning that the row remains present after the failed delete (`Restrict` prevented the delete from happening, and the service did not double-delete). Soft-delete remains an option if a future audit-trail requirement reverses the calculus. The glossary's `legal-status-transition` entry now records this iter's status-transition matrix; the existing "Soft delete" entry was already aligned with the block-when-referenced choice and needs no edit.

## Notes for future contributors

If you find yourself accepting a deferred commitment in a session — "we'll come back and clean this up later," "this is a known limitation we're shipping for now," "we'll need to revisit when X happens" — write the entry here in the same PR that creates the debt. Do not let "I'll add it later" pass; later does not arrive, and the writer's context evaporates within hours. The whole point of this register is to be the durable home for deferred work, and that only functions if entries are written at the moment the deferral is decided, not at some imagined future moment when there will be time to record them.
