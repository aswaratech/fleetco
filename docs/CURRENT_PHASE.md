# Current phase

**Phase 1 — The Spine**

Phase 0 (Kickoff) is complete: the monorepo scaffold, NestJS API skeleton, Next.js admin skeleton, Postgres + Redis via Docker Compose, Prisma baseline + auth migration, CI on GitHub Actions, the security baseline jobs from ADR-0012, the baseline documentation, the agent guardrails, the orchestration loop (ADR-0022), and basic auth scaffolding for a single admin user are all in place.

Phase 1 replaces spreadsheets and paper for daily fleet operations. The system in Phase 1 is web admin only, with the CEO as the only user. The phase is built as a sequence of vertical slices per ADR-0006 (schema → migration → service → API → UI → test), in this order: Vehicles, Drivers, Customers, Jobs, Trips, Fuel logs, Expense logs, Reports v1. Operational milestones (first production deploy, first restore-from-backup test, SLI instrumentation, OpenTelemetry, performance budget) are interleaved across the same window.

The first three Phase-1 aggregates have shipped: Vehicles (iters 1–4: schema + read + write + tests), Drivers (iters 6–7: read + write + tests), and Trips (iters 8–9: read + write + tests, with the FK Restrict → 409 mapping closing the cross-aggregate delete-blocker loop). Iter 10 landed the first cross-slice read affordance — "Recent trips" sections on the Vehicle and Driver detail pages, plus a "View N trips" deep-link inside the 409 delete-blocker dialogs. Iter 11 closed the Trip → Vehicle odometer auto-update loop: when a Trip transitions to COMPLETED, the service bumps the referenced Vehicle's `odometerCurrentKm` inside the same Prisma interactive transaction as the trip write, so a vehicle's odometer reading on the detail page reflects the latest completed trip without operator intervention. Iter 12 added per-vehicle lifetime stats (completed trip count, total km logged, most recent driver) under `GET /api/v1/vehicles/:id/stats`, surfaced as a "Lifetime stats" section on the Vehicle detail page. Iter 13 shipped the symmetric per-driver lifetime stats — `GET /api/v1/drivers/:id/stats` plus the matching "Lifetime stats" section on the Driver detail page — rotated 90° around the Trip aggregate (most-recent-vehicle replaces most-recent-driver). Iter 14 extended the Vehicle aggregate with Nepal-specific compliance metadata: eight nullable columns plus an `InsuranceType` enum (Bluebook number + expiry, insurer + policy number + type + expiry, route permit number + expiry), surfaced on the Vehicle create form, edit form, and a new "Compliance" section on the detail page. Bishesh Anumati, expiring-soon badges, and the `<NepaliDate>` BS-calendar component remain deferred to their own slices. Iter 15 (this PR) opened the Customers aggregate with the read path: `model Customer` (name, contactPerson, phone, email, panNumber unique-when-present, address, `CustomerStatus` enum ACTIVE/INACTIVE, plus the standard audit fields and `createdBy` FK), `GET /api/v1/customers` with status filter + sortable columns (name / createdAt) + pagination, `GET /api/v1/customers/:id` with 404 mapping, list / detail / loading UI under `apps/web/src/app/customers/`, and a fourth nav button on the gated home page. The pino redact list gains `*.contactPerson` so the new Tier 2 PII does not leak into logs. The remaining Phase-1 slices are Customers (write path — iter 16), Jobs, Fuel logs, Expense logs, and Reports v1; operational milestones (first production deploy, first restore-from-backup test, SLI instrumentation, OpenTelemetry, performance budget) interleave across the same window. See `docs/product/roadmap.md` §"Phase 1 — The Spine" for the full Phase 1 scope and operational commitments. When the last Phase 1 slice (Reports v1) ships and the CEO is using FleetCo daily, this file is updated to indicate Phase 2 is active.
