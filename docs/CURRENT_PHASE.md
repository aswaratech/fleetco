# Current phase

**Phase 1 — The Spine**

Phase 0 (Kickoff) is complete: the monorepo scaffold, NestJS API skeleton, Next.js admin skeleton, Postgres + Redis via Docker Compose, Prisma baseline + auth migration, CI on GitHub Actions, the security baseline jobs from ADR-0012, the baseline documentation, the agent guardrails, the orchestration loop (ADR-0022), and basic auth scaffolding for a single admin user are all in place.

Phase 1 replaces spreadsheets and paper for daily fleet operations. The system in Phase 1 is web admin only, with the CEO as the only user. A dedicated module (not folded into telematics) follows ADR-0029 c2's "the retention concern owns `traces-prune`" and the in-repo ReportsModule precedent of querying another aggregate's table via the shared PrismaService, and leaves the T3 ingestion path untouched. T5 (this slice) ships the read path and the program's last ticket: the RBAC-gated raw/derived read split (ADR-0028 c6 / ADR-0027 c7) and the FIRST PostGIS geofencing, all on the existing telematics feature module with no schema change. Three read endpoints join the composed AuthGuard + RolesGuard chain, each with its own `@RequirePermission`: `GET /api/v1/telematics/vehicles/:vehicleId/pings` (the full-resolution raw trace — paginated, time-bounded by `from`/`to`, Prisma-native Float reads that NEVER select the generated geometry column) gated `gps:read-raw` (ADMIN-only, the most-privileged operational data access in the system); and `GET …/location` (the single latest fix for the live-location map) plus `GET …/geofence-status` gated `gps:read-derived` (ADMIN + OFFICE_STAFF). The geofence-status endpoint is the derived layer's first real spatial query: it classifies a vehicle's latest fix against a **caller-parameterized** geofence via parameterized `prisma.$queryRaw` — point-in-polygon by `ST_Contains(ST_GeomFromText(<wkt>, 4326), geometry)` and proximity by `ST_DWithin(geometry::geography, <center>::geography, <meters>)` (the geography cast makes the radius meter-accurate, the documented geometry-vs-geography choice; every value is bound via the tagged template, so there is no SQL-injection surface, and the `ST_MakePoint(lon, lat)` foot-gun is guarded by inside/outside + swapped-axes tests). It returns a single boolean (+ the non-sensitive fix time, no coordinates) — a genuinely lower-resolution Tier-3 derived status, not the raw trail relabeled (ADR-0027 c6 anti-circumvention). The two read capabilities were already in the hardcoded map from the ADR-0028 RBAC slice (which fixed the split), so T5 consumed them with no map edit. Geofence-**polygon storage** (a `geometry(Polygon, 4326)` company-configuration aggregate, Tier 3 per ADR-0027 c6) is deferred to its own sibling slice — it needs its own design (depot / customer-site / route-corridor types, naming, a management surface); the geofencing query is identical once a stored-polygon slice supplies the geometry from a row instead of a query param. Tier-5 egress discipline holds on reads (coordinates go to the authorized caller over the wire but never log via the pino `redact` list and never enter a span; the spatial `$queryRaw` returns only a boolean + timestamp). 37 new tests (670 API total) prove the RBAC gate (403/200/401) and the spatial inside/outside + within/beyond classification against real PostGIS. **With T5 the GPS telematics ingestion program (ADR-0029, T1–T5) is COMPLETE** — the seam from a ping arriving to a ping stored, retained, and read back is built end-to-end.
