# Current phase

**Phase 1 — The Spine**

Phase 0 (Kickoff) is complete: the monorepo scaffold, NestJS API skeleton, Next.js admin skeleton, Postgres + Redis via Docker Compose, Prisma baseline + auth migration, CI on GitHub Actions, the security baseline jobs from ADR-0012, the baseline documentation, the agent guardrails, the orchestration loop (ADR-0022), and basic auth scaffolding for a single admin user are all in place.

Phase 1 replaces spreadsheets and paper for daily fleet operations. The system in Phase 1 is web admin only, with the CEO as the only user. The phase is built as a sequence of vertical slices per ADR-0006 (schema → migration → service → API → UI → test), in this order: Vehicles, Drivers, Customers, Jobs, Trips, Fuel logs, Expense logs, Reports v1. Operational milestones (first production deploy, first restore-from-backup test, SLI instrumentation, OpenTelemetry, performance budget) are interleaved across the same window.

The first three Phase-1 aggregates have shipped: Vehicles (iters 1–4: schema + read + write + tests), Drivers (iters 6–7: read + write + tests), and Trips (iters 8–9: read + write + tests, with the FK Restrict → 409 mapping closing the cross-aggregate delete-blocker loop). Iter 10 landed the first cross-slice read affordance — "Recent trips" sections on the Vehicle and Driver detail pages, plus a "View N trips" deep-link inside the 409 delete-blocker dialogs. Iter 11 closed the Trip → Vehicle odometer auto-update loop: when a Trip transitions to COMPLETED, the service bumps the referenced Vehicle's `odometerCurrentKm` inside the same Prisma interactive transaction as the trip write, so a vehicle's odometer reading on the detail page reflects the latest completed trip without operator intervention. Iter 12 added per-vehicle lifetime stats (completed trip count, total km logged, most recent driver) under `GET /api/v1/vehicles/:id/stats`, surfaced as a "Lifetime stats" section on the Vehicle detail page. Iter 13 shipped the symmetric per-driver lifetime stats — `GET /api/v1/drivers/:id/stats` plus the matching "Lifetime stats" section on the Driver detail page — rotated 90° around the Trip aggregate (most-recent-vehicle replaces most-recent-driver). Iter 14 extended the Vehicle aggregate with Nepal-specific compliance metadata: eight nullable columns plus an `InsuranceType` enum (Bluebook number + expiry, insurer + policy number + type + expiry, route permit number + expiry), surfaced on the Vehicle create form, edit form, and a new "Compliance" section on the detail page. Bishesh Anumati, expiring-soon badges, and the `<NepaliDate>` BS-calendar component remain deferred to their own slices. Iter 15 opened the Customers aggregate with the read path: `model Customer` (name, contactPerson, phone, email, panNumber unique-when-present, address, `CustomerStatus` enum ACTIVE/INACTIVE, plus the standard audit fields and `createdBy` FK), `GET /api/v1/customers` with status filter + sortable columns (name / createdAt) + pagination, `GET /api/v1/customers/:id` with 404 mapping, list / detail / loading UI under `apps/web/src/app/customers/`, and a fourth nav button on the gated home page. The pino redact list gains `*.contactPerson` so the new Tier 2 PII does not leak into logs. Iter 16 (this PR) closes the Customers aggregate by adding the write path: `POST /api/v1/customers` (201 + created row), `PATCH /api/v1/customers/:id` (200 + updated row), `DELETE /api/v1/customers/:id` (204), each guarded by `AuthGuard`; service-layer PAN normalization (trim + uppercase) makes the unique index case-insensitive in practice; Prisma `P2002` on `panNumber` is mapped to HTTP 409 with the response body `{ statusCode: 409, message, field: "panNumber" }` so the matching create / edit forms (under `apps/web/src/app/customers/new/` and `apps/web/src/app/customers/[id]/edit/`) surface the conflict inline on the PAN input via `form.setError`; Prisma `P2003` on `DELETE` is mapped to HTTP 409 with "Cannot delete customer: it is referenced by other records." — forward-compatible dead code today, exercised in iter 17 when Jobs adds its `customerId` FK under `onDelete: Restrict`. The detail page gains an Edit / Delete cluster mirroring Drivers iter 7; the delete dialog is a shadcn `AlertDialog` client island with a placeholder "View related records" link that iter 17 will swap to a `/jobs?customerId=…` deep-link. Iter 17 opened the Jobs aggregate with the read path: `model Job` (`jobNumber` unique [`JOB-YYYY-NNNNN`; generator lands iter 18], `customerId` FK Restrict to Customer, free-text `description`, `JobStatus` enum PLANNED/IN_PROGRESS/COMPLETED/CANCELLED, nullable scheduled/actual start+end dates, nullable `notes`, audit fields), `GET /api/v1/jobs` with status + customerId filters, sortable columns (jobNumber / scheduledStartDate / createdAt), pagination, `GET /api/v1/jobs/:id` returning the nested Customer, list / detail / loading UI under `apps/web/src/app/jobs/`, and a fifth nav button on the gated home. The Job→Customer FK flips the iter-16 forward-compatible Customer-delete-blocker from dead code to live: an integration test seeds a Customer + a referencing Job, attempts the delete, and asserts the `P2003 → 409` "Cannot delete customer: it is referenced by other records." path. Iter 18 (this PR) closes the Jobs aggregate by adding the write path: `POST /api/v1/jobs` (201 + created row with the nested Customer), `PATCH /api/v1/jobs/:id` (200 + updated row), `DELETE /api/v1/jobs/:id` (204), each guarded by `AuthGuard`; the jobNumber generator runs server-side (`JOB-YYYY-NNNNN`, current UTC year + highest existing per-year sequence + 1, retried up to 3 times on `P2002` so a concurrent insert cannot deterministically collide); `jobNumber` and `customerId` are immutable post-create (the `PATCH` schema's `.strict()` rejects both); the four date pairs (scheduled start/end, actual start/end) enforce end ≥ start as a service-layer cross-field rule that re-runs against the merged shape on `PATCH` (so a PATCH that touches a single date in a pair re-validates against the stored value for the other half); `P2002` on `jobNumber` is mapped to HTTP 409 with the response body `{ statusCode: 409, message, field: "jobNumber" }` so the matching create form can surface the (effectively unreachable, defensively handled) conflict inline; `P2003` on `customerId` is mapped to HTTP 400 with "Customer <id> does not exist." which the create form surfaces as a field-level error on the customer picker. Create / edit / delete UI lands under `apps/web/src/app/jobs/new/` and `apps/web/src/app/jobs/[id]/edit/` plus `[id]/delete-job-dialog.tsx`; the create form fetches the active customers list (sortBy=name asc, take=200) server-side and renders a native `<select>` picker; the edit form pre-fills every field, renders jobNumber + customer name as read-only, and PATCHes only the diff against initial values; the delete dialog is a shadcn `AlertDialog` client island with named "Delete job" / "Keep job" buttons. The Customer delete-dialog placeholder "View related records" link is swapped to the now-live `/jobs?customerId=<id>` URL with the clearer label "View jobs for this customer". The remaining Phase-1 slices are Fuel logs, Expense logs, and Reports v1; operational milestones (first production deploy, first restore-from-backup test, SLI instrumentation, OpenTelemetry, performance budget) interleave across the same window. See `docs/product/roadmap.md` §"Phase 1 — The Spine" for the full Phase 1 scope and operational commitments.
