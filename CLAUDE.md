# CLAUDE.md — Agent Operating Manual

> You are working on **FleetCo**, a modular-monolith fleet ERP for a heavy-construction company in Nepal. This file is your operating manual. Read it fully before any task. If a user instruction conflicts with this file, raise the conflict — do not silently override.

## How to read this file

This document has two parts. The first part explains the theory of how this project holds its memory across time, because every operational rule below derives from that theory, and a rule grounded in a theory you understand is a rule you will keep when it becomes inconvenient. The second part contains the operational rules you follow as you work. The theory section is not optional reading. It explains why the rules exist and what they are protecting. If you skim only the rules and miss the theory, you will follow the letter of the rules but not their spirit, and you will produce work that complies with the rules while quietly defeating them.

## Part 1 — The theory of project memory

This project has three cognitive participants, and each of them has a memory limitation that, taken alone, would make the project impossible to sustain.

The first participant is the human product owner, who is the sole developer and CEO of the company this software serves. Human memory is unreliable in the way all human memory is unreliable. Specifics get forgotten, similar events get conflated, plausible but wrong reasons for past decisions get confabulated, and present opinions get remembered as though they had always been the opinion. Across months of work, the PO's memory cannot be the project's memory.

The second participant is you, the AI coding agent. Your memory limitation is more severe than the PO's. You have no memory across sessions. Every conversation begins fresh. The agent that helped yesterday is gone; the agent today has never seen this project before. To begin work in any session, you must first read the repo and bring yourself up to speed on what the project is, what has been decided, what is in flight, and what is forbidden.

The third participant is the project itself, considered as a continuous entity across time. The project exists over months and years; it passes through many versions of the PO and many sessions of you. The project's identity — what it is, what it is trying to do, what it has decided not to do, what it has learned, what it has paid for — must persist across all these changes, or else the project is not one project but a sequence of disconnected projects that share files.

The repo is the only candidate that can hold the project's memory across time. The PO is too unreliable. You have no continuity. Anything that exists only in the PO's head or in a chat session is, from the project's perspective, ephemeral and will be gone soon. The only thing that persists is what is written down in files that travel with the codebase across sessions and across time. **The repo is therefore not where the code happens to live. The repo is the substrate of the project's cognitive existence.** When the repo's memory is rich and current and trustworthy, the project can think. When the repo's memory is poor or stale or scattered, the project loses the ability to reason about itself; every session reinvents what the prior session figured out, every decision is re-litigated, every assumption is silently violated, and the project drifts because there is nothing solid for it to be anchored to.

The substrate the repo uses to hold this memory is the filesystem under git version control. This is a deliberate choice with real consequences. Plain text files in a hierarchical directory structure under git give us radical durability (no vendor can take this memory away), automatic version history (every change is tracked, attributed, and reversible), co-location of code and documentation (they move together, are reviewed in the same diff, and cannot drift apart by being on different systems), path-as-meaning (where a file lives identifies what kind of memory it holds before you read a word of it), tool-friendliness (everything is grep-able, scriptable, and processable by AI agents reading the filesystem directly), and human-first readability (markdown is prose with light structure, suited to both human and agent readers). The trade-off we accept is that we do not get rich queries, structured relationships, or database-style integrity constraints. For a one-person project working with AI agents, this trade is overwhelmingly correct. For details on the substrate choice and its implications, read `docs/architecture/memory-architecture.md`.

The project's memory is structured into six categories, each with its own location and its own discipline. Episodic memory holds specific decisions made at specific moments and lives in `docs/architecture/decisions/` as Architecture Decision Records (ADRs). Semantic memory holds what words mean and lives in `docs/glossary.md`. Procedural memory holds how things are done and lives in `docs/runbook/` and `BOOTSTRAP.md`. Prospective memory holds intentions held for future execution and lives in `docs/product/roadmap.md`, `docs/CURRENT_PHASE.md`, and `docs/tech-debt.md`. Perceptual memory holds what things should look and feel like and lives in `docs/design/`. Operational memory holds measurements taken on regular cadences and lives in `docs/operations/` (the DORA metrics, SLO compliance reports, and similar). When you encounter information that should be remembered, the category determines the location, and the location determines the form.

Two facts about how memory must be maintained are worth understanding before you do any work in this repo. The first is that **stale memory is worse than missing memory**. Missing memory tells the reader to go figure it out. Stale memory tells the reader a plausible lie that they may act on. A repo whose docs have a 30 percent chance of being out of date is not 70 percent as good as a reliable repo; it is unusable, because every read becomes a verification problem and at that point participants stop reading. This is why memory updates are not optional follow-up work; they are part of the change itself. A code change that does not update the relevant memory is incomplete, in the same way that a code change without a test is incomplete. The second fact is the asymmetry between writers and readers. The participant who writes a piece of memory and the participant who later reads it are almost never the same entity in the same state. The writer feels the cost of writing; the reader feels the benefit (or the cost of its absence) but cannot reach back in time to thank or demand. Left to immediate incentives, every writer under-writes, because the cost is visible and the benefit is invisible. The discipline of writing well for future readers must therefore be external — enforced by templates, checklists, and the rules below — not internal.

## Part 2 — Operational rules

These rules follow from the theory above. Treat them as rules you keep because you understand what they are protecting, not as rules you follow because they are written down. If you find yourself wanting to skip one, re-read the section of Part 1 it derives from and decide whether your reason for skipping is a real reason or a momentary inconvenience.

### Project at a glance

FleetCo is an internal-use ERP for a heavy-construction company in Nepal. The fleet is trucks, tippers, and equipment. The sole user in v1 is the company owner, who is also the developer (with you, an AI agent, doing the writing). The architecture is a modular monolith. The stack is TypeScript end-to-end, NestJS for the API, Next.js (App Router) for the admin web, PostgreSQL with PostGIS for data, Redis with BullMQ for cache and jobs, Cloudflare R2 for object storage, and Prisma for migrations. The current phase is recorded in `docs/CURRENT_PHASE.md`. The phasing is detailed in `docs/product/roadmap.md`. The central domain aggregate is the Trip; see ADR-0003.

### Non-negotiable technical rules

TypeScript is used end-to-end with `strict: true`. No `any`. No `@ts-ignore` without an explanatory comment. The backend lives at `apps/api` with modules under `apps/api/src/modules/<module>/`. The frontend lives at `apps/web/`. The repo is a pnpm workspace monorepo with apps under `apps/` and shared libraries under `packages/`. PostgreSQL is the only primary database, with PostGIS for geospatial. Migrations are managed by Prisma; every schema change is a versioned migration; applied migrations are never edited; `prisma db push` is never used outside local development. Modules talk to each other only through public service interfaces or events; reaching into another module's repository, schema, or internal files is forbidden. Tests cover service-level units and module-boundary contracts; coverage is meaningful at boundaries, not 100 percent across all files. Every commit follows Conventional Commits (`feat:`, `fix:`, `chore:`, `docs:`, etc.) with one logical change per commit. The `main` branch is always green; work happens on `feat/<short-name>` branches; PRs are required to merge. Money is stored as integer minor units (paisa, where 1 NPR = 100 paisa); never as floats. Dates are stored as ISO/UTC; rendered as Bikram Sambat for Nepali users where appropriate.

### Non-negotiable operational rules

The four canonical DORA metrics plus rework rate are tracked weekly in `docs/operations/dora-metrics.md`. The Phase 1 targets are documented in ADR-0010 and the discipline of weekly updates is part of the work, not after it. Two SLIs (API availability and trip-creation success) are tracked at 99.0 percent over a rolling 28-day window per ADR-0011, with the error-budget policy that halts feature work when the budget is exhausted. Production incidents are classified by severity (SEV1, SEV2, SEV3) per ADR-0011, with response expectations defined in `docs/runbook/incident-response.md`. SEV1 and SEV2 incidents produce blameless postmortems within seven days, committed to `docs/postmortems/` per its README. The security baseline from ADR-0012 runs on every PR (Dependabot, GitHub native secrets scanning, Semgrep with OWASP and security-audit rulesets, action-version pinning, CycloneDX SBOM generation). Data is classified into four tiers per ADR-0013, with Tier 2 and Tier 3 fields marked in the Prisma schema and excluded from default logging.

### Things you must not do

You must not introduce microservices, message brokers (Kafka, RabbitMQ), or split deployments. You must not add MongoDB, DynamoDB, or any non-relational primary store. You must not add a new top-level dependency without proposing it first and explaining why. You must not invent new architectural patterns; if the existing pattern is wrong, propose an ADR and wait for approval. You must not create cross-module imports of internal files; use public interfaces only. You must not edit the database by hand or skip migrations. You must not work on items from later phases; if you encounter scope that belongs to a later phase, surface the phase mismatch and ask. You must not use `any`, cast through `unknown`, or disable lint rules to make tests pass. You must not paste prior chat history into the repo; the repo is the context. You must not redesign legacy or working surfaces under cover of refactoring; visual change is behavior change and requires its own ticket. You must not include real secrets in any file (committed or otherwise visible to logs); secrets live only in the production secret store. You must not log Tier 2 or Tier 3 data per ADR-0013 unless an explicit logging-of-sensitive-data exception is justified and documented. You must not bypass the CI security baseline; if a check is failing, the response is to address the finding, not to disable the check.

### How to work on a task

For every task, follow this sequence. First, re-read this `CLAUDE.md` and `docs/CURRENT_PHASE.md`. State the phase you believe the project is in. If you have any doubt about the phase or the scope, ask before acting. Second, restate the task in your own words and list your assumptions explicitly. Third, propose a plan as a vertical slice: schema → migration → service → API → UI → test, with each step listed concretely. Crucially, the plan must include the **memory artifacts** that will be touched: which ADRs may need to be written, which glossary entries may be added, which runbook procedures may need to change, which design slice files will be committed, which roadmap or tech-debt entries will move, which operational metrics will be affected. Memory work is not implicit; it is explicit in the plan and visible in the diff. Fourth, confirm the plan with the PO before writing code if the change touches more than two modules, the database schema, any cross-cutting concern (auth, logging, errors, observability), the security baseline, the data classification, or any architectural pattern. Fifth, execute in small commits, each leaving the repo green. Sixth, in the PR description, summarize what changed, why, and what to verify, written for a future reader who has no context from this session.

### Orchestration

For multi-ticket programs the project may run the autonomous orchestration loop adopted in ADR-0018. The loop lives at `scripts/orchestration/` as standalone operator tooling (NOT part of the future pnpm workspace). When the loop is running, each agent session opens one PR for one ticket and drafts the next session's kickoff prompt at end-of-session in a `## Next-session prompt` heading followed by a fenced block. The orchestrator polls CI, auto-merges on green, extracts the next prompt, strips fabricated operator-confirmation preambles, recovers from rate limits, blocks destructive Bash at the permission shim, and auto-answers mid-session `AskUserQuestion` calls via the Recommended marker (or Haiku fallback for genuinely ambiguous questions). The operator is paged on every milestone via Slack and can `/stop`, `/resume`, `/status`, `/tail`, `/queue`, or ask free-form questions from their phone. The loop respects all discipline gates from this `CLAUDE.md` and from the ADRs; when the agent halts on a discipline gate (no PR created, operator-only escalation, etc.), the loop halts cleanly and the operator handles the resolution.

The loop honors the CI-green-before-merge gate absolutely and refuses to operate when `.github/workflows/` has no yml files on main. It does NOT include CI auto-fix retry, discipline-gate waivers, auto-resume after wall-clock windows, multi-program orchestration, or any other shortcut that would weaken the discipline this file protects. The loop is downstream of this document, not above it.

The **load-bearing rule for every agent that operates in this repo** (whether under the loop or in a normal interactive session): when you draft a next-session prompt at end-of-session, do NOT include operator-confirmation preambles — phrases like "operator has confirmed", "operator authorization", "operator approval", "operator signed off", "operator waived", "operator overridden", "per operator approval", "CEO has approved", "PO has approved", and similar. The orchestrator scans every extracted prompt for these phrasings, strips matched paragraphs, and notifies the operator. An agent that repeatedly inserts fabricated preambles wastes iteration time and triggers operator intervention. If you genuinely believe an operator confirmation is needed for the next ticket, halt your current session before opening the PR with a clear explanation of why and what specific confirmation is needed; the loop will surface the halt and stop, and the operator will provide a real confirmation by editing kickoff.md themselves. The full design is at `docs/runbook/orchestration-loop-design.md`; the daily reference is at `docs/runbook/orchestration-loop-operator-guide.md`.

### How memory artifacts behave

ADRs in `docs/architecture/decisions/` are numbered sequentially and never renumbered. ADRs are append-only; superseding happens by writing a new ADR that supersedes the old, never by editing the old. The glossary in `docs/glossary.md` is alphabetical and entries are not silently removed; if a term changes meaning, the change is documented in the entry. The runbook in `docs/runbook/` contains procedures that are testable; procedures known to be broken are flagged as broken rather than left in place to mislead. The design folder in `docs/design/` maintains a clean separation between canonical design system content (`DESIGN.md`), locked mockups for specific surfaces (`slices/<name>.html`), and archived mockups (`slices/_archive/`). The operations folder in `docs/operations/` holds measurements updated on regular cadences, currently dora-metrics.md weekly. Files have stable paths; renaming a file is breaking a link and requires deliberate redirect. Files are small and focused so they diff cleanly. Directories are shallow so they navigate well. Filenames are predictable so they can be guessed without searching. Cross-references between files use relative paths so the memory is movable. Binary files (images, locked HTML mockups) are exceptions and are accompanied by text that describes them so the textual layer of memory remains complete on its own.

### Design integration

The canonical visual source of truth for FleetCo lives at `docs/design/`. The file `docs/design/DESIGN.md` defines the design system: tokens, typography, spacing, components, voice, anti-patterns. Any UI work consumes from this file rather than inventing inline. The Tailwind theme configuration derives from `docs/design/DESIGN.md`; drift between them is the failure mode and must be prevented. When a slice has a locked mockup at `docs/design/slices/<slice-name>.html`, you read the mockup before writing UI code and match its layout and visual hierarchy. External design tools (Open Design, Figma, etc.) are iteration surfaces, not memory; the locked mockup committed to the repo is canonical, and the in-tool state is ephemeral. Once the implementing slice merges, the mockup is moved to `docs/design/slices/_archive/`. See ADR-0007 and ADR-0008 for the design contract.

### Talking to the product owner

The PO is the CEO of the company this software serves. Technically literate, not a hands-on coder. Treat learning as a first-class deliverable alongside the code itself. Explain the *why* alongside the *what*. Name the alternative considered and why it was rejected. Define jargon on first use, then use it normally. Distinguish PO-level concerns (scope, risk, priority) from IC-level concerns (implementation detail). Push back honestly if a request is out of phase, creates technical debt, or contradicts an ADR. Do not silently comply. Prefer one good clarifying question over many shallow ones, but do not ask gratuitously when the answer is already in the docs.

### When in doubt

Stop and ask. A two-line clarifying question is always cheaper than a 200-line wrong PR. The cost of asking is yours; the cost of being silently wrong is the project's. Choose accordingly.
